## Description

`docs/oddrn.md` uses the term "Ingestion API" twice (line 17: "efficient handling of Ingestion API payloads"; line 23: "consumers of the Ingestion API"). "Ingestion API" is a registered alias of "ODD Specification" (see `main-concepts.md` Terms & Aliases: *"ODD Specification ↔ Ingestion API spec, ingress API"*).

Neither use on oddrn.md links to the specification repo, to `main-concepts.md`, or to any other page that explains what the Ingestion API is. Readers arrive at oddrn.md to learn about ODDRN, see references to an "Ingestion API", and have no onward path.

## Acceptance Criteria

- [ ] The first mention of "Ingestion API" on `oddrn.md` (line 17) is turned into a link pointing to either the ODD Specification (external: `https://github.com/opendatadiscovery/opendatadiscovery-specification`) or to the ODD Specification section of `main-concepts.md` (internal anchor)
- [ ] The choice between external-spec link and internal main-concepts link is made deliberately — prefer the internal link if main-concepts.md's ODD Specification section is substantive enough to orient the reader; otherwise link directly to the spec repo
- [ ] The second mention (line 23) does not need its own link — once is enough
- [ ] Link is a plain markdown link (no `"mention"` shortcut)
- [ ] Post-merge live-site check: WebFetch `https://docs.opendatadiscovery.org/oddrn`, confirm the new link renders correctly and resolves to either the specification repo or the internal main-concepts anchor

## Implementation Notes

- Internal link is usually better — it keeps readers inside the docs site and lets main-concepts.md's "ODD Specification" section route them further (including to the GitHub spec repo). Match how `main-concepts.md` itself handles the "Ingestion API" alias.
- If main-concepts.md's ODD Specification section is too brief to serve as a destination when you check it, link directly to the spec repo instead — don't send the reader to a sentence-length stub.
