## Description

Editorial audit on 2026-05-03 (during `/review batch:feature/docs-features-md-spotcheck`) surfaced four orphaned `.md` files at the doc root that are **not referenced from `docs/SUMMARY.md`** and serve no current purpose:

### File 1 — `docs/Overview.md` (parallel surface with drift)

`docs/SUMMARY.md:3` maps the **Overview** entry to `README.md`. `docs/Overview.md` is a separate file containing similar but **drifted** content:

- Uses 6 bullets instead of 5 ("AI-based UX" appears in Overview.md but not in README.md).
- Has its own in-page TOC at the top (4 entries — `About ODD`, `The platform for your business`, `Features and use cases`, `Start using ODD`); README.md does not.
- Uses `.svg` image extensions; README.md uses `.png`.
- Has a "Start using ODD" CTA with a Calendly demo link that doesn't appear in README.md.
- Contains a stray `/` typo at the start of the About section: `/The open-source Data Discovery Platform improves productivity...`

The two files are clearly the result of a doc-restructure that moved Overview content from `Overview.md` → `README.md` (so the GitBook root is the Overview page), but `Overview.md` was never deleted. It now sits as a phantom that diverges from the canonical Overview every time README.md is edited.

### File 2 — `docs/quick_start.md` (abandoned draft with typos)

Contents (entire file, 9 lines):

```markdown
# Quick Start
## Main steps
1. To understand how the ODD works and what features it has, setup (??) a local demo invironment (link).
2. Set required parameters for the Collector and fro the Platfrom (link).
3. Deploy the Collector and the Platfrom using Helm (link). Also to connect custom microservices to the ODD, deploy the Tracing Gateway (link).     
## Next steps
This section provides recomendations for those who have successfully connected required datasources to the Platfrom and are ready to use ODD features.
### Ingest your metadata
### Add users
```

Multiple typos (`invironment`, `fro`, `Platfrom` ×3, `recomendations`), placeholder syntax `(??)` and `(link)` instead of real links, and empty H3 sections. This is an abandoned draft. The user-facing "quick start" today lives in `Try locally` (`configuration-and-deployment/trylocally.md` — referenced from SUMMARY).

### File 3 — `docs/platform.md` (empty)

The file has 0 lines (or 1 empty line per file-stat). True ghost — likely an artifact of a renamed/deleted page that left the file behind.

### File 4 — `docs/Data_Compliance_DS.md` (legacy PascalCase duplicate)

Contents are functionally identical to `docs/dc_data_compliance.md` (which is in SUMMARY.md as a use-case sub-page). Both `README.md:37` and `Overview.md:36-38` carry commented-out references to `Data_Compliance_DS.md`, evidence of an old SUMMARY entry that was migrated to the snake_case form `dc_data_compliance.md` but the original PascalCase file was not deleted.

## Why this slipped past prior batches

`pillars/documentation/gates.md` Gate 7 (Layout and completeness) checks **SUMMARY ↔ page list sync** for pages that exist in SUMMARY (no orphans pointing *out* of SUMMARY). It does not check the inverse — pages on disk that aren't in SUMMARY. Adding an explicit step to Gate 7 would catch this entire class:

> For every `docs/**/*.md` not in SUMMARY.md, classify as: legitimate non-SUMMARY page (e.g., `.gitbook/includes/*`); orphan-to-delete; missing-from-SUMMARY (add the entry).

Editorial-audit case-law for `playbooks/doc-product-editorial-read.md` — failure shape: Phantom file (file exists in tree but not reachable from canonical navigation).

## Context

- **Failure shape**: Phantom file (per `playbooks/doc-product-editorial-read.md` failure-shape table) + Parallel surfaces with drift (Overview.md vs README.md).
- **Why medium (not high)**: orphan files are not reachable from the SUMMARY navigation, so most readers never encounter them. But (a) a reader who lands via direct URL (search-engine result, an old shared link) gets a stale or broken page; (b) the doc tree's coherence as an authored product is compromised — every future maintainer wonders whether these files are work-in-progress or dead.
- **Why not low**: `quick_start.md` actively misleads — its placeholders and typos read as a published quick-start that the project couldn't be bothered to finish. Same for `Overview.md`'s drift.
- **Recurring pattern**: a doc-restructure leaves orphan files behind; the absence of a Gate-7 disk-vs-SUMMARY sweep means they accumulate over time. Fold into Gate 7.
