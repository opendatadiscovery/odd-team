## Description

Line 139 of `odd-platform.md`, under the "Internal PostgreSQL" session provider section, shows `SESSION_PROVIDER=IN_MEMORY`. This is wrong — for the Internal PostgreSQL provider the value must be `INTERNAL_POSTGRESQL`. The sibling YAML tab (lines 101-106) correctly uses `provider: INTERNAL_POSTGRESQL`, so only the env-var tab has the bug. A user following the env-var tab ends up running the in-memory provider despite expecting PostgreSQL-backed sessions.

- [ ] The page's explanatory text (if any) does not contradict the corrected value

## Re-Audit (2026-04-23)

Flipped from `done` to `review-ready` after consumer-read re-audit under the new eight-gate Quality Bar (commit `96b28c4` on `feature/pipeline-hardening`).

**No in-scope amendment required.** DOC-006 was a narrow 1-line default-value correction and the shipped change is still correct. The consumer-read audit did, however, surface three deep operator caveats for the session-provider feature that DOC-006 was not responsible for covering:

- `IN_MEMORY` — `ReactiveMapSessionRepository` wraps a raw `ConcurrentHashMap` with no eviction beyond Spring Session TTL; long-running high-cardinality-session processes can OOM
- `INTERNAL_POSTGRESQL` — `PostgreSQLSessionHousekeepingJobHandler.java:13` uses `@Scheduled(fixedRate = 1, timeUnit = HOURS)`; expired sessions sit in `SPRING_SESSION` for up to 1h past TTL; cadence is not configurable
- `REDIS` — `@EnableRedisWebSession` has no local connection wiring; operator must supply `spring.data.redis.*` (host, port, credentials, TLS, timeout, pool); undocumented

**Classification**: out-of-scope for DOC-006 → logged as `backlog/docs/DOC-059.md` (`scanner_source: "discovered-during-DOC-006-reaudit"`) on `feature/pipeline-hardening`.

**Action**: This item is awaiting `/review` in a separate session — findings file `findings/docs-accuracy-integration-caveats/2026-04-23-reaudit-critical-odd-platform-config.md` confirms the in-scope change is correct.
