## Description

Discovered during the same user spot-check as DOC-091 (2026-05-03). The Features.md `## Alerting` section's notification-setup narrative has three cross-link defects that prevent a reader from getting to the canonical configuration home.

### Site 1 — lines 146-152 carry no cross-link at all

Lines 146-152 narrate Slack / webhook / email alert notification setup in prose:

> These alerts can be sent via Slack, Webhook and/or email, serving as a notification that something requires attention.
>
> For webhook notifications users need to specify an endpoint to receive HTTP POST requests.
>
> For email notifications users need to place the recipient's email address and configure SMTP details ([Google SMTP](https://support.google.com/a/answer/176600?hl=en), [AWS SMTP](https://repost.aws/knowledge-center/ses-set-up-connect-smtp), etc) including sender and receiver information and the SMTP server's hostname.
>
> For Slack notifications users need to add the [Slack webhook](https://api.slack.com/messaging/webhooks) URL and specify the channel or recipient for receiving notifications.
>
> With these setup options, ODD users can seamlessly integrate alerting mechanisms into their preferred workflow channels.

These paragraphs describe **what** the operator needs to set up but do not link to **where** the actual config keys live (`docs/configuration-and-deployment/odd-platform.md:325` `## Enable Alert Notifications`, lines 371-386 enumerating the eleven `notifications.*` keys). A reader has to guess the right anchor on the configuration page, or scroll the entire 1000+-line config doc to find it.

This violates **Cornerstone 3**: "Bidirectional cross-links between feature pages and their configuration sections are mandatory."

### Site 2 — line 281 cross-link is broken (wrong anchor + absolute Medium URL)

Line 281 (under `### Setting up alert notifications`):

> For setting up the alert notifications within the platform all required [variables](https://docs.opendatadiscovery.org/configuration-and-deployment/odd-platform?source=post\_page-----86ddd04e5aaf--------------------------------#postgresql-configuration) must be defined in configuration files.

Two problems:
1. **Wrong anchor**: `#postgresql-configuration` is a different config H2 (the Postgres connection settings, line 234 of odd-platform.md) — has nothing to do with alert-notification setup. The intended target is `#enable-alert-notifications` (line 325).
2. **Absolute Medium-tracking URL**: the link uses `https://docs.opendatadiscovery.org/configuration-and-deployment/odd-platform?source=post_page-----86ddd04e5aaf--------------------------------#postgresql-configuration` — clearly copy-pasted from a Medium article. Per `pillars/documentation/authoring.md`, intra-repo cross-links must be relative paths (e.g., `configuration-and-deployment/odd-platform.md#enable-alert-notifications`); absolute URLs to the live site break offline rendering and carry stale tracking parameters.

### Site 3 — line 287 has the same defect class

Line 287 (under `### Cleaning Up`):

> ODD users can easily disable Alert Notification functionality by following several steps described in [configurations](https://docs.opendatadiscovery.org/configuration-and-deployment/odd-platform?source=post\_page-----86ddd04e5aaf--------------------------------#cleaning-up).

Same Medium tracking URL pattern. The `#cleaning-up` anchor on the configuration page may or may not exist — verify before fixing; if it doesn't, the right target is the body of `## Enable Alert Notifications` that documents the disable flow (a sub-section TBD or the surrounding prose).

## Why this slipped past prior batches

The Features.md `## Alerting` section was authored long before the current Cornerstone 3 / Quality Bar regime; the Medium-URL pattern dates to a Medium article a maintainer copy-pasted from. Subsequent reviews focused on alerting *content correctness* (lifecycle, halt config, retry semantics — DOC-087 territory), not cross-link hygiene. A scanner specifically for "Cornerstone 3 cross-link coverage between sibling docs" is in scope for `scanners/docs/quality/` (per `pillars/documentation/cornerstones.md` Cornerstone 3 closing line) but has not yet been authored.
