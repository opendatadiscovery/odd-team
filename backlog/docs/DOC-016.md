## Description

**Rescoped on 2026-04-29.** Original DOC-016 (2026-04-21) called for a new `docs/developer-guides/adapter-reference/` subtree (README + 5 collector-specific pages) listing every adapter with its config fields. That work has since been done — and is being completed — under the canonical Integrations hub:

- `docs/integrations/README.md` (DOC-042) — pull/push hub with vocabulary, common config, decision tree.
- `docs/integrations/collectors/{odd-collector,odd-collector-aws,odd-collector-azure,odd-collector-gcp,odd-collector-profiler}.md` (DOC-042) — per-collector pages with full inventories and spotlight per-adapter tables.
- DOC-070 (in-progress this batch) — extends `odd-collector.md` to all 38 non-spotlighted adapters.
- DOC-071 (in-progress this batch) — extends `odd-collector-aws.md` to all 9 non-spotlighted adapters.
- DOC-072 + DOC-073 (done) — Azure and GCP collectors already at full per-adapter coverage.

Building a parallel `developer-guides/adapter-reference/` subtree would violate Cornerstone 2 of `CLAUDE.md` (one canonical location per piece of detail; every other mention is a cross-link).

What does need to ship is the **bidirectional cross-link** the operator-facing build-and-run page is missing today. `docs/integrations/README.md:43` already links into `build-and-run-odd-collectors.md#full-configuration-reference` for the SDK-level config; the reverse direction does not exist. The build-and-run page sends operators to GitHub `config_examples/` URLs (line 85) and never points them at the canonical per-adapter documentation. Cornerstone 3 of `CLAUDE.md` makes the bidirectional link mandatory.

## Context

- **Why the rescope is necessary** — the canonical home for adapter detail moved to `docs/integrations/collectors/` when DOC-042 shipped (commits e7ebe54..8961a1f on documentation main, 2026-04-26..2026-04-28). Authoring under `developer-guides/adapter-reference/` after that point produces parallel content. The 2026-04-22 retrospective on DOC-003 (S2S) and DOC-013 (collector secrets backend) named this exact failure mode; Cornerstone 2 was added in response.
- **Why the bidirectional gap is the right fix** — `integrations/README.md:43` already crosses into build-and-run for SDK-level top-level fields. Build-and-run not crossing back into integrations means a developer reading "configure collector-config.yaml" sees only the GitHub `config_examples/` URLs and has to discover the integrations hub separately. The canonical pattern is bidirectional (Cornerstone 3 of `CLAUDE.md`).
- **DOC-070's Implementation Notes** explicitly assume DOC-016 has been rescoped to "developer-facing — how to author a new adapter, SDK protocol". That interpretation is **also superseded**: that surface is DOC-044's territory (`docs/developer-guides/custom-collectors.md`, pending). DOC-016 = bidirectional cross-link only.
