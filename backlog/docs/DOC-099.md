## Description

Editorial audit on 2026-05-03 (during `/review batch:feature/docs-features-md-spotcheck`) surfaced two real issues with `docs/Architecture.md` plus a vocabulary-extension opportunity that crosses to `docs/main-concepts.md`.

### Issue 1 — Collector category vs implementation conflation

`Architecture.md:11` says:

```
**ODD Collector** is a lightweight service which gathers metadata from all your data sources:
```

But `main-concepts.md:23` (the canonical taxonomy) establishes that **Collector** is a *category* — "container of pull adapters + runtime (launcher, logging, Platform-API client, configuration reader, scheduling)" — and `odd-collector` is one specific implementation among several. The integrations/README.md table at lines 13-22 lists 5 collector implementations: `odd-collector`, `odd-collector-aws`, `odd-collector-azure`, `odd-collector-gcp`, `odd-collector-profiler`.

Saying "ODD Collector is a lightweight service" implicitly claims a single `odd-collector` is *the* collector, contradicting the per-cloud and profiler collectors documented elsewhere. A reader who lands on Architecture.md first builds the wrong mental model.

### Issue 2 — Sparse coverage for a top-level SUMMARY entry

`Architecture.md` is **17 lines total** (entire file):
- Diagram image (line 5)
- Pointer to Main Concepts for vocabulary (line 7)
- One paragraph on Push-client (line 9) — *valid in the client-server topology framing; see Issue 3 below*
- One bullet list on the Collector (lines 11-14) — *carries the conflation above*
- One paragraph on ODDRN (line 16)

That's it. Cornerstone-2 establishes that top-level SUMMARY entries are scarce — each one signals "primary navigation pillar." Architecture is one of those, yet the page barely scratches the surface. Missing:
- Description of the data flow (producer → ingest → store → query → render).
- Explanation of which components run where (Platform; Collectors as deployable services hosting plugins; Push adapters embedded in the source's runtime; optional Tracing Gateway).
- Coverage of cross-cutting concerns (auth, alerting, lineage, search, attachments).
- Trade-off discussion (pull vs push topology, when to choose which).

A reader landing on Architecture.md from SUMMARY today gets less than they would from main-concepts.md, which has 7 H2 sections and ~110 lines.

### Issue 3 — Vocabulary extension opportunity (Architecture.md is RIGHT to use Push-client; main-concepts.md should capture both framings)

`docs/main-concepts.md:102` (Terms & Aliases) lists `Push adapter` as canonical with `Push-client` as the alias. `Architecture.md:9` uses **Push-client** as its headline term — and that is **correct** for the client-server topology framing the Architecture page operates in.

In the client-server architectural framing, the ODD Platform is the server and **any** adapter — push *or* pull — is a client to it. "Push-client" positions the adapter on the network topology (it is a *client* of the Platform that uses *push* strategy). "Push adapter" positions the same component in the producer-side mapping vocabulary (it is an *adapter* that uses *push* strategy to deliver mappings). Both terms are meaningful; both deserve canonical status in their respective framings.

Treating "Push-client" as drift from "Push adapter" would impoverish the doc — it would force Architecture.md to be rewritten in producer-side mapping vocabulary it isn't operating in. The right fix is to **extend** main-concepts.md Terms & Aliases to capture both framings. Same may apply to other multi-framing terms (Collector — deployment unit vs. category; Plugin — configured-adapter-instance vs. configuration-block).

## Why this slipped past prior batches

Architecture.md was authored at the very beginning of the project and never refactored as the doc product grew. Subsequent batches focused on per-feature surfaces and the canonical-home work. The Pre-authoring stance check Question 4 ("Does this change preserve the WHY?") would have flagged the sparseness on any new authoring; but the page hasn't been authored against in a long time.

The Issue-3 vocabulary-extension finding required re-framing what counts as "Conceptual drift" — the canonical guidance (`feedback_extend_vocabulary.md`, captured 2026-05-03 from user spot-check during this same audit) is that two terms describing the same concept from different *framings* are not drift; they are complementary canonical terms in different vocabularies. The doc-product editorial-audit playbook should reflect that.

Editorial-audit case-law for `playbooks/doc-product-editorial-read.md` — failure shape: Coverage hole (top-level SUMMARY page is sparse) + Conceptual drift candidate that **is not actually drift** when the two terms operate in different framings (a new shape category to add to the playbook's failure-shape table: **"Multi-framing vocabulary"** — same concept, two valid canonical terms, one per framing; resolution is to extend Terms & Aliases, not to rewrite either page).

## Acceptance Criteria

### Phase A — fix the Collector category conflation

- [ ] Replace `**ODD Collector** is a lightweight service which gathers metadata from all your data sources:` on `Architecture.md:11` with prose that distinguishes the category from the implementations. Suggested:
  > **Collectors** are containers of pull adapters plus the runtime around them — adapter launcher, logging, Platform-API client, configuration reader, scheduling. The canonical implementation is `odd-collector` (40+ bundled pull adapters); specialist collectors exist for AWS (`odd-collector-aws`), GCP (`odd-collector-gcp`), Azure (`odd-collector-azure`), and data profiling (`odd-collector-profiler`). What an operator deploys is the collector container; what each collector hosts is one or more configured plugins, each carrying one pull adapter's connection settings. See [Integrations](integrations/README.md) for the per-implementation pages.
- [ ] Cross-link from Architecture.md to `main-concepts.md` "Adapter, Plugin, Collector, Push adapter" table (line 46-57) for the canonical four-term distinction.

### Phase B — extend Terms & Aliases to capture client-server framing (the vocabulary fix)

- [ ] Add a row to `docs/main-concepts.md` Terms & Aliases that captures the client-server framing of producer-side components. Suggested shape (verify against the table format):
  - **Push-client** (framing: client-server topology) — Producer-side component that runs **inside** the source system's own runtime as a client of the ODD Platform server, using the *push* strategy to deliver metadata. Equivalent to **Push adapter** in producer-side mapping vocabulary; the framings are complementary. Use **Push-client** when the surrounding text is about deployment topology or network position; use **Push adapter** when the surrounding text is about extract-and-map mechanics. → cross-link to existing Push adapter row.
- [ ] Optionally: capture the same multi-framing pattern for **Pull-client** (the producer-side framing equivalent for a Collector hosting pull adapters) if the team uses that term in any surface — sweep `grep -rn 'pull-client' docs/` first to confirm whether the term appears.
- [ ] **Do not rewrite** `Architecture.md:9`'s use of Push-client to "Push adapter" — that page operates in the client-server topology framing where Push-client is the right term. The Terms & Aliases extension is what carries the canonical reconciliation.

### Phase C — coverage extension (medium, scope expansion — could split as DOC-NNN+1)

- [ ] Decide whether to expand Architecture.md to a real architectural-overview page, or repurpose it as a thin index that points at main-concepts.md. If expanded, content should cover: (a) the data flow from producer to UI; (b) the deployment topology (Platform, Collectors, Push-clients in source runtimes, optional Tracing Gateway); (c) cross-cutting concerns (auth, alerting, lineage, search, attachments); (d) where to read the code (point at `navigation/domains/*.md` from the workspace, or at the relevant feature pages).
- [ ] If repurposing as a thin index, ensure no SUMMARY entry breaks; the current top-level Architecture entry should still be reachable.

### Phase D — Sources footer

- [ ] Sources footer cites: `Term: main-concepts.md §Terms — "Collector"` (the canonical category vs implementation distinction), `Cross-doc: integrations/README.md:13-22` (the collector-implementations table), plus `Term: main-concepts.md §Terms — "Push-client"` (the new client-server-framing row this item adds).
