### F-022: OKTA env var copy-paste bug — provider set to "google"
- **Location**: `documentation/docs/configuration-and-deployment/enable-security/authentication/oauth2-oidc.md` line 231
- **Issue**: OKTA environment variables example sets `AUTH_OAUTH2_CLIENT_OKTA_PROVIDER=google` instead of `AUTH_OAUTH2_CLIENT_OKTA_PROVIDER=okta`. Users following this will configure OKTA but the platform will treat it as Google, likely causing authentication failures.
- **Evidence**: Line 231: `AUTH_OAUTH2_CLIENT_OKTA_PROVIDER=google`. The YAML tab (line 215) correctly uses `provider: okta`. Copy-paste from the Google section.
- **Severity**: critical
- **Suggested fix**: Change `google` to `okta` on line 231.

### F-023: Permissions list is severely outdated — 20+ missing, 4 ghost entries
- **Location**: `documentation/docs/configuration-and-deployment/enable-security/authorization/permissions.md`
- **Issue**: The permissions list is significantly out of sync with the codebase. Missing permissions include: `DATA_ENTITY_ALERT_CONFIG_UPDATE`, `DATA_ENTITY_ATTACHMENT_MANAGE`, `DATA_ENTITY_STATUS_UPDATE`, `DATASET_FIELD_DESCRIPTION_UPDATE`, `DATASET_FIELD_INTERNAL_NAME_UPDATE`, `DATASET_FIELD_TAGS_UPDATE`, `DATASET_FIELD_ADD_TERM`, `DATASET_FIELD_DELETE_TERM`, all `QUERY_EXAMPLE_*` (7), all `LOOKUP_TABLE_*` (9), `OWNER_RELATION_MANAGE`. Also lists 4 permissions that don't exist: `LABEL_CREATE`, `LABEL_UPDATE`, `LABEL_DELETE` (labels aren't a feature), and `DATASET_FIELD_INFO_UPDATE` (was split into `DATASET_FIELD_DESCRIPTION_UPDATE` + `DATASET_FIELD_INTERNAL_NAME_UPDATE`). Also lists `DATA_ENTITY_GROUP_DELETE` which is not in the Permission enum.
- **Evidence**: `odd-platform-specification/components.yaml` lines 158-235 define the full Permission enum. Compare against docs lines 15-78.
- **Severity**: critical
- **Suggested fix**: Regenerate the full permissions list from the OpenAPI spec. Remove LABEL_* and DATASET_FIELD_INFO_UPDATE. Add all QUERY_EXAMPLE_*, LOOKUP_TABLE_*, and other missing entries.

### F-024: Session provider env var copy-paste bug (formalized from X-001)
- **Location**: `documentation/docs/configuration-and-deployment/odd-platform.md` line 139
- **Issue**: Under "Internal PostgreSQL" environment variables section, the value is `SESSION_PROVIDER=IN_MEMORY` — should be `SESSION_PROVIDER=INTERNAL_POSTGRESQL`. A user following this will get in-memory sessions instead of PostgreSQL sessions, losing sessions on restart.
- **Evidence**: Line 136: "**In memory (default)**" section correctly says `SESSION_PROVIDER=IN_MEMORY`. Line 139: "**Internal PostgreSQL**" section copy-pasted the same value.
- **Severity**: critical
- **Suggested fix**: Change `IN_MEMORY` to `INTERNAL_POSTGRESQL` on line 139.

### F-025: Metrics config docs missing storage backend and Prometheus options
- **Location**: `documentation/docs/configuration-and-deployment/odd-platform.md` lines 163-187 (Enable Metrics)
- **Issue**: Docs only describe `metrics.export.enabled` and `metrics.export.otlp-endpoint`. Code has 3 additional properties: `metrics.storage` (INTERNAL_POSTGRES or PROMETHEUS), `metrics.prometheus-host`, and the default storage is INTERNAL_POSTGRES. Users don't know they can use Prometheus as a metric storage backend.
- **Evidence**: `application.yml` lines 158-163: `metrics.storage: INTERNAL_POSTGRES`, `metrics.prometheus-host: http://localhost:9090`.
- **Severity**: high
- **Suggested fix**: Document `metrics.storage` and `metrics.prometheus-host` properties. Explain the INTERNAL_POSTGRES vs PROMETHEUS choice.

- **Location**: Missing from all documentation pages

Zero documentation exists.
- **Evidence**: 14 Java files in `odd-platform-api` matching `*LookupTable*`. `DataEntityTypeDto.java` line 36: `LOOKUP_TABLE(24)`. `components.yaml` lines 194-202: 9 LOOKUP_TABLE_* permissions. UI: `src/components/MasterData/LookupTables.tsx`.
- **Severity**: high
- **Suggested fix**: Create documentation page for the Lookup Tables / Master Data feature covering creation, column definitions, data management, and permissions.
