## Description

Editorial audit on `/review batch:feature/docs-bigbatch-2026-05-06` surfaced a **wrong-anchor cross-link** in the Catalog Overview page section of Features.md.

### The defect

`docs/Features.md:329` (inside the Catalog Overview page's "Sections" bullet list):

```markdown
* **Owner association** — appears only when authentication is enabled; lets a signed-in user request to be linked to an [Owner](Features.md#manual-object-tagging).
```

The link target is `Features.md#manual-object-tagging` — i.e. it points at the **Manual Object Tagging** H2 section. Manual Object Tagging is about applying **tags** to data assets and dataset columns; it is not about **Owners** or owner association. A reader clicking the link to learn more about "Owner" lands on a section about tagging — semantic mismatch.

### What the link should target

The Catalog Overview's "Owner association" section describes the user-facing flow where a signed-in user requests to be linked to a platform Owner. The canonical homes for that content live elsewhere:

- **Owner concept (the platform-managed user identity)**: `docs/configuration-and-deployment/enable-security/authorization/owners.md`.
- **The user-owner association flow itself (admin approval, request UI)**: `docs/configuration-and-deployment/enable-security/authorization/user-owner-association.md`.

Either is a valid target for L329's `[Owner]` link, depending on whether the reader needs the concept (owners.md) or the workflow (user-owner-association.md). The user-owner-association.md flow is the closer fit since the surrounding sentence is specifically about the request-to-be-linked workflow — "lets a signed-in user request to be linked to an Owner".

### Reader-experience cost

A reader on the Catalog Overview page section sees "Owner" cross-linked, clicks expecting a definition of what an Owner is and how association works, and lands on the Tags feature description instead. This is exactly the magic-link problem (link text doesn't match what the reader gets) but in its **wrong-anchor** sub-shape rather than its uninformative-link-text sub-shape.

## Failure shape

**A4 (Reference integrity) — wrong-anchor sub-shape** (per `pillars/documentation/quality-framework.md` and `playbooks/doc-product-editorial-read.md` failure-shape table). Distinct from DOC-129 / DOC-110 magic-links — the link text here is fine ("Owner"), but the destination anchor is semantically unrelated to the surrounding sentence's promise.
