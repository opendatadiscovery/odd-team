## Description

Editorial audit on 2026-05-03 (during `/review batch:feature/docs-features-md-spotcheck`) surfaced a **duplicate HTML anchor** on `docs/Features.md`:

- Line 173 — `### Setting up alert types  <a href="#id-128d" id="id-128d"></a>`
- Line 279 — `### Setting up alert notifications <a href="#id-128d" id="id-128d"></a>`

Both H3 sections inside the `## Alerting` H2 carry the **same** GitBook-generated shadow anchor `id-128d`. HTML allows only one element per `id` — clicking any inbound link to `#id-128d` lands on the first occurrence (Setting up alert types), regardless of which section the author intended. Browsers (and screen readers) silently honour the first match.

The two sections cover distinct topics — `Setting up alert types` walks through the `Notification Settings` button (UI configuration of which alert categories fire); `Setting up alert notifications` walks through the YAML/env-var configuration of the alert-delivery channels. A reader landing on the wrong one wastes time before realising the mismatch.

The duplication is a GitBook artifact — when a heading is renamed or copy-edited inside the GitBook editor, the existing shadow anchor sometimes survives and gets re-attached to the new copy. Neither Gate 7 (in-page TOC sync) nor Gate 8 (live-site verification) catches this class because they verify each section in isolation.

## Why this slipped past prior batches

The 10 Quality Bar gates verify per-item correctness. Gate 7 checks that the in-page TOC indexes every H2 — but does not check anchor-ID uniqueness within the page. The DOC-092 batch added new cross-links into `## Setting up alert notifications` (`#enable-alert-notifications`, `#cleaning-up`) but did not touch the legacy `id-128d` shadow anchor on either H3. The duplicate has been on the page since before DOC-083 era.

Editorial-audit case-law for `playbooks/doc-product-editorial-read.md` Internal contradiction (failure shape: anchor-ID collision within a single page).
