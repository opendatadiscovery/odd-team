## Description

Attachment storage is completely undocumented. `application.yml` (lines 215-224) defines these keys:

- `attachment.storage` — `LOCAL` or `REMOTE` (default: `LOCAL`)
- `attachment.max-file-size` — MB (default: 20)
- `attachment.local.path` — default: `/tmp/odd/attachments`
- `attachment.remote.url`, `attachment.remote.access-key`, `attachment.remote.secret-key`, `attachment.remote.bucket`

The default `LOCAL` path `/tmp/odd/attachments` is inside the container filesystem and is wiped on every pod/container restart. A user (Stefano Messina) reported this as a production data-loss issue. The feature shipped without documentation, and the ephemeral default is a trap in any Kubernetes deployment.

## Related code issue (to log as `odd-platform` GitHub Issue)

**Title:** Attachment storage: `MinioAsyncClient` built without region, restricts AWS S3 to `us-east-1`

**Body:**
`MinioConfig.java` builds the REMOTE-mode MinIO client with only endpoint + credentials:

```java
return MinioAsyncClient.builder()
    .endpoint(url)
    .credentials(accessKey, secretKey)
    .build();
```

No `.region(...)` call is present. The MinIO Java SDK defaults to `us-east-1` for SigV4 signing in this case. Against AWS S3 this means only `us-east-1` buckets work — any other region fails with `AuthorizationHeaderMalformed` or `PermanentRedirect`. Self-hosted MinIO and most S3-compatible services are unaffected.

**Proposed fix:** Accept an optional `attachment.remote.region` property and, when non-empty, pass `.region(region)` to the builder. Default-empty behavior can remain the current `us-east-1` to stay backwards-compatible. Documented as a known limitation in `docs/configuration-and-deployment/odd-platform.md` until fixed.

**Reporter:** Stefano Messina.

**In-scope amendments required** (F-CAV-002 in `findings/docs-accuracy-integration-caveats/2026-04-23-reaudit-critical-odd-platform-config.md`):

- `MinioConfig.java:19-25` — `MinioAsyncClient.builder()` leaves `.httpClient(...)` unset; MinIO SDK default is OkHttp with read/writeTimeout **≈ 5 minutes**; large-file uploads (> 5 min wall-time) fail; connect timeout is not configurable via any ODD property. (Region caveat already shipped in commit `0f8e5e3`; this is a separate unset-parameter gap in the same builder.)
- `RemoteFileUploadServiceImpl.java:61-77` — `completeFileUpload` reassembles chunks from a **LOCAL filesystem directory** (`FileUtils.getChunkDirectory(uploadId)`) **even in REMOTE mode** before uploading the assembled object. In a container restart mid-upload, the chunk directory is wiped — same failure class as the LOCAL-mode ephemeral-default warning DOC-008 already carries, but for REMOTE mode the doc is silent.
- `RemoteFileUploadServiceImpl.java:107-135` — `putObject`, `getObject`, `removeObject` wrap any exception as `MinioException` with no retry; transient S3/MinIO failures (503, network blip) fail the call immediately. Not documented — operator expectation mismatch.
