## What

The "Distribution anomaly" toggle on the entity-level **Notification Settings** UI (and its `distribution_anomaly_halt_until` field on `PUT /api/dataentities/{id}/alert_config`) is a no-op. Setting it has no effect on whether new Distribution Anomaly alerts get created — the platform persists the timestamp to the `alert_halt_config` table but never reads it on the alert-creation path.

Operators silencing a noisy entity expect Distribution Anomaly halts to behave like the other three alert-type halts (Failed job, Failed data quality test, Backwards incompatible schema change), all of which **are** enforced. They aren't.

## Where

The four halt-type toggles are persisted by the same code path:

- `odd-platform-api/.../service/AlertHaltConfigServiceImpl.java:35-43` — `saveAlertHaltConfig` writes all four halt timestamps to `alert_halt_config`.
- `odd-platform-api/.../components.yaml:2379-2393` — `DataEntityAlertConfig` schema exposes all four fields publicly.

Halt enforcement only happens in **one** place — the ingestion-driven path:

- `odd-platform-api/.../service/ingestion/processor/AlertIngestionRequestProcessor.java:54-56` builds `AlertActionResolver` with the `haltConfigs` map.
- `odd-platform-api/.../service/ingestion/alert/AlertActionResolverImpl.java`:
  - Lines `38-56` (`resolveActions(taskRuns, taskRunType)`) dispatches `FAILED_JOB` and `FAILED_DQ_TEST` cases through `toHalt(...)` and applies the halt filter at line 54.
  - Lines `58-85` (`resolveActions(candidates)`) handles `BACKWARDS_INCOMPATIBLE_SCHEMA` only, with halt check at line 71.
  - Lines `176-191` (`toHalt(...)`) — the switch supports all four types **including** `DISTRIBUTION_ANOMALY` at line 187, but the `DISTRIBUTION_ANOMALY` arm is dead code: nothing in this class ever calls `toHalt(...)` with that argument.

The actual Distribution Anomaly alert-creation path is **separate** and **never consults the halt config**:

- `odd-platform-api/.../controller/AlertManagerController.java:21-26` — `POST /ingestion/alert/alertmanager` → delegates to `alertService.handleExternalAlerts(req.getAlerts())`.
- `odd-platform-api/.../service/AlertServiceImpl.java:152-191` (`handleExternalAlerts`):
  ```java
  public Mono<Void> handleExternalAlerts(final List<ExternalAlert> externalAlerts) {
      if (CollectionUtils.isEmpty(externalAlerts)) { return Mono.empty(); }
      // ...
      for (final ExternalAlert externalAlert : externalAlerts) {
          // ... build AlertPojo with type = DISTRIBUTION_ANOMALY, status = OPEN ...
          final AlertPojo alert = new AlertPojo()
              .setType(AlertTypeEnum.DISTRIBUTION_ANOMALY.getCode())
              .setDataEntityOddrn(externalAlert.getLabels().get("entity_oddrn"))
              .setStatus(AlertStatusEnum.OPEN.getCode());
          alerts.add(alert);
          // ...
      }
      return createAlerts(alerts, alertToChunks, emptyList());
  }
  ```
  No injection of `AlertHaltConfigRepository`, no `getDistributionAnomalyHaltUntil()` lookup, no filter — the alert is unconditionally written to the DB.

`AlertServiceImpl` does not even have a field for `AlertHaltConfigRepository` (verified: `grep -n 'AlertHaltConfig' AlertServiceImpl.java` returns zero hits in the field-declaration block).

## Why it matters

- **Operator surprise.** The UI gives Distribution Anomaly a halt toggle identical to the other three types. The OpenAPI schema (`DataEntityAlertConfig.distribution_anomaly_halt_until`) is exposed in the public spec. Documentation we are about to ship (DOC-020) lists the toggle in the per-alert-type halt table. Setting the halt and then watching alerts continue to fire is a trust-eroding bug — operators will assume their config is broken before they assume the platform is.
- **Noisy-entity scenarios are exactly when halts matter.** Distribution anomaly alerts are typically high-volume (Prometheus AlertManager fires once per evaluation interval per entity). The halt mechanism exists specifically to mute a noisy entity for hours/days while the underlying issue is fixed. Distribution anomaly is the alert type with the most traffic and the least functional halt.
- **Silent failure.** No log message, no warning, no UI hint. The halt timestamp is written to the database; the UI shows the toggle as "active" until the timer expires; the alerts keep arriving. There is nothing to grep, nothing to alarm on.
- **The dead code at `AlertActionResolverImpl.java:187` makes the gap easy to miss in review.** A cursory reader sees `case DISTRIBUTION_ANOMALY -> haltCfg.getDistributionAnomalyHaltUntil()` and assumes the halt is wired up. It isn't — the switch is exhaustive against `AlertTypeEnum`, but the only callers of `toHalt(...)` pass `FAILED_JOB`, `FAILED_DQ_TEST`, or `BACKWARDS_INCOMPATIBLE_SCHEMA`.

## Suggested fix

Two options. Either is acceptable; (B) is closer to the existing pattern.

**(A) Inject the halt repository into `AlertServiceImpl` and apply the filter in `handleExternalAlerts`.**

```java
// AlertServiceImpl.java
private final AlertHaltConfigRepository alertHaltConfigRepository;

public Mono<Void> handleExternalAlerts(final List<ExternalAlert> externalAlerts) {
    if (CollectionUtils.isEmpty(externalAlerts)) { return Mono.empty(); }

    final Set<String> entityOddrns = externalAlerts.stream()
        .map(a -> a.getLabels().get("entity_oddrn"))
        .filter(Objects::nonNull)
        .collect(toSet());

    return alertHaltConfigRepository.getByOddrns(entityOddrns)
        .flatMap(haltConfigs -> {
            final LocalDateTime now = DateTimeUtil.generateNow();
            final List<ExternalAlert> notHalted = externalAlerts.stream()
                .filter(a -> {
                    final String oddrn = a.getLabels().get("entity_oddrn");
                    final AlertHaltConfigPojo cfg = haltConfigs.get(oddrn);
                    if (cfg == null) return true;
                    final LocalDateTime until = cfg.getDistributionAnomalyHaltUntil();
                    return until == null || !until.isAfter(now);
                })
                .toList();
            // ... existing alert-creation logic on notHalted ...
        });
}
```

**(B) Add a halt-aware "external alert action resolver" mirroring `AlertActionResolverImpl`'s pattern.** Reuse the existing `Map<String, AlertHaltConfigPojo> haltConfigs` shape so the halt-filter logic lives in one place. `handleExternalAlerts` becomes a thin orchestrator that fetches halt configs, builds actions, and filters the same way `AlertIngestionRequestProcessor` does for ingestion-driven alerts.

Either fix should also remove the `case DISTRIBUTION_ANOMALY` arm from `AlertActionResolverImpl.toHalt(...)` if it's no longer needed (or document it as defensive against future ingestion-driven DA paths).
