## What

`odd.platform-base-url` is a single configuration key, but the three `@Value` consumers in the platform inject it with **inconsistent inline defaults**:

- `notification/config/NotificationConfiguration.java:105` (email sender) → `:http://localhost:8080`
- `config/SlackMessageGeneratorConfiguration.java:15` (Slack sender) → `:http://localhost:8080`
- `integration/StaticArgumentMappingContext.java:16` (integration-parameter substitution) → `:http://your.odd.platform`

If an operator does not set the key, the same logical value resolves to `http://localhost:8080` for the notification senders and to the placeholder string `http://your.odd.platform` for integration substitution. The two defaults disagree about what the platform URL is.

## Where

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/integration/StaticArgumentMappingContext.java:14-20`:

```java
public StaticArgumentMappingContext(
    @Value("${odd.platform-base-url:http://your.odd.platform}")
    final String platformUrl
) {
    this.parameterToValue = Map.of(PLATFORM_URL_PARAM_NAME, platformUrl);
}
```

The two notification consumers, for comparison:

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/notification/config/NotificationConfiguration.java:105`:

```java
@Value("${odd.platform-base-url:http://localhost:8080}") final String platformHost,
```

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/config/SlackMessageGeneratorConfiguration.java:15`:

```java
@Value("${odd.platform-base-url:http://localhost:8080}")
final URL platformBaseUrl
```

`application.yml:208-209` declares the key but leaves it commented out (no platform-wide default), so the inline `@Value` defaults are the only thing that resolves when the operator does not set the key.

## Why it matters

- **Silent divergence between subsystems.** When the key is unset, an integration receives a different value than the notification senders for the same logical concept. Any feature that compares the two (or that depends on operators reasoning about a single "platform URL") sees inconsistent state.
- **The integration default is not even a URL the platform will ever serve from.** `http://your.odd.platform` is a placeholder that resolves DNS to nothing on a normal network. Operators who deploy integrations without notifications and forget to set `ODD_PLATFORM_BASE_URL` see integrations fail at runtime with a confusing target — there is no log on the platform side, since the platform only serializes the value into a parameter map and hands it to the integration runtime.
- **The doc-side caveat now warns operators**, but the underlying inconsistency is a code-quality issue: a single config key should have a single default. The doc fix mitigates the immediate operator confusion; this issue is the request to make the code self-consistent.

## Suggested fix

Two equivalent approaches; either fixes the inconsistency.

### Option A — single source of default in `application.yml`

Remove all three inline defaults and declare the canonical default once in `application.yml`:

```diff
 odd:
-#  platform-base-url:
+  platform-base-url: http://localhost:8080
```

```diff
- @Value("${odd.platform-base-url:http://localhost:8080}") final String platformHost,
+ @Value("${odd.platform-base-url}") final String platformHost,
```

```diff
- @Value("${odd.platform-base-url:http://localhost:8080}")
- final URL platformBaseUrl
+ @Value("${odd.platform-base-url}")
+ final URL platformBaseUrl
```

```diff
- @Value("${odd.platform-base-url:http://your.odd.platform}")
- final String platformUrl
+ @Value("${odd.platform-base-url}")
+ final String platformUrl
```

Trade-off: every consumer sees `http://localhost:8080` when the key is unset. That value is at least a *real* URL on the host machine — better than a placeholder string the platform never serves from — but it is still operator-unfriendly to default to anything when the value matters at runtime.

### Option B — fail-fast on unset

Drop all defaults and require the key:

```diff
- @Value("${odd.platform-base-url:http://localhost:8080}") final String platformHost,
+ @Value("${odd.platform-base-url}") final String platformHost,
```

…and the same for the other two consumers. Without `application.yml` providing a default, Spring Boot will fail startup with `IllegalStateException: Could not resolve placeholder 'odd.platform-base-url'`. Loud, deterministic, and prevents the silent-placeholder failure mode entirely.

Trade-off: breaks any local-development setup that currently relies on the `http://localhost:8080` default to "just work" without setting an env var. Worth the noise — local dev should set `ODD_PLATFORM_BASE_URL=http://localhost:8080` explicitly.

Either option is acceptable from the documentation side; the goal is to have one default value (or none), not three.

## How discovered

Found while implementing `DOC-060` in the ODD Team workspace (a backlog item to document `odd.platform-base-url` after `DOC-018` removed a stale mention). The consumer-read audit (Quality Bar #4) walks every `@Value` consumer of a documented key; that walk surfaced the third consumer in `StaticArgumentMappingContext` and immediately the inconsistent default. The doc-side caveat warning operators about the placeholder now ships alongside `DOC-060`; this draft tracks the upstream code adjustment.
