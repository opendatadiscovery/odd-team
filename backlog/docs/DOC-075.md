### Finding 1 — Kafka `broker_conf` default column wrong

`docs/integrations/collectors/odd-collector.md` (Spotlight: Kafka, line ~203) shows:

| Field | Type | Required | Default | Description |
|---|---|---|---|---|
| `broker_conf` | dict | yes | `{}` | Passed to `confluent_kafka.AdminClient`… |

`KafkaPlugin` in `odd-collector/odd_collector/domain/plugin.py:82-87` declares `broker_conf: dict` — a **required** field with **no default**. The Default column should read `—` (the convention used elsewhere in the same table for required fields without defaults), not `{}`. As written, an operator could reasonably interpret `Default: {}` as "I can leave it empty"; Pydantic will reject the config at startup if they do.

Compare the sibling row in the same table — `schema_registry_conf: Optional[dict] = {}` is correctly documented as `Required: no, Default: {}`.

### Finding 2 — Misleading "Enable AlertManager Notifications" cross-link

`docs/integrations/README.md:119` — the "Token and datasource registration" closing sentence reads:

> "see [Enable security → Ingestion authentication](../configuration-and-deployment/enable-security/README.md) for the production posture, and [Configure ODD Platform → Enable AlertManager Notifications](../configuration-and-deployment/odd-platform.md) for the broader `/ingestion/**` configuration."

`docs/configuration-and-deployment/odd-platform.md` does not contain a section named "Enable AlertManager Notifications". The closest section is "Prometheus AlertManager Integration" (line 538), which is about *receiving* AlertManager webhooks (a separate feature), not about configuring `/ingestion/**` endpoints. The actual `/ingestion/**` discussion in `odd-platform.md` lives at line 607 ("Security implications" — Spring Security whitelisting of `/ingestion/**`, `auth.ingestion.filter.enabled` semantics, etc.) but is presented as a sub-discussion of the AlertManager section, not as its own top-level section.

Two acceptable fixes:

1. **Drop the trailing clause.** The first link (`Enable security → Ingestion authentication`) already covers the production posture for `/ingestion/entities` auth; the AlertManager-related `/ingestion/alert/alertmanager` is unrelated to "data source registration" and probably doesn't belong in the hub's overview prose.
2. **Repoint with accurate text.** Change to e.g. "[Configure ODD Platform → Prometheus AlertManager Integration](../configuration-and-deployment/odd-platform.md#prometheus-alertmanager-integration)" — but only if the hub actually wants to cross-reference the AlertManager-side `/ingestion/alert/alertmanager` endpoint (which is a different topic from data source registration and arguably out of scope for the integrations hub).

Recommend option 1 — drop the trailing clause. It removes the misleading claim without inflating the hub's scope.

### Finding 3 — `docs/collector.md` is orphan in `SUMMARY.md` → bridge content unreachable, `/collector` returns 404

DOC-042 rewrote `docs/collector.md` from an empty file (`e69de29`) to a 15-line bridge ("The Collector documentation now lives under [Integrations](integrations/README.md) …") with five collector deep-link bullets. The intent (per the original AC line 131 in DOC-042) was: *"docs/collector.md is not deleted but reduced to a short bridge linking to integrations/, so existing external links don't 404."*

But `docs/collector.md` is **not in `SUMMARY.md`** — it never was, and DOC-042 did not add it. GitBook only generates public URLs for SUMMARY entries, so `/collector` returns 404 regardless of file content. Verified by `WebFetch https://docs.opendatadiscovery.org/collector` after PR #39 merge propagation: the rendered page is the GitBook 404 ("The URL `collector` does not exist"). Verified by `git show origin/main:docs/SUMMARY.md | grep -i 'collector\.md'` returning only the new `integrations/collectors/odd-collector.md` row (line 18) — no `[Collector](collector.md)` entry.

Operational impact is nil today: `/collector` was already 404 pre-merge (the file was empty and SUMMARY-orphan), so no external link breakage was introduced. But the bridge content as authored is **invisible to readers** — anyone landing on the supposed bridge URL hits a 404 page with no signpost to the Integrations hub.

Two acceptable resolutions:

**(a) Adopt** — add `* [Collector](collector.md)` to `SUMMARY.md` so the bridge renders as a public URL. Best slot is probably the legacy spot (right above or below the new `## Integrations` block) so readers following stale references find the redirect signpost. This makes the AC's "preserve external links" rationale actually work going forward.

**(b) Delete** — `git rm docs/collector.md`. There were never external links to preserve (file was empty, never in SUMMARY). The hub at `/integrations/integrations` and the Terms & Aliases row at `/main-concepts` already serve as canonical entry points. Deletion removes a misleading orphan and avoids cluttering SUMMARY with a one-off redirect entry.

**Recommend (b) — delete** unless the user knows of external-system links (blog posts, dashboards, runbooks) that hit `/collector`. The bridge content was authored under the assumption it would be reachable; since it isn't and never has been, the cheaper fix is removal.

### Finding 4 (RESOLVED 2026-04-26 in the Gate 8 closeout state commit) — DOC-042 implementation log "Affected live-site URLs" list was wrong (GitBook routing artifact)

> **Status: resolved — no action needed in this DOC-075 commit.** The state-side log amendment shipped alongside the verdict block in the Gate 8 closeout commit on `feature/docs-integrations-hub-state` (the same commit that flipped DOC-042 to `done`). Record retained below for audit-trail completeness.

`backlog/docs/DOC-042.md` lines 217–229 listed the affected live-site URLs as:

- `https://docs.opendatadiscovery.org/integrations` (hub landing)
- `https://docs.opendatadiscovery.org/integrations/collectors/odd-collector` (and four siblings)
- `https://docs.opendatadiscovery.org/integrations/push-adapters/odd-airflow-2` (and four siblings)
- `https://docs.opendatadiscovery.org/collector` (bridge — see finding 3)
- `https://docs.opendatadiscovery.org/main-concepts` (Terms & Aliases row)

Verified during `/review` Gate 8 closeout: only `/main-concepts` is at the listed URL. The other 11 sub-page URLs all 404. Working URLs are:

- `https://docs.opendatadiscovery.org/integrations/integrations` (hub body — 7 of 7 sections render)
- `https://docs.opendatadiscovery.org/integrations` is GitBook's section landing — only the lede + sub-page list visible, not the full hub body
- `https://docs.opendatadiscovery.org/integrations/integrations/odd-collector` (and the 9 other sub-pages)

Why: GitBook generates URLs from `SUMMARY.md` hierarchy, not file paths. The `## Integrations` section + the `[Overview](integrations/README.md)` page both have slug `integrations`, so the hub lands at `/integrations/integrations`. All 10 sub-pages are nested under `Overview` in SUMMARY (one level deeper, no separate `### Collectors` / `### Push adapters` headers), so they get `/<section>/<parent-slug>/<this-slug>` = `/integrations/integrations/<slug>`. The `collectors/` and `push-adapters/` folder names in the file paths are invisible to GitBook's URL generator.

Two acceptable resolutions:

**(a) Amend the log** — update `backlog/docs/DOC-042.md` lines 217–229 to reflect the actual live URLs. This is purely a state-side correction; no documentation-repo change. Makes the audit trail truthful for any future reviewer who tries to fetch the listed URLs.

**(b) Restructure SUMMARY** — split the SUMMARY entries into two sub-section headers:

```
## Integrations

* [Overview](integrations/README.md)

### Collectors
* [odd-collector (generic)](integrations/collectors/odd-collector.md)
* [odd-collector-aws](integrations/collectors/odd-collector-aws.md)
…

### Push adapters
* [odd-airflow-2](integrations/push-adapters/odd-airflow-2.md)
…
```

This would (probably — verify with a dry-run preview before merging) cause GitBook to generate URLs like `/integrations/collectors/odd-collector` and `/integrations/push-adapters/odd-airflow-2` — matching the file paths and the implementer's intent. It also cleans up the doubled `/integrations/integrations` form. Cost: one SUMMARY edit + one Gate 8 re-run + a temporary URL change for any pages that are already (or about to be) linked from elsewhere.

**Recommend (a) — amend the log** as the safer / cheaper fix unless the user wants the cleaner URL form. (b) is structurally nicer but takes a second deploy cycle to confirm GitBook respects the sub-section headers, and any external bookmarks already targeting `/integrations/integrations/...` would break (low risk — pages are <24h old).
