## Description

Editorial audit on 2026-05-03 surfaced an **internal contradiction** on `docs/configuration-and-deployment/odd-platform.md`: the **same** environment variable for `odd.platform-base-url` is rendered **two different ways on the same page**:

### Site A — Enable Alert Notifications section (line 449)

```
ODD_PLATFORM-BASE-URL={platform_url}
```

### Site B — Gmail SMTP example (line 498)

```
ODD_PLATFORM-BASE-URL=https://odd.your-domain.com
```

### Site C — Data Collaboration section (line 722)

```
ODD_PLATFORM_BASE_URL={odd_platform_base_url}
```

The same key (`odd.platform-base-url` in YAML) gets `ODD_PLATFORM-BASE-URL` with hyphens at sites A and B, but `ODD_PLATFORM_BASE_URL` with underscores at site C. Site C is the **correct** form.

### Other affected env vars on the same page (Alert Notifications section)

The hyphens-in-env-vars pattern is repeated for several other keys in the same Alert Notifications tab (lines 435-449):

| Documented form (hyphens) | Correct form (underscores) | YAML key |
|---|---|---|
| `NOTIFICATIONS_MESSAGE_DOWNSTREAM-ENTITIES_DEPTH` | `NOTIFICATIONS_MESSAGE_DOWNSTREAM_ENTITIES_DEPTH` | `notifications.message.downstream-entities-depth` |
| `NOTIFICATIONS_WAL_ADVISORY-LOCK-ID` | `NOTIFICATIONS_WAL_ADVISORY_LOCK_ID` | `notifications.wal.advisory-lock-id` |
| `NOTIFICATIONS_WAL_REPLICATION-SLOT-NAME` | `NOTIFICATIONS_WAL_REPLICATION_SLOT_NAME` | `notifications.wal.replication-slot-name` |
| `NOTIFICATIONS_WAL_PUBLICATION-NAME` | `NOTIFICATIONS_WAL_PUBLICATION_NAME` | `notifications.wal.publication-name` |
| `ODD_PLATFORM-BASE-URL` | `ODD_PLATFORM_BASE_URL` | `odd.platform-base-url` |

### Why the hyphenated form is broken

Standard POSIX shells (`sh`, `bash`, `zsh`, `dash`) restrict environment variable names to `[A-Za-z_][A-Za-z0-9_]*` — **hyphens are not allowed**. Operators copy-pasting the documented form into a shell will hit:

```
$ export NOTIFICATIONS_WAL_ADVISORY-LOCK-ID=100
bash: export: `NOTIFICATIONS_WAL_ADVISORY-LOCK-ID=100': not a valid identifier
```

Hyphenated env vars work in **some** environments (Docker `environment:` blocks in YAML, Kubernetes pod-spec env, `env -i` invocation), but the most common operator path — setting env vars in a shell or a `.env` file — fails.

Spring Boot's `RELAXED` binding rule is the underlying mechanism that makes both `ODD_PLATFORM_BASE_URL` and `odd.platform-base-url` resolve to the same property — when the env var is set with underscores, Spring's `SystemEnvironmentPropertySource` canonicalizes it to `odd.platform-base-url` automatically. The hyphenated form is unnecessary AND broken in shells.

### Cross-page consistency

The SAME page, in **other sections**, gets this right:
- Data Collaboration section (lines 715-722): all underscores. ✓
- Stale period (lines 791-793): `ODD_DATA_ENTITY_STALE_PERIOD`. ✓
- Activity-feed partitioning (lines 833-836): `ODD_ACTIVITY_PARTITION_PERIOD`. ✓
- Database section (lines 81-90): `SPRING_CUSTOM_DATASOURCE_URL` (note: YAML key is `spring.custom-datasource.url`, env var converts hyphens to underscores). ✓
- Logging (lines 997-1005, fixed by DOC-100): all underscores. ✓
- Attachment storage (lines 919-925): `ATTACHMENT_REMOTE_ACCESS_KEY`. ✓

Only the **Notifications section** (Site A + Site B above) and the (separate) `## Enable Alert Notifications` legacy block use the hyphenated form. Likely a single round of authoring that copied the YAML key directly into the env-var tab without applying the convention.

## Context

- **Why high (not medium)**: an operator copy-pasting the documented `ODD_PLATFORM-BASE-URL=...` into bash gets `bash: export: ... not a valid identifier` and the platform never gets the URL set. This is **broken examples** that fail in the most common operator workflow. The fact that the same page documents the SAME key correctly in another section makes this internal contradiction even more confusing — operators will pick one of the two forms based on which section they read first.
- **Why not critical**: the YAML form on the same page works; an operator who chooses YAML is unaffected. The damage is limited to the env-var path.
- **Recurring pattern**: every YAML-key-to-env-var rendering on the doc tree should follow the convention "convert dots and hyphens to underscores" (Spring Boot's RELAXED binding). Worth a Gate-9 extension: a `Config-env-var:` claim class in the SoT table, requiring the env-var form to be cited from the consumer code's `@Value` annotation OR from Spring Boot's RELAXED binding rules, not invented.
