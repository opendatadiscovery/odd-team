## Description

DOC-083 review (2026-05-03) surfaced two related findings that bound DOC-083's scope but warrant a follow-up item:

**1. Phase A regex was narrow.** The audit grep targeted `## API reference`, `## API surface`, `### API surface`, `## API endpoints`, `## HTTP API`. Concrete patterns it didn't catch:

- `docs/Features.md:97-102` — `### Query parameters` H3 under `## End-to-end Data Objects Lineage` documenting the `lineage_depth` and `expanded_entity_ids` query parameters for `GET /api/dataentities/{data_entity_id}/lineage/upstream` (and the mirror `/lineage/downstream`). This is request-schema content — the same content type as the operationId tables DOC-083 migrated, just expressed as parameter prose rather than an endpoint table.
- `docs/Features.md:104-106` — `### Group lineage` H3 documenting the `GET /api/dataentitygroups/{data_entity_group_id}/lineage` endpoint (purpose: return the lineage graph for the group's children). Endpoint-description content of the same shape — should be a row on the canonical home's `## Lineage` section, with a one-line pointer here.
- `docs/Features.md:163-167` — Alerting tab table with an "API endpoint" column listing `GET /api/alerts` (`getAllAlerts`), `GET /api/alerts/my` (`getAssociatedUserAlerts`), `GET /api/alerts/dependents` (`getDependentEntitiesAlerts`). Operation IDs and HTTP paths in a feature-page table — Gate 10 content-type signal.
- `docs/Features.md:226` — paragraph under `### Halt notifications per entity` carrying substantive endpoint content for `GET /api/dataentities/{data_entity_id}/alert_config` and `PUT /api/dataentities/{data_entity_id}/alert_config`: the four halt-timestamp request/response field names (`failed_job_halt_until`, `failed_dq_test_halt_until`, `incompatible_schema_halt_until`, `distribution_anomaly_halt_until`), the ISO-8601 `date-time` format requirement, the `null`-clears-the-halt semantics, and the `ALERT_HALT_CONFIG_UPDATED` activity-feed event emission. This is request/response-schema content embedded in feature prose — should consolidate on the canonical home's `## Alerts` section. **Important not-to-touch**: the H4 `#### Distribution anomaly halt is currently unenforced` (228-234) immediately after — that is Gate 3 known-limitation content (legitimately on a feature page) and stays put; the API path mention inside the warning is contextual to explaining the bug.
- `docs/Features.md:356-359` — under `### Term-to-term links` H3 (Dictionary terms feature), an embedded 2-bullet API list documents the term-to-term TermController endpoints: `POST /api/terms/{term_id}/term` (operation `addLinkedTermToTerm`) and `DELETE /api/terms/{term_id}/term/{linked_term_id}` (operation `deleteLinkedTermFromTerm`). Bullet-list endpoint embedding is the same Gate 10 drift class as the table embeddings DOC-083 migrated. Note: these are **distinct** from the queryexample↔term endpoints (`POST/DELETE /api/terms/{term_id}/queryexamples/{example_id}`) that the canonical home's `## Query Examples` section already documents — these are pure term-to-term linking endpoints that need a new home (a `## Glossary` / `## Terms` H2 on the canonical home).
- `docs/Features.md:575` — Query Examples teaser cites "the 11-endpoint API surface (CRUD / faceted search / per-entity and per-term linking)" — drift: the canonical home `## Query Examples` section actually carries **16 endpoints** (CRUD 5 + faceted search 5 + per-entity/per-term lookup-and-linking 6 per `api-reference.md:64-101`). The teaser count predates DOC-083's migration. Update to `16-endpoint` in the same Phase C sweep.
- `docs/Features.md` — additional inline endpoint references (alert reopen PUT at line 262, alert audit GET at line 274). Mostly contextual prose, but include in the Phase C sweep for one-line cross-link consideration.
- `docs/configuration-and-deployment/odd-platform.md:544` — `### Endpoint` H3 under `## Prometheus AlertManager Integration`. Single-endpoint description (POST `/ingestion/alert/alertmanager`) plus payload-shape JSON schema. Borderline content-type case — the surrounding `### Authentication` and `### Example AlertManager receiver configuration` sections are operator-deployment, but the endpoint description itself is API-reference-shaped. Judgment call: link from canonical home to the deployment context, or migrate the endpoint description to the canonical home and leave the operator-setup context here?

**2. Canonical home is sectioned by 6 feature areas; substantial API surfaces are not yet covered.** DOC-083 sectioned `developer-guides/api-reference` by the 6 areas it migrated content for (Data Collaboration, Directory, Integrations, Query Examples, Reference Data, Relationships). The platform exposes additional API surfaces with substantial endpoint counts:

- **Alerts** — at least 7 endpoints surfaced in Features.md spot-check: getAllAlerts / getAssociatedUserAlerts / getDependentEntitiesAlerts (the three tab queries at lines 165-167), alert_config GET + PUT (the halt-timestamp request/response schema currently at line 226 — four field names, ISO-8601 format, `null`-clears semantics, `ALERT_HALT_CONFIG_UPDATED` event), alert reopen PUT (line 262), alert audit GET (line 274). The canonical home's `## Alerts` section needs to own all seven plus the related semantics (RBAC gating, the manual-reopen guard, the housekeeping-driven cleanup).
- **Lineage** — Features.md:97-106 carries two embedded API-content H3s under `## End-to-end Data Objects Lineage`: (a) `### Query parameters` (request-parameter spec for `lineage_depth` and `expanded_entity_ids`); (b) `### Group lineage` (endpoint description for `GET /api/dataentitygroups/{id}/lineage`). Plus an inline mention of `GET /api/dataentities/{id}/lineage/upstream` / `/lineage/downstream` in the lead-in paragraph at line 99. All three pieces should consolidate on the canonical home's `## Lineage` H2; Features.md keeps a one-line pointer per H3 (anchors `#query-parameters` and `#group-lineage` preserved for external bookmarks).
- **Glossary / Terms** — at minimum the term-to-term linking endpoints (`addLinkedTermToTerm`, `deleteLinkedTermFromTerm`) currently embedded as bullets on Features.md:356-359 plus any other TermController endpoints (term CRUD, term-search, term-to-data-entity linking) from the OpenAPI spec.
- **Search / Tags / Owners / DataSources / DataEntities (CRUD)** — not surveyed in this review; OpenAPI spec at `odd-platform/odd-platform-specification/openapi.yaml` is the SoT for the full surface.

These are out of DOC-083's scope (which targeted embedded fragments to migrate), but they leave the canonical home incomplete relative to the platform's full API surface — a Gate 6 (bidirectional code↔doc) gap for the API reference content type.

**Why this matters now.** DOC-083 established the canonical home and the homing rule. The next API surface accretion (DOC-084 odd-tracing-gateway is a known one) and any future feature-page authoring need a fully-populated canonical home and a broadened audit regex so they don't re-introduce the drift Cornerstone 5 / Gate 10 catch.
