## Description

`POST /ingestion/alert/alertmanager` accepts Prometheus AlertManager webhook payloads (labels, generatorURL, startsAt) and is completely undocumented. Users running Prometheus/AlertManager stacks have no way to discover this native integration point and either reinvent it or miss the integration entirely.

## Acceptance Criteria

- [ ] Alerting documentation describes the AlertManager webhook integration
- [ ] The exact endpoint path `POST /ingestion/alert/alertmanager` is shown
- [ ] The expected payload shape is documented (fields: `labels`, `generatorURL`, `startsAt`)
- [ ] A minimal AlertManager receiver config example is included
- [ ] Authentication behavior is documented (works alongside `auth.ingestion.filter.enabled` — DOC-004 is the companion item)

## Context

- Finding: F-004 in `findings/docs-accuracy-feature-behavior/2026-04-20-alerting.md`
- Evidence: `AlertManagerController.java` lines 21-25, `ExternalAlert.java` payload model

## Platform-side follow-up filed

- `issues/odd-platform/PLT-002.md` — **Add OpenAPI spec for `POST /ingestion/alert/alertmanager` webhook endpoint.** The controller carries an explicit `// TODO: define OpenAPI spec based on alert provider contract` comment; the endpoint is absent from the spec and therefore missing from Swagger UI and the generated Java/TS clients. The doc-side caveat (payload shape authored in prose) ships here in DOC-019; the spec addition is the upstream work.
