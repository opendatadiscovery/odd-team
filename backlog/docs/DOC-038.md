## Description

The Lookup Tables (UI "Master Data") subsystem is partially documented on `main` (Features.md subsection) but lacks a dedicated page, SUMMARY.md entry, and RBAC permissions coverage. Scope of the feature itself:

- Service/repo/mapper: `DataEntityLookupTableService` and 13+ associated Java files
- Entity type: `LOOKUP_TABLE(24)` with dedicated search entrypoint
- UI: `MasterData/LookupTables.tsx`
- API: column definitions + row-level CRUD
- RBAC: 9 `LOOKUP_TABLE_*` permissions

### Why this is REJECTED, not "PASS-with-nit"

The "Management → Master Data → Lookup Tables" string is the operator's **first instruction** for finding the feature in the UI — it appears as the second sentence of the Lookup Tables page, the orientation sentence on the Master Data Management landing, the discovery sentence in the Features.md teaser, and the Terms & Aliases row that resolves searches. An operator who follows it will click into Management's submenu, fail to find Master Data, and either Slack the team or close the tab. This is exactly the "operator follows our guidance off a cliff" failure class CLAUDE.md calls out at the top — it is the same shape as the DOC-008 attachment-storage default that wiped production attachments, just less destructive: an instruction that contradicts the running code. Three of the four occurrences are in newly-authored content; the fourth is in a Terms & Aliases row that **this PR added**. Catching this at review is the whole purpose of Gate 6 / Gate 9, and the same untouched Features.md:85 sentence "*the Catalog tab, but in Query examples, Master Data, Management and Dictionary tabs as well*" was 30 seconds away from the implementer the entire time.
