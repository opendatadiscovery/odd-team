## Description

Earlier implementation batches produced dedicated pages for features that already had sections in `Features.md`, without cross-linking:

1. **Collector secrets backend** (`docs/configuration-and-deployment/collectors-secrets-backend.md`, from DOC-013) duplicates `Features.md` → "Alternative Secrets Backend".
2. **Server-to-server (S2S) authentication** (`docs/configuration-and-deployment/enable-security/authentication/s2s.md`, from DOC-003) duplicates `Features.md` → "Machine-to-Machine (M2M) tokens".

Root cause: the `docs/accuracy/config-options` scanner reads configuration pages and the `docs/accuracy/feature-behavior` scanner reads `Features.md`; neither cross-checks the other, and `/implement` did not require a pre-authoring duplication sweep. Result: two pages per feature, no cross-reference, conflicting aliases (S2S vs M2M; alternative vs collector secrets backend).

This item remediates the two known duplication pairs under the new protocol (pre-authoring sweep + Terms & Aliases) that was put in place by the 2026-04-22 retrospective.

## Context

- Trigger: user-reported confusion on 2026-04-22 that `/features#alternative-secrets-backend` and `/configuration-and-deployment/collectors-secrets-backend` describe the same feature with no cross-reference; same pattern for `/features#machine-to-machine-m2m-tokens` and `/configuration-and-deployment/enable-security/authentication/s2s`.
- User feedback: *"in general if we proceed with this approach we'll generate a lot of confusion, duplication and redundancy not saying about possible contradictions"*.
- This work item closes the retrospective loop for the **two already-shipped** duplicates. Prevention of future duplicates is handled by:
  - `/implement` step 3.5 — pre-authoring duplication sweep (see `.claude/skills/implement/SKILL.md`)
  - `scanners/docs/quality/duplication.md` — ongoing detection
  - `main-concepts.md` Terms & Aliases — user-facing alias index
- Related commit: `feature/doc-052-cross-link-duplicates` on the documentation repo (PR pending review).

## Implementation Notes

- The Features.md teaser is not a stub — it needs to convey enough context that a reader scanning `Features.md` understands what the feature is and when they would care, before following the link.
- For M2M in particular, the teaser has to distinguish **user session tokens** (interactive login) from **S2S API keys** (programmatic clients) — these are often conflated by readers who see "token" and assume one mechanism.
- The dedicated-page alias mentions go in the **opening paragraph** so readers arriving via search find the alias immediately, not buried deep in the page.
