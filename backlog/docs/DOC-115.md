## Description

Baseline per-page audit on 2026-05-03 (first run of the doc-quality framework, axes A1 + A3 + A8 cross-checked between `main-concepts.md` and `Architecture.md`) surfaced an **unresolved canonical-home question** that DOC-114 and DOC-099 both depend on but neither resolves explicitly.

### The conflict

Both pages currently carry overlapping conceptual content:

- **`main-concepts.md`** has `## The architecture chain` (lines 11-27) with full prose definitions of Data source, Adapter, Plugin, Collector, Push adapter, ODD Platform, plus the pull-path / push-path diagrams. Plus `## Push vs Pull strategies`. Plus `## Adapter, Plugin, Collector, Push adapter` table. Plus Terms & Aliases rows for those four terms. (DOC-114 catalogues this 4× redundancy and proposes a restructure.)
- **`Architecture.md`** has the architecture diagram + 17 lines of producer-side concept descriptions, using "Push-client" as primary term (where main-concepts.md treats "Push-client" as an alias of "Push adapter") and "ODD Collector" (where main-concepts.md uses "Collector"). (DOC-099 catalogues this sparseness + vocabulary drift and proposes either expansion or merging.)

Both pages are at SUMMARY top level as primary peers (Overview / Main Concepts / Architecture / ODDRN / Features / ...). Both claim canonical-home status for the same conceptual content. **Neither DOC-114 nor DOC-099 explicitly decides which page wins** — DOC-114 proposes restructuring main-concepts.md without saying whether Architecture.md absorbs the content or stays separate; DOC-099 proposes restructuring Architecture.md without saying whether main-concepts.md cedes the architecture-chain section.

This decision must be made BEFORE either DOC-114 or DOC-099 is implemented, otherwise the two restructures conflict.

## Failure shape

**Canonical-home question** + **A1 (canonical-home discipline)** violation between two pages claiming the same role. (Per `pillars/documentation/quality-framework.md` axis A1.)

## Acceptance Criteria

This item resolves the structural decision; the resolution is then implemented by DOC-114 + DOC-099. Recommended decision options to evaluate:

- [ ] **Option A (main-concepts.md is canonical, Architecture.md is the diagram-and-pointer page)**: keep `## The architecture chain` on `main-concepts.md` as the canonical conceptual home with prose definitions. `Architecture.md` becomes a one-section page carrying ONLY the diagram + a one-line pointer ("For the producer-side vocabulary, see [Main Concepts → The architecture chain](main-concepts.md#the-architecture-chain)"). Architecture.md's IA placement (top-level peer) becomes defensible because it's the canonical home for the diagram (a content type distinct from prose definitions). DOC-114 then removes the redundant `## Push vs Pull strategies` and `## Adapter, Plugin, Collector, Push adapter` table sections and reduces Terms & Aliases rows to alias-only.
- [ ] **Option B (Architecture.md is canonical, main-concepts.md links to it)**: move the architecture-chain prose definitions FROM main-concepts.md TO Architecture.md. Architecture.md becomes a richer page (diagram + producer-side concept definitions + pull/push paths). main-concepts.md's `## The architecture chain` section becomes a one-line pointer to Architecture.md. DOC-114 still removes the redundant sub-sections on main-concepts.md; DOC-099 expands Architecture.md with the migrated content + canonical vocabulary.
- [ ] **Option C (merge main-concepts.md and Architecture.md)**: delete Architecture.md entirely; keep main-concepts.md as the single canonical home with the diagram embedded inline. SUMMARY.md drops the Architecture entry. Main Concepts at SUMMARY top level absorbs the architecture role.
- [ ] **Decide and document the chosen option** in this item (or as a draft ADR if the decision warrants the ADR format). Then update DOC-114 and DOC-099's acceptance criteria to align with the chosen option BEFORE either is implemented.
- [ ] Sources footer for the implementing commit: `Sources: none (canonical-home structural decision; no factual claim)`.

## Context

- **Why high (not medium)**: this is a blocking decision for two existing high-priority items (DOC-114 + DOC-099). Without a decision, the two implementations could conflict (main-concepts.md restructure removes the architecture-chain section while Architecture.md's restructure assumes main-concepts.md still has it, OR vice versa).
- **Why not critical**: the conflict surfaces at implementation time, not at reader time; readers today get redundant content but no broken behavior.
- **Recurring pattern**: when two pages both claim canonical-home status for the same content type, an explicit decision is required BEFORE restructuring either. The framework's axis A1 (canonical-home discipline) catches this at audit time; the catch is what surfaced this finding.

**Recommendation (preliminary)**: Option A. Rationale — `main-concepts.md` is already the canonical home for terminology (Terms & Aliases registry); extending it to also be the canonical home for conceptual definitions is consistent. `Architecture.md` carrying only the diagram + a pointer is honest about its role (a visual rather than a vocabulary page) and justifies its top-level SUMMARY placement (the diagram is a content type distinct from prose). But the user owns this decision; surfacing it for choice rather than implementing unilaterally.

## Decision (2026-05-04, batch `feature/docs-coherence-closeout`)

**Chosen: Option A** — `main-concepts.md` is the canonical home for the architecture-chain prose definitions; `Architecture.md` is reduced to the diagram-and-pointer page.

### Why Option A wins

1. **Consistency with the existing canonical-home discipline.** `main-concepts.md` already owns the Terms & Aliases registry — making it the canonical home for the conceptual definitions the registry's rows reference is the cohesive choice. A reader who lands on a Terms & Aliases row pointing at "see [Architecture.md] for the canonical four-term distinction" then navigates back to a thin diagram page experiences a worse reader-flow than landing on the prose right next to the registry.
2. **Architecture.md's role becomes honest.** A 17-line page that claims top-level SUMMARY peer status with `Main Concepts` cannot defend that placement on prose volume. Reframing the page around the **diagram** (a content type distinct from prose definitions) makes top-level SUMMARY placement defensible: the diagram is the visual anchor that prose cannot substitute for.
3. **Lowest restructure cost.** Option A requires DOC-114 to delete redundant sections of `main-concepts.md` and DOC-099 to thin `Architecture.md` to its diagram + pointer. Option B requires migrating prose between pages (higher risk of broken cross-links). Option C deletes a top-level SUMMARY entry — the largest IA shape change of the three.
4. **Vocabulary reconciliation lands on Architecture.md naturally.** Architecture.md currently uses "Push-client" and "ODD Collector" — DOC-099 Phase B was going to extend `main-concepts.md` Terms & Aliases to capture client-server framing. With Option A, `Architecture.md`'s thin form carries a one-line caveat ("for the producer-side mapping vocabulary used elsewhere in this manual, see [Main Concepts → The architecture chain](main-concepts.md#the-architecture-chain)") and the Terms & Aliases extension on `main-concepts.md` (DOC-099 Phase B) does the cross-framing work. No prose lives in two places.

### Direction for DOC-114 (main-concepts.md restructure)

- Keep `## The architecture chain` (lines 11-27) as **the canonical conceptual home**.
- Delete `## Push vs Pull strategies` (lines 35-44). Fold the "Pick pull when… Pick push when…" sentences into `## The architecture chain` as a closing paragraph.
- Delete `## Adapter, Plugin, Collector, Push adapter` table (lines 46-57). The four-row table re-explains the prose definitions; commonly-conflated framing is editorial gloss.
- Reduce Terms & Aliases rows for Adapter / Plugin / Collector / Push adapter (lines 99-102) to **alias-only** form: canonical term + aliases + one-line summary + link to `#the-architecture-chain`. Full definitions live on `## The architecture chain`.
- Sweep cross-references: `grep -rn 'main-concepts.md#' docs/` — re-target any inbound link that hits `#push-vs-pull-strategies` or `#adapter-plugin-collector-push-adapter` to `#the-architecture-chain`.

### Direction for DOC-099 (Architecture.md restructure)

- Phase A (Collector category-vs-implementation conflation) — fold into the thin form. Architecture.md's prose collapses to the diagram + a one-paragraph framing of pull-vs-push at the topology level + a one-line pointer to `main-concepts.md#the-architecture-chain` for the canonical definitions.
- Phase B (vocabulary extension to capture client-server framing) — implement on `main-concepts.md` Terms & Aliases as planned. Add the **Push-client (framing: client-server topology)** row that complements the producer-side **Push adapter** row. Optional: add **Pull-client** if `grep -rn 'pull-client' docs/` finds usage.
- Phase C (coverage extension) — *resolved as "thin index"*. Architecture.md does NOT expand to a real overview page; it carries the diagram + topology paragraph + pointer. The reader flow is: SUMMARY → Architecture (visual + topology framing) → main-concepts.md (vocabulary deep-dive).
- Phase D (Sources footer) — cite the new Push-client Terms & Aliases row (added by Phase B) and the deletion of the "ODD Collector" sentence.

### Sequence

They can ship in either order or in the same commit — neither modifies the other's primary surface (DOC-114 = main-concepts.md only; DOC-099 = Architecture.md + Terms & Aliases extension on main-concepts.md). Recommended: same batch, two commits, in either order. Live-site verification fetches both `main-concepts` and `architecture` URLs in one pass.
