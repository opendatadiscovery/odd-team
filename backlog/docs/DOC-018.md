## Description

`odd-platform.md` line 287 lists `NOTIFICATIONS_RECEIVERS_SLACK_PLATFORM-BASE-URL` as an environment variable. This variable doesn't exist — no code binds it, and setting it is silently ignored. Users copying the example believe they've configured the Slack base URL when they haven't.

The only Slack-receiver variable that exists is `NOTIFICATIONS_RECEIVERS_SLACK_URL` (property `notifications.receivers.slack.url`), which takes the **Slack incoming webhook URL** (`https://hooks.slack.com/services/…`). There is no nested `notifications.receivers.slack.*` binding for a platform base URL.

The clickable links that appear inside Slack notification messages (e.g. `{base}/dataentities/{id}/overview`) are built from a **separate**, top-level property: `odd.platform-base-url` → `ODD_PLATFORM_BASE_URL`. This same variable is **shared** with the email notification sender — both `SlackMessageGenerator` and `EmailNotificationSender` read from the same `odd.platform-base-url` binding. It is not Slack-specific, which is why it lives outside the `notifications.receivers.*` tree.

The bogus line likely originated from someone assuming "Slack also needs the platform URL → therefore it must nest under `notifications.receivers.slack`". The code does not work that way.

## Acceptance Criteria

- [ ] The `NOTIFICATIONS_RECEIVERS_SLACK_PLATFORM-BASE-URL` line is removed from the env var section (line 287 of `odd-platform.md`)
- [ ] No YAML example still shows the nonexistent `notifications.receivers.slack.platform-base-url` key
- [ ] The `ODD_PLATFORM_BASE_URL` entry remains (already present at line 294) and is explicitly referenced from the Slack setup instructions as the var that drives clickable links inside Slack messages
- [ ] The doc states that `ODD_PLATFORM_BASE_URL` is **shared** across Slack and email notifications (and any other sender that embeds deep links), so operators understand they only set it once
- [ ] The doc states the default for `odd.platform-base-url` (`http://localhost:8080`) and warns that the default produces unusable links in any non-local deployment
- [ ] The minimal Slack setup is shown end-to-end (enabled flag, WAL config, `NOTIFICATIONS_RECEIVERS_SLACK_URL`, `ODD_PLATFORM_BASE_URL`) so the reader doesn't have to stitch fragments together

Keep the fix in one place — don't split the Slack webhook var and the base-URL reference across distant sections, or the next reader will reinvent the same mistake. A short note like "Slack notification messages link back to the platform using `ODD_PLATFORM_BASE_URL` (shared with email notifications)" next to the `NOTIFICATIONS_RECEIVERS_SLACK_URL` line is enough.
