## What

`GenAIProperties` (`@ConfigurationProperties("genai")` POJO) declares three fields — `enabled`, `url`, `requestTimeout` — with **no field-level defaults**. When an operator sets only `genai.enabled=true` and leaves the other two unset (the natural happy-path "let me try this" first move), the platform builds the `genAiWebClient` Spring bean with:

* `baseUrl` = the empty string (Java `String` field default), so every outbound `POST /query_data` is sent to a malformed URL.
* `responseTimeout` = `Duration.ofMinutes(0)` (Java primitive `int` field default), so every request times out **immediately**, before the external AI service has a chance to respond.

The platform accepts the misconfiguration at startup with no warning, and surfaces it only at the first `POST /api/genai/ask` call as a `GenAIException` wrapping a connection or timeout error. Neither error message points at the missing config.

The example values shown in `application.yml` (`# url: http://localhost:5000` and `# request_timeout: 2`) **are commented out** — they look like defaults to a casual reader but never reach the bean.

## Where

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/config/properties/GenAIProperties.java`:

```java
@ConfigurationProperties("genai")
@Data
public class GenAIProperties {
    private boolean enabled;
    private String url;
    private int requestTimeout;
}
```

— three plain Java fields, no `= ...` initialisers, no `@DefaultValue` annotations, no JSR-303 validation.

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/config/WebClientConfiguration.java:20-30` consumes those values eagerly at bean-creation time:

```java
@Bean("genAiWebClient")
public WebClient webClient() {
    final HttpClient httpClient = HttpClient.create()
        .responseTimeout(Duration.ofMinutes(genAIProperties.getRequestTimeout()));
    final ClientHttpConnector connector = new ReactorClientHttpConnector(httpClient);

    return WebClient.builder()
        .clientConnector(connector)
        .baseUrl(genAIProperties.getUrl())
        .build();
}
```

— so the misconfiguration is baked into the WebClient at startup; changing the values later requires a Platform restart.

`odd-platform-api/src/main/resources/application.yml:17-20`:

```yaml
genai:
  enabled: false
#  url: http://localhost:5000
#  request_timeout: 2
```

— the explicit `enabled: false` is the only key actively reaching the bean; the other two are commented suggestions.

## Why it matters

* **Operator path is misleading.** "Let me turn on `genai.enabled` to see what happens" doesn't fail at startup; it fails silently at the first `/api/genai/ask` call with an error that doesn't mention the missing config. Time-to-diagnose is high.
* **Documentation cannot fully save the operator.** The new `docs/genai.md` page (DOC-045, shipped 2026-04-30) flags this as a Known Limitation with a warning admonition, but the doc-side fix only helps operators who read the doc — many will set `genai.enabled=true` from memory based on the application.yml comments.
* **Severity is medium** — the misconfiguration doesn't lose data or expose anything; it just makes a beta-feature's first-touch experience confusing. Severity rises to **high** if a downstream service depends on `/api/genai/ask` returning successfully (an internal AI assistant would gate user-facing features on the platform's response, and the immediate-timeout path would surface as a flaky symptom).

## Suggested fix

Two complementary changes — either (or both) closes the gap:

**Option A — declare safe defaults on the POJO.**

```java
@ConfigurationProperties("genai")
@Data
public class GenAIProperties {
    private boolean enabled = false;
    private String url = "";
    private int requestTimeout = 2;   // minutes — match the value already shown in application.yml
}
```

— at minimum, default `requestTimeout` to a non-zero value so an enabled-but-otherwise-unconfigured deployment fails on the URL (a more diagnosable error than an immediate timeout). The `application.yml` commented examples become real defaults.

**Option B — fail-fast validation at startup.** Add JSR-303 / `@Validated` constraints to the POJO so the application context fails to start when `enabled=true` but `url` is blank or `requestTimeout <= 0`:

```java
@ConfigurationProperties("genai")
@Data
@Validated
public class GenAIProperties {
    private boolean enabled;

    @NotBlank(groups = WhenEnabled.class)
    private String url;

    @Positive(groups = WhenEnabled.class)
    private int requestTimeout;
}
```

(or a custom `@AssertTrue` post-binding check — the exact mechanism is up to the maintainer; the principle is "refuse to start with broken GenAI config rather than fail at first request").

Option A is the smaller change; Option B is more defensive and matches the pattern other parts of the platform use (e.g., `S2sTokenProvider` validation pattern). Either is a 1-2 line addition.
