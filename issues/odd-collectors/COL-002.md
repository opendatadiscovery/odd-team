## What

The `odd-collectors` repo's top-level `README.md` advertises supported adapters in a series of "Supported data sources" bulleted lists. For the **generic `odd-collector`** package, that list is **out of date** relative to the actual `PLUGIN_FACTORY` registration map in `odd-collector/odd_collector/domain/plugin.py`.

Four adapters are present in the code (registered, working) but missing from the README's `# odd-collector` block list:

- `databricks` — `DatabricksPlugin` (`plugin.py:318-322`, `PLUGIN_FACTORY` line 386)
- `couchbase` — `CouchbasePlugin` (`plugin.py:303-310`, `PLUGIN_FACTORY` line 384)
- `opensearch` — `OpensearchPlugin` (`plugin.py:337-346`, `PLUGIN_FACTORY` line 390)
- `oracle` — `OraclePlugin` (`plugin.py:248-253`, `PLUGIN_FACTORY` line 377)

The Azure section also drifts: the README's `# odd-collector-azure` list shows three sources (`Azure SQL`, `Blob Storage`, `PowerBI`); the code adds a fourth — `azure_data_factory` — registered as `DataFactoryPlugin` in `odd-collector-azure/odd_collector_azure/domain/plugin.py:49-55` and exposed through `PLUGIN_FACTORY`.

## Where

`README.md` at the repo root, "Supported data sources" lists under each `# odd-collector*` section. Compare against the corresponding `domain/plugin.py` `PLUGIN_FACTORY` constant in each subpackage:

- `odd-collector/odd_collector/domain/plugin.py:349-391` — 41 PLUGIN_FACTORY keys (README's generic list shows ~33 distinct entries — `airbyte`, `cassandra`, `ckan`, `clickhouse`, `cockroachdb`, `couchbase` MISSING, `cubejs`, `databricks` MISSING, `dbt`, `druid`, `duckdb`, `elasticsearch`, `feast`, `fivetran`, `hive`, `kafka`, `kubeflow`, `metabase`, `mlflow`, `mode`, `mongodb`, `mssql`, `mysql`, `neo4j`, `odbc`, `odd_adapter`, `opensearch` MISSING, `oracle` MISSING, `postgresql`, `presto`, `redash`, `redshift`, `scylladb`, `singlestore`, `snowflake`, `sqlite`, `superset`, `tableau`, `tarantool`, `trino`, `vertica`).
- `odd-collector-azure/odd_collector_azure/domain/plugin.py:57-62` — 4 PLUGIN_FACTORY keys (README's Azure list shows 3 — `azure_data_factory` MISSING).

The AWS and GCP sections appear to match their `PLUGIN_FACTORY` maps at the time of writing.

## Why it matters

- Operators who land on the README's table assume the listed adapters are the complete set. Real adapters that they could use go undiscovered.
- Documentation drift in the canonical-source-of-truth README erodes trust in the broader project — the same readers go on to other ODD docs assuming the same risk applies there.
- Severity is **low** — the adapters work, no operator is blocked by this; the README is just under-advertising. Concrete consequence: 2026-04-26 ODD Team docs work (DOC-042 — the user-facing Integrations hub) had to flag the README drift as a "Known limitation" and pin the inventory to `PLUGIN_FACTORY` in `plugin.py` to avoid propagating the gap into `docs.opendatadiscovery.org`.

## Suggested fix

1. Replace each per-section "Supported data sources" bulleted list with a regenerated version that mirrors the keys of `PLUGIN_FACTORY` in the corresponding subpackage's `domain/plugin.py`. A simple dev-time script (or a CI check) can compare the two on every release.
2. As a follow-on hardening: add a release-time CI step that diffs the README's bullet lists against `PLUGIN_FACTORY.keys()` for each subpackage and fails the release if they don't match. This prevents the same drift from accumulating again.
