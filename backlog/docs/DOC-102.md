## Description

Editorial audit on 2026-05-03 (during `/review batch:feature/docs-editorial-audit-closeout`) surfaced a **cross-link mismatch** on `docs/integrations/integration-wizard.md` at two sites:

### Site A — line 35

```
…see [Configure ODD Platform → odd.platform-base-url](../configuration-and-deployment/odd-platform.md#enable-alert-notifications) for the full caveat about this default.
```

### Site B — line 64

```
* [Configure ODD Platform → `odd.platform-base-url`](../configuration-and-deployment/odd-platform.md#enable-alert-notifications) — set this before opening the wizard so static `platform_url` substitution carries a working URL.
```

Both links promise the reader a section about `odd.platform-base-url` specifically. The anchor `#enable-alert-notifications` lands on the broader `## Enable Alert Notifications` H2 — a long section covering the entire alert-notification subsystem (PostgreSQL replication, ODD Platform configuration, Gmail SMTP example, known limitations, cleanup). The reader has to scroll/search inside that section to find the bullet about `odd.platform-base-url` (line 392 of `odd-platform.md`).

The deeper structural question: `odd.platform-base-url` is a **platform-wide** key consumed by 3+ subsystems (alerts, email, integration-substitution context — explicitly called out in odd-platform.md line 392 and again in the H2 `## Platform-level settings (odd.*)` line 770). Its canonical home should be the Platform-level settings section, not buried inside Alert Notifications. Currently the Platform-level settings H2 (line 770) carries a back-pointer to the alert-notifications section: "A fifth key in the same namespace, `odd.platform-base-url`, is documented above in [Enable Alert Notifications](#enable-alert-notifications)…" — confirming the canonical-home placement is conscious but suboptimal.

## Failure shape

**Cross-link mismatch** + **canonical-home question** (per `playbooks/doc-product-editorial-read.md` failure-shape table).

## Acceptance Criteria

Two paths — pick (A) for the lower-risk fix, or (B) for the structural improvement:

- [ ] **Path A (re-anchor only)**: add a dedicated H3 anchor for `odd.platform-base-url` inside `## Enable Alert Notifications` (e.g., `### odd.platform-base-url (platform-wide URL)` adjacent to line 392's bullet) and re-target the two integration-wizard.md cross-links to that precise anchor. Sweep `grep -rn 'odd-platform.md#enable-alert-notifications' docs/` to verify all inbound cross-links point at the right anchor.
- [ ] **Path B (structural)**: move the canonical content of `odd.platform-base-url` from its bullet inside `## Enable Alert Notifications` (line 392) to a dedicated H3 inside `## Platform-level settings (odd.*)` (line 770). Replace the original bullet with a short pointer ("`odd.platform-base-url` — see [Platform-level settings → odd.platform-base-url](#platform-level-settings-odd) for the full reference"). Update integration-wizard.md cross-links to target the new home. Update the back-pointer in line 770's intro paragraph.
- [ ] Live-site verification (Gate 8 / `/review`): WebFetch `https://docs.opendatadiscovery.org/integrations/integration-wizard` and confirm the link from "odd.platform-base-url" lands on a section that visibly documents that specific key.
- [ ] Sources footer: `Sources: none (cross-link cleanup, no factual claim)` for Path A; for Path B, `Sources: Config: odd-platform-api/.../*Properties.java consumer for odd.platform-base-url`.

## Context

- **Why high (not medium)**: cross-link mismatch confuses readers and the canonical-home structural issue is recurring — every future maintainer landing on the integration-wizard.md page hits this same defect.
- **Why not critical**: the link does land on a section that contains the content; the reader's path is impaired but not blocked.
- **Recurring pattern**: shared keys (`odd.platform-base-url`, `auth.ingestion.filter.enabled`) consumed by multiple subsystems tend to be documented in whichever subsystem's section first covered them. Worth a Cornerstone-5 decision: shared-key documentation should live in the section dedicated to the canonical namespace (here `## Platform-level settings (odd.*)`), with one-line pointers from the consuming subsystem sections.
