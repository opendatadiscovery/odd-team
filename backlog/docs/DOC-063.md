## Description

`docs/main-concepts.md` currently describes the integration chain as **"Data source → Adapter → Collector (or Push-client) → ODD Platform"** and names three producer-side abstractions (Adapter, Collector, Push-client) without surfacing **Plugin** at all. The user's canonical definitions — which are the authoritative wording and now live in `navigation/architecture.md` — make four distinctions the current page misses:

1. An **adapter** is the *mapping code* (extract-and-map from source to spec). It is push or pull. It is **not** a deployment unit by itself.
2. A **collector** is a *container of pull adapters* + supporting runtime (launcher, logging, Platform-API client, config reader). A collector is what you deploy. "Pull adapter" is not a synonym for "collector".
3. A **plugin** is a *configured adapter instance inside a collector* — one plugin carries one adapter's connection/schedule settings. A single collector hosts many plugins (e.g., two PostgreSQL plugins pointing at different hosts + one MySQL plugin).
4. A **push adapter** (alias: **push-client**) is a *push-strategy adapter embedded in the source's runtime* (dbt plugin, GE checkpoint action, Airflow plugin, Spark listener). Still an adapter — the difference from a pull adapter is deployment topology.

Landing this vocabulary in `main-concepts.md` is a Gate 9 prerequisite for the Integrations hub (DOC-042). Once this item ships, DOC-042 can reference the canonical terms without redefining them.

## Scope clarifier

This item updates three areas of `main-concepts.md` and nothing else:
