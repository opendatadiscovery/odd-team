## Description

`POST /ingestion/alert/alertmanager` accepts Prometheus AlertManager webhook payloads (labels, generatorURL, startsAt) and is completely undocumented. Users running Prometheus/AlertManager stacks have no way to discover this native integration point and either reinvent it or miss the integration entirely.

## Acceptance Criteria

- [ ] Alerting documentation describes the AlertManager webhook integration
- [ ] The exact endpoint path `POST /ingestion/alert/alertmanager` is shown
- [ ] The expected payload shape is documented (fields: `labels`, `generatorURL`, `startsAt`)
- [ ] A minimal AlertManager receiver config example is included
- [ ] Authentication behavior is documented (works alongside `auth.ingestion.filter.enabled` — DOC-004 is the companion item)

## Context

- Finding: F-004 in `findings/docs-accuracy-feature-behavior/2026-04-20-alerting.md`
- Evidence: `AlertManagerController.java` lines 21-25, `ExternalAlert.java` payload model

## Platform-side follow-up filed

- `issues/odd-platform/PLT-002.md` — **Add OpenAPI spec for `POST /ingestion/alert/alertmanager` webhook endpoint.** The controller carries an explicit `// TODO: define OpenAPI spec based on alert provider contract` comment; the endpoint is absent from the spec and therefore missing from Swagger UI and the generated Java/TS clients. The doc-side caveat (payload shape authored in prose) ships here in DOC-019; the spec addition is the upstream work.

## Review-ready notes (2026-04-24)

**What shipped** (commit `8896c46` on `feature/docs-odd-platform-config`, `documentation` repo):

1. **New H2 section in `odd-platform.md`**: "Prometheus AlertManager Integration" placed between "Enable Alert Notifications" (outbound) and "Enable Data Collaboration". Covers:
   - Exact endpoint path (`POST /ingestion/alert/alertmanager`) and response code (`204 No Content`).
   - Payload schema (JSON) with the three consumed fields (`labels`, `generatorURL`, `startsAt`), note that other AlertManager top-level fields are accepted-and-ignored.
   - **Warning admonition**: the `entity_oddrn` label is required for routing; alerts without it are silently orphaned. This is critical because the reference example (`docker/examples/config/alertmanager.yaml`) does not show it — operators copying the example verbatim would hit this.
   - Minimal AlertManager receiver config (YAML).
   - A Prometheus alerting-rule example showing how to attach `entity_oddrn` as a label.
   - Authentication section: points to the ingestion auth filter (`auth.ingestion.filter.enabled`) and cross-links to the S2S keys page.
2. **Features.md Alerting section**: one paragraph added naming the two alert origins (internal pipeline / AlertManager inbound) and cross-linking to the detailed config. Keeps Features.md as the feature index; detailed config stays in odd-platform.md.
3. **Alert type is `DISTRIBUTION_ANOMALY` (hardcoded)**: documented implicitly via "Each inbound alert becomes a **Distribution Anomaly** alert". The service layer hardcodes this and the generatorURL becomes part of the alert description with `moment_input` / `end_input` query params appended — documented in the prose.
4. **Platform-side follow-up filed** at `issues/odd-platform/PLT-002.md` covering the OpenAPI spec gap (the controller's own TODO). Not implemented in this session — upstream code work.
