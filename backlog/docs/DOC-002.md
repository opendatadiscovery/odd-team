## Description

The Keycloak provider in `application.yml` (line 127) supports a `pkce` config option (Proof Key for Code Exchange) but the Keycloak section of `oauth2-oidc.md` does not mention it. Users deploying Keycloak realms that require PKCE won't discover that the platform supports this setting.

## Acceptance Criteria

- [ ] Keycloak section of `oauth2-oidc.md` documents the `pkce` option (type, default, when to set)
- [ ] At least one code-flow example shows `pkce` in context
- [ ] A short explanation of when PKCE is required (public clients, confidential clients with PKCE enforcement in Keycloak) is included

**Caveats surfaced (shipped in the same commit):**
- **In-scope, shipped:** `pkce: true` combined with a non-empty `client-secret` has no effect on the OAuth2 client registration — Spring's authentication-method selection falls back to confidential-client (client-secret) authentication, and PKCE parameters are not added to the authorization request. Documented as a `hint style="warning"` admonition under the new "Keycloak with PKCE" subsection. (Gate 5 caveat-defaulted parameter for any provider that requires both PKCE enforcement AND a client secret — operator must choose public-client mode.)
- **Out-of-scope, not logged as upstream issue:** the platform does not actively reject the `pkce: true` + non-empty `client-secret` combination at startup. A defensive validator could fail-fast here. Severity is low (the doc caveat now warns operators); no PLT issue drafted.
