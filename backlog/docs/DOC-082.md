## Description

The user surfaced three concrete IA drifts on 2026-04-30 after reviewing the bigbatch (`feature/docs-bigbatch-2026-04-30`). Each is a SUMMARY.md placement that violates the rule "hierarchy depth must reflect conceptual depth" (CLAUDE.md Cornerstone 2, codified during the same review). All three are pre-existing drift that the bigbatch did not make worse but did not fix either; this item closes the gap.

### Drift 1 — `Directory` is at SUMMARY top level next to `Main Concepts` and `Features`

Current placement (`docs/SUMMARY.md` line 8, added by DOC-047 in the bigbatch):

```
* [Features](Features.md)
* [Directory](directory.md)              ← top level
* [Data Modelling](data-modelling.md)
  * [Query Examples](query-examples.md)
  * [Relationships](relationships.md)
* [Master Data Management](master-data-management.md)
  * [Lookup Tables](lookup-tables.md)
```

**Problem:** Directory is a Data Discovery sub-feature — a browse-oriented complement to Search, both of which are Data Discovery primitives. Top-level placement implies peer status with `Main Concepts` (the catalog's foundational concept page) and the pillar landings, which it isn't.

**Fix:** Add a `Data Discovery` pillar landing (Cornerstone-2 sibling of `Data Modelling` and `Master Data Management`) and nest Directory under it. The pillar landing should also host links to the Search feature surface (currently a section in `Features.md`) and the Catalog Overview page concept. Resulting layout:

```
* [Features](Features.md)
* [Data Discovery](data-discovery.md)
  * [Search and Filtering](Features.md#search-and-filtering)   (or migrate to its own page)
  * [Directory](data-discovery/directory.md)                     (moved)
  * [Catalog Overview page](Features.md#catalog-overview-page)   (or migrate)
* [Data Modelling](data-modelling.md)
  ...
```

This mirrors the Cornerstone-2 IA shape codified in CLAUDE.md ("aspect landing per pillar in `main-concepts.md` Data Governance map").

### Drift 2 — `GenAI assistant` is at SUMMARY top level

Current placement (`docs/SUMMARY.md` line ~14, from PR #49 / DOC-045):

```
* [Master Data Management](master-data-management.md)
  * [Lookup Tables](lookup-tables.md)
* [GenAI assistant](genai.md)            ← top level
* [Use cases](Use_cases.md)
```

**Problem:** GenAI is one of several "active platform features" that operate on the catalog rather than describe it. Peers include Alerting, Notifications, Activity Feed, Data Collaboration — these all live as sections inside `Features.md` or `odd-platform.md` today, scattered across surfaces. Putting GenAI alone at the top is convenience-placement: the user sees one active feature elevated and the others buried.

**Fix:** Group all active-platform features under a shared parent. Two reasonable options — pick one:

- **Option A — new "Active platform features" group** with subpages:
  ```
  * [Active platform features](active-platform-features.md)   (new pillar landing)
    * [Alerting](active-platform-features/alerting.md)
    * [Notifications](active-platform-features/notifications.md)
    * [Activity Feed](active-platform-features/activity-feed.md)
    * [Data Collaboration](active-platform-features/data-collaboration.md)
    * [GenAI assistant](active-platform-features/genai.md)     (moved)
  ```
  Each subpage is created from the existing scattered content; the parent landing is a hub-style overview.

- **Option B — extend an existing pillar** if one fits (e.g., a "Catalog Operations" or "Catalog Workflows" landing). Would need to be reasoned through with the user — Cornerstone-2 IA shape question.

Option A is the cleaner pre-emption of Cornerstone-2 ("aspect landing per pillar"). Option B is lighter-weight if a fitting pillar already exists.

### Drift 3 — `Build a custom collector` is a sibling of the `Build and run` group

Current placement (`docs/SUMMARY.md` line ~62, in the Developer Guides section):

```
## Developer Guides
* [Build and run](developer-guides/build-and-run/README.md)
  * [Build and run ODD Platform](developer-guides/build-and-run/build-and-run-odd-platform.md)
  * [Build and run ODD Collectors](developer-guides/build-and-run/build-and-run-odd-collectors.md)
* [Build a custom collector](developer-guides/custom-collectors.md)   ← sibling of the group
* [GitHub organization overview](developer-guides/github-organization-overview.md)
* [How to contribute](developer-guides/how-to-contribute.md)
```

**Problem:** "Build and run" hosts platform + collector build instructions; "Build a custom collector" is the third peer in that conceptual group ("how do I build the standard X" + "how do I build a custom X"). Placing it next to the group rather than inside makes it look like a different conceptual level.

**Fix:** Move under `Build and run`:

```
* [Build and run](developer-guides/build-and-run/README.md)
  * [Build and run ODD Platform](developer-guides/build-and-run/build-and-run-odd-platform.md)
  * [Build and run ODD Collectors](developer-guides/build-and-run/build-and-run-odd-collectors.md)
  * [Build a custom collector](developer-guides/build-and-run/custom-collectors.md)   (moved)
```

The file should physically move to `developer-guides/build-and-run/custom-collectors.md` to match the GitBook URL slug under SUMMARY hierarchy (DOC-079 / DOC-081 lesson). Inline relative links + back-links elsewhere in `docs/` updated in the same commit.

## Context

- **Trigger:** user feedback 2026-04-30 immediately after the bigbatch review. The user said: "we have high level topics like 'Main Concepts' and then on the same level we have 'Directory' (that is actually a sub topic for Data Discovery from my point of view) and GenAI assistant that should be a sub topic of kind of 'General', that should include Alerting, Notifications, Actions (Audit), Collaboration features as a subpages."
- **Rule codified in same conversation:** CLAUDE.md Cornerstone 2 ("Hierarchy depth must reflect conceptual depth") + Quality Bar Gate 7 IA-sanity sub-rule + review SKILL.md Gate 7 protocol + implement SKILL.md pre-authoring placement check + memory entry `feedback_ia_hierarchy_sanity.md`.
- **Why this is its own item, not bundled into the existing changes:** the three drifts are pre-existing — the bigbatch inherited them but did not make them worse. Per the same Gate 7 rule ("Pre-existing drift that the current change inherits without making worse is not a per-change fail — log it as a separate DOC-NNN refactor item"), the right response is a dedicated refactor.
- **Scope:** Drift 2 (GenAI grouping) is the largest because Option A creates a new pillar landing and migrates ~5 pages-worth of content from existing surfaces. The user should pick Option A vs Option B before authoring begins. Drift 1 + Drift 3 are smaller: each is one new/moved page plus SUMMARY + back-link rewiring.

## Implementation Notes

- **Sequence the work.** Drift 3 is the smallest and lowest-risk (mechanical move + SUMMARY rewire + back-links). Ship it first to exercise the Gate 7 IA-sanity rule on a small change.
- **Pause on Drift 2.** Before authoring, surface the Option-A-vs-B decision to the user. Adding a pillar landing is a Cornerstone-2 decision that warrants explicit alignment.
- **Drift 1 is medium-effort.** Creating a `Data Discovery` pillar landing means deciding what its Overview says, what existing Search/Catalog Overview content moves into it (or stays cross-linked), and how it relates to `main-concepts.md`'s Data Governance map. The Cornerstone-2 IA shape gives the template (look at `data-modelling.md` for the existing pattern).
- **DOC-081 is adjacent** — it covers file-path-vs-URL-slug realignment for `lookup-tables.md`, `query-examples.md`, `relationships.md`. Some of those files may end up moving under their pillar parents anyway as part of this refactor; coordinate to avoid double-moving.
- **All three drifts should ship one-per-PR** (or at least one-per-commit on a single batch branch) so each fix is independently reviewable. Bundling them into one mega-commit makes the review verdict ambiguous.
- **Live-site verification is mandatory per drift** — moves of indexed-by-SUMMARY pages always carry GitBook URL slug risk (DOC-079 / DOC-081 lesson).
