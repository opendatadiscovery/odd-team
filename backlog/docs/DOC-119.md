## Description

Editorial audit during `/review batch:feature/docs-editorial-audit-closeout` (2026-05-04) surfaced a copy-paste-hazardous code block on `docs/developer-guides/build-and-run/build-and-run-odd-collectors.md` in the **`### Running ODD Collector on M1`** troubleshooting section.

### Site A — line 150 (three exports concatenated)

The current line 150 reads (as a single line in the markdown source):

```
export LDFLAGS="-L/opt/homebrew/lib  -L/opt/homebrew/Cellar/unixodbc/2.3.11/include -L/opt/homebrew/opt/freetds/lib -L/opt/homebrew/opt/openssl@3/lib"export CFLAGS="-I/opt/homebrew/Cellar/unixodbc/2.3.11/include -I/opt/homebrew/opt/freetds/include"export CPPFLAGS="-I/opt/homebrew/include -I/opt/homebrew/Cellar/unixodbc/2.3.11/include -I/opt/homebrew/opt/openssl@3/include"
```

Three distinct `export` statements (`LDFLAGS`, `CFLAGS`, `CPPFLAGS`) are glued on one line with no separator. An operator pasting this into a `.bashrc` / `.zshrc` (which the section explicitly recommends — "The easiest way is to add all export statements to your .bashrc/.zshrc file") gets a single broken statement: `LDFLAGS` is set to a value that contains the literal substring `"export CFLAGS=…"export CPPFLAGS=…"`, while `CFLAGS` and `CPPFLAGS` are never set at all. The next `pip install` against `confluent-kafka` will fail with the same compile errors the section is supposed to prevent.

### Site B — line 154 (export glued to comment and to next export)

The current line 154 reads (as a single line):

```
export PATH="/opt/homebrew/opt/openssl@3/bin:$PATH"# grpcio
```

The `# grpcio` is a section header comment that should be on its own line introducing the grpcio block. Glued to the end of the `export PATH=...` it makes `PATH` end with the literal substring `"...# grpcio"` (the `#` is inside the quoted value so it is **not** a comment). The `export GRPC_PYTHON_BUILD_SYSTEM_*` lines that follow are also separate statements that should have a blank-line + `# grpcio` heading above them.

### Side note — typo

Line 149 introduces the section as `# cunfluent-kafka` — typo for `confluent-kafka`. (DOC-113 already catalogues this typo class on other pages; this page's `cunfluent` is also covered under DOC-113's A12 sweep.)

## Why this slipped past prior batches

The block has been on the page since early ODD docs era. The defect is **not visible** when the markdown is rendered — GitBook's code-block syntax highlighter produces a visually-plausible single line that hides the missing newlines. A reader scanning the rendered output sees what looks like a wrapped command and doesn't realise it's a single broken statement until they paste it.

This is the same defect class as **DOC-100** (the Logging Settings env-var concatenation on `odd-platform.md:1004`) — two `KEY: value` assignments glued on one line. DOC-100 was discovered by the same editorial-audit playbook (`playbooks/doc-product-editorial-read.md`) on a different page; this finding is the second instance of the pattern. The recurring pattern is: code-block content authored once, copy-edited later by an editor who flattened newlines (intentionally or not), and the rendered output looks acceptable.

Editorial-audit case-law for `pillars/documentation/quality-framework.md` axis **A12** (Prose and rendering) — failure shape: **Rendering defect** (concatenated statements that should have newlines).

## Context

- **Failure shape**: Rendering defect (per `pillars/documentation/quality-framework.md` axis A12 derived shapes) — code block formatting that produces broken output when copy-pasted.
- **Why high (not medium)**: the section explicitly directs operators to "add all export statements to your .bashrc/.zshrc file"; an operator following that instruction with the current text gets a broken shell config that breaks subsequent `pip install` runs **silently** (the broken config doesn't error at shell-rc time; the `cflags`/`cppflags` are just empty when the next compile runs). Operationally damaging on a primary developer-onboarding path. Same severity class as DOC-100.
- **Why not critical**: no data loss / production-system impact; only developer-machine friction. Critical is reserved for active operator-harming defects in the production deployment path.
- **Recurring pattern**: same defect class as DOC-100 (concatenated YAML env-vars on `odd-platform.md`). Worth a Gate-7 / Gate-12 extension: every code block in the doc tree should be paste-tested before review. Log as a separate framework-level item if surfaced again.
