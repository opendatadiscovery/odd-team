## Description

Resolved alerts are silently deleted after 30 days by the `AlertHousekeepingJob`. The alerting documentation doesn't mention this, so users investigating historical alerts hit an unexplained data-disappearance wall. The underlying config is being documented by DOC-009; this item ensures the alerting section itself flags the behavior and links to the config.

## Acceptance Criteria

- [ ] Alerting section of Features.md notes that resolved alerts are automatically cleaned up after a configurable retention period (default 30 days)
- [ ] The note links (or references) the housekeeping config section covered by DOC-009
- [ ] The config key `housekeeping.ttl.resolved_alerts_days` is referenced by name

**Caveats surfaced and shipped in scope:**
- **Hard delete, not soft delete.** The housekeeping job removes `ALERT_CHUNK` first, then `ALERT`; once the retention window passes, the alert is gone from the database with no recovery path. Documented in the paragraph. Discovered via reading `AlertHousekeepingJob.java:36-43`.
- **Both `RESOLVED` and `RESOLVED_AUTOMATICALLY` are eligible.** Manual and auto resolutions age out on the same retention timer — there is no separate retention for system-resolved alerts. Documented.

**No upstream issues filed** — cleanup mechanism is implemented coherently; the operator-relevant caveat (raise the value before alerts age out) is captured in the doc.

**Sits inside the lifecycle section created by DOC-022** — this is by design; the `### Alert lifecycle: statuses, resolution, cleanup` heading was named with "cleanup" precisely so DOC-021's content lands as an `#### Auto-cleanup of resolved alerts` H4 within it.
