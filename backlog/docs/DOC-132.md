## Description

Editorial audit on `/review batch:feature/docs-bigbatch-2026-05-06` ran the DOC-129 magic-link sweep regex with **case-insensitive** + **multi-word** extensions and surfaced 6 residual sites the original DOC-129 sweep didn't catch.

### The defect class

DOC-129 closed 3 magic-link sites by rephrasing `[here]` / `[read]` to descriptive link text (Architecture.md L9, how-to-contribute.md L14, trylocally.md L80). Its sweep regex was `\[(here|read|click here|this|see|link|details|more)\]\(` — case-sensitive single-token match.

This catch missed two sub-shapes:

1. **Case-variant** — `[Here]` (capital H) at `docs/README.md:37`. The regex's `[here]` literal does not match `[Here]`.
2. **Multi-word** — `[Read more]` (two-word phrase) at five sites in `docs/use-cases.md:4,6,8,10,12`. The regex's `[read]` literal does not match `[Read more]`.

### The 6 magic-link sites

| File | Line | Defect |
|---|---|---|
| `docs/README.md` | 37 | `[Here](use-cases.md) you can find most popular use cases for different titles of your data team.` — link text "Here" doesn't name the target. |
| `docs/use-cases.md` | 4 | `[Read more](use-cases/dc-data-compliance.md)` — at the end of the "Data compliance for Data Scientists" use-case bullet. |
| `docs/use-cases.md` | 6 | `[Read more](use-cases/de-deprecation.md)` — at the end of the "Deprecation for Data Engineer / Analyst" use-case bullet. |
| `docs/use-cases.md` | 8 | `[Read more](use-cases/dq-visibility.md)` — at the end of the "Visibility for Data Quality Engineer" use-case bullet. |
| `docs/use-cases.md` | 10 | `[Read more](use-cases/viz-preparation.md)` — at the end of the "Data preparation for Visualization Engineer" use-case bullet. |
| `docs/use-cases.md` | 12 | `[Read more](use-cases/service-presales.md)` — at the end of the "Service Provider and Pre-Sales" use-case bullet. |

### Reader-experience cost

Same as DOC-129 / DOC-110 Site A: the link text doesn't describe the target. A reader scanning the page can't predict from the link text what they'll get. Rephrase so the link text names the destination ("the [Data compliance for Data Scientists use case](...)" / "the [GitHub organization overview](...)" / etc.).

## Failure shape

**A4 (Reference integrity) — magic-link sub-shape, case-variant + multi-word residue** (per `pillars/documentation/quality-framework.md` and `playbooks/doc-product-editorial-read.md` failure-shape table).

## Context

- **Why low**: each defect is a small reader-flow friction; no operator hits a hard failure. But the world-class doc bar holds that link text should match what the reader gets — magic-links are friction the reader pays per click.
- **Why log at all**: per `pillars/documentation/quality-framework.md` A4 audit cue, the magic-link sub-shape is canonical case-law (DOC-110 Site A, DOC-129's 3 sites). DOC-129 swept the tree but its regex missed case-variants and multi-word phrases — these 6 sites are tree-wide residue that the extended regex catches.
- **Recurring pattern**: the magic-link sweep should be **extended** to be case-insensitive AND to cover multi-word phrases (`[Read more]`, `[See here]`, `[Click here]`, `[More info]`, `[Find out more]`, `[Learn more]`). Worth folding into `pillars/documentation/quality-framework.md` A4 audit cue as the canonical sweep regex.
