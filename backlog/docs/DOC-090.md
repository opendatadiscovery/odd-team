## Description

DOC-087 review (2026-05-03) found the hint admonition at `docs/Features.md:38` carries a stale section list:

> {% hint style="info" %}
> **Looking for the HTTP API?** Every Platform endpoint is documented at [API Reference](developer-guides/api-reference.md) — grouped by feature area (Data Collaboration, Directory, Integrations, Query Examples, Reference Data, Relationships) with operation IDs, paths, and back-links to each feature page.
> {% endhint %}

The parenthetical lists the **6 canonical-home sections that existed at the end of DOC-083** (Data Collaboration, Directory, Integrations, Query Examples, Reference Data, Relationships). DOC-087 Phase B added three more H2 sections — **Alerts, Glossary, Lineage** — bringing the canonical home to 9 sections. The Features.md hint admonition was not updated in the same batch and now reads as a stale parallel reference: a Features.md reader sees the 6-name list and may assume those are the only sections.

This is a small Gate 1 (no duplicates, bi-directional sweep) miss for DOC-087 Phase B — when adding Alerts/Glossary/Lineage to the canonical home, the implementer should have grepped `docs/` for the prior 6-name list and updated every reference. The hint admonition was the only such reference (verified during this review by a quick grep for the section names).

## Acceptance Criteria

- [ ] Update `docs/Features.md:38` hint admonition. Two viable approaches:
  - **(a) extend the list**: rewrite the parenthetical to list all 9 sections — `(Alerts, Data Collaboration, Directory, Glossary, Integrations, Lineage, Query Examples, Reference Data, Relationships)`. Future-fragile — every canonical-home accretion (DOC-088 (b)-promotion, DOC-084 odd-tracing-gateway) needs to update this parenthetical too.
  - **(b) remove the list (recommended)**: drop the parenthetical entirely so the hint reads "Every Platform endpoint is documented at [API Reference](developer-guides/api-reference.md) with operation IDs, paths, and back-links to each feature page." The link alone is sufficient — readers who follow the link see the canonical home's in-page TOC. This decouples Features.md from the canonical home's section count.
- [ ] Sweep `docs/` for any other place that lists the canonical-home's section names (grep for the 6-name list as a phrase, plus any other phrasing). Update accordingly.
- [ ] Live-site verification (Gate 8 / `/review`) on `docs.opendatadiscovery.org/features` — verify the hint admonition renders the new wording.
- [ ] Sources footer cites: `Backlog: DOC-087 (done — added Alerts/Glossary/Lineage to canonical home)`, `Backlog: DOC-083 (done — original 6-section authoring)`.

## Context

- **Discovered during DOC-087 review (2026-05-03).** DOC-087 Phase B added the 3 new H2 sections to the canonical home; the hint admonition on Features.md:38 (DOC-083 era) was not updated in the same batch.
- **Why low priority**: the link still works; readers who click through see the full 9-section list. The harm is small (a Features.md-only reader may think the canonical home covers fewer feature areas than it does).
- **Why this slipped past Phase A**: Phase A's broader regex was scoped to *embedded API-reference content* (heading shapes, table columns matching `(GET|POST|PUT|DELETE|PATCH)\s+/api/`, bullet-list endpoints). The hint admonition is a cross-page reference describing the canonical home, not embedded API reference content; it didn't match the regex. The miss is a Gate 1 bi-directional sweep miss, not a Gate 10 miss.
- **Recurring pattern**: every canonical-home accretion needs a sweep for "places that describe the canonical home" — not just "places that embed canonical-home content". Worth folding into the next batch of `pillars/documentation/gates.md` Gate 1 case-law.
