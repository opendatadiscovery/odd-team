## Description

DOC-007 (review `2026-04-24`) surfaced that the authored metrics configuration section at `odd-platform.md:216-230` describes `metrics.storage` as a **read-side** switch only — "where the platform **reads metrics back**" for `INTERNAL_POSTGRES` vs "metrics are **queried from** an external Prometheus instance" for `PROMETHEUS`. The code tells a different story: `metrics.storage` is a **storage-tier** switch that routes both reads AND writes.

When `metrics.storage=PROMETHEUS`:

- `InternalMetricReader` / `InternalIngestionMetricsServiceImpl` are NOT loaded (conditional `@ConditionalOnProperty(name="metrics.storage", havingValue="INTERNAL_POSTGRES", matchIfMissing=true)`).
- `ExternalMetricReader` is loaded — queries Prometheus via `/api/v1/query` on `metrics.prometheus-host`.
- `ExternalIngestionMetricsServiceImpl` is **also** loaded — does Prometheus **remote-write** to `/api/v1/write` on the **same** `metrics.prometheus-host`, using Snappy encoding + Protobuf-serialized write requests.

Operator consequence: a Prometheus instance that does not accept remote-write (Thanos querier, Prometheus without `--web.enable-remote-write-receiver`, or Mimir in query-only mode) will **silently lose** all ingested metric data. Reads succeed; writes are dropped or 404; operators don't see an error in the normal ingestion API because the write is asynchronous downstream of the collector's HTTP 204 acknowledgement. This is the DOC-008-class silent-data-loss failure mode the pipeline-hardening pass exists to prevent.
