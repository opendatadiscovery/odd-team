## Description

A full Azure AD OAuth2 provider is implemented in `odd-platform-api/src/main/resources/application.yml` (lines 128-156) but is completely absent from the OAuth2/OIDC documentation. The docs cover Cognito, GitHub, Google, OKTA, and Keycloak — Azure AD is one of the most widely used enterprise identity providers and its omission is a critical coverage gap.

The implementation supports both multi-tenant and single-tenant deployments and exposes 15+ config keys including `azure-tenant-id`, `authorization-uri`, `token-uri`, `user-info-uri`, `admin-attribute`, `admin-principals`, and `admin-groups`.

## Re-Audit (2026-04-23)

Flipped from `done` to `review-ready` after consumer-read re-audit under the new eight-gate Quality Bar (Phase C).

**In-scope amendment required and shipped on `feature/phase-c-reaudit-amendments` (documentation), commit `b11279a`.** The Gate 4 consumer-read on the Azure handler chain revealed that the original DOC-001 description of `admin-groups` was wrong by default — the page stated the values are matched against the `groups` claim, but `AzureUserHandler.getDefaultGroupsClaim()` returns `"roles"`. An operator following the doc verbatim (enabling the `groups` claim in Azure Token Configuration and listing security group names in `admin-groups`) would see no ADMIN grant, because ODD reads the `roles` claim by default and never looks at `groups` unless `groups-claim: groups` is explicitly set on the provider.

Amendment highlights:

- Prerequisites step 6 now splits into two paths: **Azure AD App Roles** (the default, recommended) vs. **Azure AD security groups** (requires `groups-claim: groups` override + the Azure Token Configuration step).
- `admin-groups` bullet in Notes leads with the App-Roles default and adds a sub-bullet for the security-groups path, including the group-ID-vs-display-name distinction Azure emits.
- Troubleshooting bullet for "admin-groups has no effect" splits into App-Roles and security-groups decode steps, matching the two configuration paths.

**Consumer-read records** (verified in the current `odd-platform` codebase):

- `AzureUserHandler.java:35-37` — `getDefaultGroupsClaim()` returns `"roles"`. (Cognito returns `"cognito:groups"`; CustomOIDC returns `null`.)
- `AbstractOIDCUserHandler.java:44-55` — `groupsClaim = provider.getGroupsClaim() || getDefaultGroupsClaim()`; falls back to `getAttribute(groupsClaim)` with `admin-groups` membership check.
- `ODDOAuth2Properties.java:32-52` — POJO fields for every Azure config key documented (clientId, clientSecret, clientName, redirectUri, scope, issuerUri, authorizationUri, tokenUri, userInfoUri, jwkSetUri, userNameAttribute, adminAttribute, **groupsClaim**, adminGroups, adminPrincipals, pkce). `azureTenantId` is **not** a POJO field but resolves via Spring property-placeholder substitution in `application.yml:140-147` — relaxed binding confirmed.
- `application.yml:128-156` — Azure config template, commented-out; exact property names match the POJO.
- `GrantedAuthorityExtractor.java:12-17` — `isAdmin=true` grants `UserProviderRole.ADMIN`; `false` grants `UserProviderRole.USER`.
- `AzureCondition.java` + `AzureLogoutSuccessHandler.java` — confirm the Azure handler is gated on `auth.type=OAUTH2` with an `azure` client and that logout redirects to `{issuer}/oauth2/v2.0/logout` (not documented, but not currently in DOC-001's scope — tracked as a follow-up candidate for the troubleshooting section).
