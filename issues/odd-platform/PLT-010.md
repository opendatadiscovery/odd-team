## What

When `datacollaboration.enabled=false` (the default), the **Discussions** tab on every data-entity detail page is **still rendered and clickable** — only the API routes return `404 Not Found`. The platform UI does not gate tab visibility (or the route handler) on the data-collaboration feature flag, only on the entity's status (`isStatusDeleted`). An operator who has not configured Data Collaboration sees a tab that loads a component that fails to fetch its data; the user-visible result is a broken or empty Discussions surface on every catalog page.

## Where

Tab definition — `odd-platform-ui/src/components/DataEntityDetails/DataEntityDetailsTabs/DataEntityDetailsTabs.tsx:95-99`:

```tsx
{
  name: t('Discussions'),
  link: dataEntityDetailsPath(dataEntityId, 'discussions'),
  hidden: isStatusDeleted,        // <-- only hidden for deleted entities
},
```

Route definition — `odd-platform-ui/src/components/DataEntityDetails/DataEntityDetailsRoutes/DataEntityDetailsRoutes.tsx:106-117`:

```tsx
<Route
  path='discussions/*'
  element={
    <RestrictedRoute
      isAllowedTo={!isStatusDeleted}     // <-- only restricted for deleted entities
      redirectTo='../overview'
      component={DataCollaboration}
    />
  }
>
  <Route path=':messageId' />
</Route>
```

Backend (correct, for contrast) — every controller in `datacollaboration/controller/` is annotated `@ConditionalOnDataCollaboration`, so `/api/dataentities/{id}/messages`, `/api/dataentities/{id}/channels`, `/api/datacollaboration/...` all return `404 Not Found` when the feature is disabled. The backend gating is correct; the UI gating is missing.

## Why it matters

- **First-launch experience is broken by default.** A fresh ODD Platform deployment ships with `datacollaboration.enabled=false`. Every operator opening any catalog entity for the first time clicks through the entity tabs to learn the surface — the Discussions tab loads, the network call returns 404, and depending on the loading-state handling in `DataCollaboration.tsx` the user sees either an infinite spinner, an empty state, or an error banner. None of those communicates "this is a feature you can turn on" — they all communicate "this is broken".
- **Operators planning to disable the feature have no clean rollback.** If a team enables Data Collaboration, gathers some discussions, then later sets `datacollaboration.enabled=false` (e.g., to migrate Slack workspaces), the Discussions tab on every entity still appears in the UI but cannot fetch the (still-stored) message history. There is no "feature disabled" state.
- **Inconsistent with how other conditional features behave on the platform.** The pattern elsewhere (e.g., the `Test reports` tab is hidden if the entity is not a dataset or has no test runs; the `Linked entities` tab is hidden if `!dataEntityDetails?.hasChildren`) is to hide a tab when its content would be empty or unavailable. Discussions does not follow this pattern.
- **Severity is low** because the API itself is correctly gated (no security or data-integrity issue) and the workaround for operators is "ignore the broken tab". But the first-launch UX is real.

## Suggested fix

Two options — both small:

**Option A — surface the feature flag to the UI and gate the tab.** Add a global "data collaboration enabled" boolean to the platform's UI bootstrap response (similar to how other feature flags surface today, e.g., the auth-related flags exposed via `/api/info` or equivalent), then gate both the tab definition and the route on that flag:

```tsx
// DataEntityDetailsTabs.tsx
{
  name: t('Discussions'),
  link: dataEntityDetailsPath(dataEntityId, 'discussions'),
  hidden: isStatusDeleted || !isDataCollaborationEnabled,
},
```

This is the same shape as `hidden: !isDataset` and other content-aware conditionals already in the file.

**Option B — render a "feature disabled" placeholder inside the tab.** Keep the tab visible (it advertises the feature's existence to operators evaluating the platform), but render a lightweight "Data Collaboration is not enabled. Ask your operator to set `datacollaboration.enabled=true`. See [docs link]." panel in `DataCollaboration.tsx` when the API returns 404. The placeholder doubles as feature discovery for evaluators.

The team should pick based on UX intent — Option A is invisible-when-off (consistent with the rest of the tab pattern); Option B is discovery-friendly but accepts the broken-tab anti-pattern in exchange.

## How discovered

Surfaced during implementation of DOC-091 (Author Data Collaboration on Features.md) on 2026-05-03. The DOC-091 implementer was verifying the user-facing tab name (`'Discussions'`) for the Features.md showcase content and noticed the route's only guard was `isStatusDeleted`, with no `datacollaboration.enabled` check anywhere in the UI tree.

The doc-side handling: DOC-091's Features.md section documents the API-level behavior accurately ("when `datacollaboration.enabled=false`, the per-entity messaging routes return `404 Not Found`") and adds a brief operator-facing note that the Discussions tab is currently shown regardless of the flag, with a pointer back to this issue draft for the upstream fix. No doc-side claim presupposes the tab is hidden when disabled — so a future platform fix that hides the tab won't invalidate the doc.
