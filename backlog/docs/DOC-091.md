## Description

Discovered during a user spot-check at the close of `/review batch:feature/docs-content-type-homing-followups` (2026-05-03): "why there is no mentioning about data collaboration at https://docs.opendatadiscovery.org/features at all?"

The platform's **Data Collaboration** feature (in-app per-entity messaging with Slack as the message backend) is documented across three surfaces but absent from the primary user-facing showcase:

- **Configuration**: `docs/configuration-and-deployment/odd-platform.md:622` `## Enable Data Collaboration` H2 — 7 config keys (`datacollaboration.enabled`, `datacollaboration.receive-event-advisory-lock-id`, `datacollaboration.sender-message-advisory-lock-id`, `datacollaboration.message-partition-period`, `datacollaboration.sending-messages-retry-count`, `datacollaboration.slack-oauth-token`, plus the `@Value("${datacollaboration.message-partition-period:30}")` consumer in `MessageTablePartitionManager`), advisory-lock setup, message-partition rationale, retry behavior, Slack app-manifest setup.
- **API Reference**: `docs/developer-guides/api-reference.md:81-105` `## Data Collaboration` H2 (DOC-083 era) — 7 endpoints across 3 groups (outbound to Slack: `getSlackChannels` + `postMessageInSlack`; per-entity threads & history: `getDataEntityMessages` + `getMessages` + `getChannels` + `redirect`; inbound webhook: `POST /api/slack/events` handling `url_verification` + `event_callback`).
- **Navigation**: `navigation/domains/collaboration.md` carries a dedicated "Data Collaboration (Slack messaging)" subsection naming `DataCollaborationProperties.java`, `DataCollaborationFeatureCondition.java`, `SlackAPIClient.java`, `SlackAPIClientImpl.java`, `SlackMessageProviderClient.java`, `MessageServiceImpl.java`.
- **Features.md**: **nothing.** No H2, no in-page TOC entry at lines 3-36, no mention. The two existing "Slack" mentions (lines 146 + 152) are about the **alerting** Slack webhook (different feature, configured via `notifications.receivers.slack.*`, not `datacollaboration.*`).

A first-time reader scrolling Features.md or its in-page TOC has no path to discover Data Collaboration exists. They'd only encounter it by reading the configuration page (operator surface) or the API reference (developer surface). This breaks **Cornerstone 1** ("Discoverability without context") and **Cornerstone 4** ("Three audiences, AI-maintained consistency") — operators see it, developers see it, **users don't**.

## Why this slipped past prior batches

DOC-083 (2026-05-03) added the API Reference `## Data Collaboration` H2 alongside `odd-platform.md`'s pre-existing `## Enable Data Collaboration` config H2 (DOC-034 territory). Neither author surfaced "is there a Features.md showcase for this?" — both were content-type-homing refactors, not feature-discovery audits. The collaboration navigation domain even lists `Features.md#manual-object-tagging` and `Features.md#activity-feed-for-monitoring-changes` under "Documentation" — those are *tagging* + *activity feed* (two different features); the navigation file has the same blind spot.

This is the failure mode **Cornerstone 1** is meant to catch: a feature can be 100% documented on its operator + developer surfaces and still be undiscoverable from the user-facing landing.
