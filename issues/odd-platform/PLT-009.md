## What

The `/api/messages/{message_id}/url` redirect endpoint declares one HTTP status code in the OpenAPI contract and returns a different one at runtime. The spec at `odd-platform-specification/openapi.yaml:1788` declares `'301': Moved Permanently`; the runtime implementation in `DataCollaborationController.java:42-48` returns `HttpStatus.FOUND` (HTTP **302**). Both responses are functionally redirects (a browser following the `Location` header lands on the provider's deep link either way), but spec-level consumers — generated clients, API gateways, HTTP caching proxies, contract-test harnesses — will disagree with the runtime behavior. The user-visible difference is that 301 is treated as cacheable-by-default by intermediaries while 302 is not, so a deployment behind a caching proxy (CloudFront, Cloudflare, internal Squid) may behave differently than the spec implies.

## Where

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/datacollaboration/controller/DataCollaborationController.java:42-48`:

```java
@Override
public Mono<ResponseEntity<Void>> redirect(final UUID messageId, final ServerWebExchange exchange) {
    return dataCollaborationService.resolveMessageUrl(messageId)
        .map(providerUrl -> ResponseEntity
            .status(HttpStatus.FOUND)                  // <-- HTTP 302
            .headers(headers -> headers.setLocation(URI.create(providerUrl)))
            .build()
        );
}
```

`odd-platform-specification/openapi.yaml:1776-1791`:

```yaml
/api/messages/{message_id}/url:
  get:
    summary: Redirects user to correspondent message in the provider
    operationId: redirect
    parameters:
      - name: message_id
        in: path
        schema:
          type: string
          format: uuid
        required: true
    responses:
      '301':                                            # <-- spec says 301
        description: Moved Permanently
    tags:

```

## Why it matters

* **Contract drift.** Generated clients (TypeScript, Python, Java) embed status-code expectations from the spec. A `302` from runtime where `301` is contracted means the generated client either treats the response as an error (strict mode) or silently follows the redirect anyway (typical default) — neither is contract-conformant.
* **Caching semantics differ.** RFC 7231 specifies 301 as cacheable-by-default; 302 is not (RFC 7231 §6.4.3). A deployment behind a caching CDN sees different behavior than the spec implies; an operator reading the spec to plan caching policy will misconfigure.
* **Reverse-engineering tools see the wrong shape.** API contract testers (Schemathesis, Dredd) exercising the spec against the live platform will report a mismatch.
* **Severity is low** because the operator-facing redirect still works (browsers follow both); the bug is in the seam between spec and runtime that causes confusion for tool authors and contract testers.

## Suggested fix

Two options — both are 1-line changes:

**Option A — match spec to runtime (recommended).** The redirect from a stable platform UUID to a provider message URL is properly a temporary redirect: the provider may rotate or reorganize the URL, the platform UUID does not. `302 Found` (or, per RFC 7231, `307 Temporary Redirect` for stricter method preservation) is semantically correct.

```yaml
# openapi.yaml:1788
responses:
  '302':
    description: Found
```

**Option B — match runtime to spec.** If the redirect is intended to be permanent (provider message URLs are immutable in your model):

```java
// DataCollaborationController.java:45
.status(HttpStatus.MOVED_PERMANENTLY)   // 301
```

We weakly prefer Option A because (a) it's a smaller-blast-radius change (spec-only, no runtime behavior change), (b) the redirect is from an opaque platform UUID to an external provider URL — the platform cannot guarantee the provider URL is permanent, so 302 (or 307) is more honest about the semantic, (c) operators currently relying on the existing 302 (in caching proxies, in test harnesses) keep working unchanged.

## How discovered

Surfaced during `/review` of DOC-034 (Document data collaboration API endpoints) on 2026-04-30. The DOC-034 implementer cited both the controller (`Builder:` Sources line) and the spec (`Spec:` Sources line) but did not cross-reference the controller's actual `HttpStatus.FOUND` against the spec's declared `'301'`. The doc shipped with the spec's `301 Redirect` claim. Review caught it via direct Read of the controller line range and Read of the spec.

The doc-side fix ships as part of unblocking DOC-034 (which the review flipped `review-ready` → `blocked`); this issue draft is the upstream code/spec adjustment that lets a future doc revision drop any "spec/runtime drift" hedging.
