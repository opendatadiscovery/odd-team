## Findings

### F-047: Azure AD OAuth2 provider completely undocumented
- **Location**: Missing from `documentation/docs/configuration-and-deployment/enable-security/authentication/oauth2-oidc.md`
- **Issue**: `application.yml` lines 128-156 define a full Azure AD OAuth2 provider with 15+ config keys including: `azure-tenant-id`, `authorization-uri`, `token-uri`, `user-info-uri`, support for both multi-tenant and single-tenant configurations, admin groups, and admin principals. This is a complete provider implementation — not documented at all. Azure AD is one of the most widely used enterprise identity providers.
- **Evidence**: `application.yml` lines 128-156: `auth.oauth2.client.azure.*` with provider, client-id, azure-tenant-id, client-secret, scope (openid, offline_access, graph), multi-tenant issuer-uri, authorization-uri, token-uri, user-info-uri, user-name-attribute, admin-attribute, admin-principals, admin-groups.
- **Severity**: critical
- **Suggested fix**: Add Azure AD section to oauth2-oidc.md alongside Cognito, GitHub, Google, OKTA, Keycloak. Include both multi-tenant and single-tenant configurations.

### F-048: S2S (Server-to-Server) API key authentication undocumented
- **Location**: Missing from all auth documentation
- **Issue**: `application.yml` lines 39-42 define `auth.s2s.enabled` (default: false) and `auth.s2s.token`. This enables API key authentication via `X-API-Key` header for server-to-server integrations (e.g., CI/CD pipelines, automated scripts). This is entirely undocumented — users who need programmatic API access don't know this exists.
- **Evidence**: `application.yml` lines 39-42: `auth.s2s.enabled: false` and `auth.s2s.token` (commented). Comment says "Server-To-Server token. Header 'X-API-Key: token'".
- **Severity**: critical
- **Suggested fix**: Document S2S authentication in the security section. Explain: enable with `auth.s2s.enabled=true`, set token, use via `X-API-Key` header. Note that this works alongside other auth types.

### F-049: Housekeeping configuration completely undocumented
- **Location**: Missing from all documentation pages
- **Issue**: `application.yml` lines 165-170 define a `housekeeping` section with 4 config keys that control automatic cleanup of stale data: `housekeeping.enabled` (default: true), `housekeeping.ttl.resolved_alerts_days` (default: 30), `housekeeping.ttl.search_facets_days` (default: 30), `housekeeping.ttl.data_entity_delete_days` (default: 30). Users don't know that resolved alerts auto-delete after 30 days, or that they can tune these TTLs.
- **Evidence**: `application.yml` lines 165-170. `AlertHousekeepingJob.java` implements the cleanup. `DataEntityHousekeepingJob.java` handles entity deletion.
- **Severity**: high
- **Suggested fix**: Document housekeeping in the platform configuration page. Explain all TTL values and their defaults.

### F-050: Ingestion auth filter undocumented
- **Location**: Missing from auth documentation
- **Issue**: `application.yml` lines 46-48 define `auth.ingestion.filter.enabled` (default: false). When enabled, this applies authentication to the ingestion API (`/ingestion/*` endpoints), meaning collectors must authenticate. This is a critical security feature — without it, the ingestion API is open even when auth is enabled for the UI. Users deploying with security enabled may not realize ingestion endpoints are unprotected by default.
- **Evidence**: `application.yml` line 48: `auth.ingestion.filter.enabled: false`.
- **Severity**: high
- **Suggested fix**: Document in the security section. Explain that enabling auth does NOT protect ingestion endpoints unless `auth.ingestion.filter.enabled=true`. This is a security-critical note.

### F-051: Keycloak PKCE support undocumented
- **Location**: Missing from `oauth2-oidc.md` Keycloak section
- **Issue**: `application.yml` line 127 shows `pkce` as a config option under the Keycloak provider. PKCE (Proof Key for Code Exchange) is a security enhancement for OAuth2. The docs' Keycloak example doesn't mention this option. Users deploying Keycloak with PKCE requirements won't know the platform supports it.
- **Evidence**: `application.yml` line 127: `pkce` under `auth.oauth2.client.keycloak`.
- **Severity**: medium
- **Suggested fix**: Add `pkce` option to the Keycloak documentation example.

### F-052: `odd.data-entity-stale-period` and `odd.tenant-id` undocumented
- **Location**: Missing from all documentation pages
- **Issue**: `application.yml` lines 210-211 define: `odd.tenant-id` (no default — multi-tenancy identifier) and `odd.data-entity-stale-period: 7` (days — controls when entities are marked stale if not re-ingested). These affect core platform behavior but are not documented. The stale period is important for users who run collectors on schedules longer than 7 days.
- **Evidence**: `application.yml` lines 210-211.
- **Severity**: medium
- **Suggested fix**: Document both. `data-entity-stale-period` is particularly important — explain what happens when an entity is marked stale.

### F-053: `auth.login-form-credentials` config exposed in application.yml
- **Location**: Not documented (and shouldn't be used in production)
- **Issue**: `application.yml` line 37 defines `auth.login-form-credentials: admin:admin,root:root` with comment "For dev/demo purposes only." This is default credentials shipped in the config file. While there's no documentation issue per se, the fact that LOGIN_FORM auth docs don't mention this property (or warn against it) means users might not realize default credentials exist.
- **Evidence**: `application.yml` line 37. Comment: "For dev/demo purposes only -- username1:password1,username2:password2,etc".
- **Severity**: medium
- **Suggested fix**: Either remove from default config or document the LOGIN_FORM section with a warning that default credentials must be changed.

### F-054: `spring.session.timeout` and `spring.codec.max-in-memory-size` undocumented
- **Location**: Missing from session/general configuration documentation
- **Issue**: `application.yml` lines 2-3 set `spring.session.timeout: -1` (infinite session) and line 15 sets `spring.codec.max-in-memory-size: 20MB` (affects max request body size). Neither is documented. The infinite session timeout is a security consideration. The codec size affects file upload limits.
- **Evidence**: `application.yml` lines 2-3 and 15.
- **Severity**: medium
- **Suggested fix**: Document `spring.session.timeout` in the session provider section. Document `spring.codec.max-in-memory-size` with the attachment config.

### F-055: `odd.activity.partition-period` undocumented
- **Location**: Missing from all documentation pages
- **Issue**: `application.yml` lines 212-213 define `odd.activity.partition-period: 30` (days). This controls the PostgreSQL table partitioning period for the activity feed. Not documented — users with high-volume activity may need to tune this.
- **Evidence**: `application.yml` line 213.
- **Severity**: medium
- **Suggested fix**: Document as an advanced/operational config option.
