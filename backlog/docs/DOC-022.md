## Description

The platform has three alert statuses — `OPEN`, `RESOLVED` (manual), and `RESOLVED_AUTOMATICALLY`. Current docs mention only a "resolve" button and don't describe auto-resolution (e.g., an alert disappearing when a subsequent job run succeeds). Users are surprised when alerts resolve themselves.

## Acceptance Criteria

- [ ] Alerting section documents the three statuses with short descriptions
- [ ] Auto-resolution triggers are listed (the primary one: a previously failed job run followed by a successful run)
- [ ] The difference between `RESOLVED` and `RESOLVED_AUTOMATICALLY` is explained

**Caveats surfaced and shipped in scope:**
- **Auto-resolution applies only to `Failed job` and `Failed data quality test`.** `Backwards incompatible schema` and `Distribution anomaly` never auto-resolve. Documented in the body. Discovered via the constructor switch in `IngestionTaskRunAlertState.java:41-45` which throws `IllegalStateException` for the other two types — they never reach `reportSuccess()`.
- **Manual reopen has a guard.** Reopening to OPEN fails if another open alert of the same type exists on the same data entity. Shipped as an info admonition with the literal error message. Discovered via `AlertServiceImpl.java:124-132`.
- **`RESOLVED` vs `RESOLVED_AUTOMATICALLY` is preserved on the activity feed via the `systemEvent` flag** — both clear the alert from open counts but the audit trail distinguishes them. Documented in the lifecycle paragraph.
