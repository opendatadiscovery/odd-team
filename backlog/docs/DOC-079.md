## Description

DOC-040 shipped `docs/deployment.md` at the docs root, indexed in `SUMMARY.md` under the `## Configuration and Deployment` group. GitBook computes URL slugs from SUMMARY hierarchy rather than file paths, so the published URL is `https://docs.opendatadiscovery.org/configuration-and-deployment/deployment`, not `https://docs.opendatadiscovery.org/deployment` as the DOC-040 handoff anticipated. The page renders correctly, all back-links resolve, and inline cross-links work — this is purely a file-path / URL-slug discrepancy.

The discrepancy is maintainer-facing only: a future contributor grepping `docs/configuration-and-deployment/` for the Deployment Options page won't find it (it's at `docs/deployment.md` instead), and clicking a search result for "deployment" lands on `/configuration-and-deployment/deployment` even though the file lives at root. Operators don't see file paths, so this isn't a user-visible defect — but it costs maintainer time at the next touch.

## Acceptance Criteria

- [ ] Move `docs/deployment.md` → `docs/configuration-and-deployment/deployment.md` so the file path matches the published URL
- [ ] Update `docs/SUMMARY.md` line 36 from `[Deployment Options](deployment.md)` to `[Deployment Options](configuration-and-deployment/deployment.md)`
- [ ] Update inline relative links inside the moved page — `configuration-and-deployment/X.md` references collapse to `X.md` (sibling), `developer-guides/X.md` references become `../developer-guides/X.md`, etc.
- [ ] Update the three deep-dive back-links from `[Deployment Options](../deployment.md)` to `[Deployment Options](deployment.md)` (sibling, no `../`)
- [ ] Live-site verification: `https://docs.opendatadiscovery.org/configuration-and-deployment/deployment` continues to render the same content; the in-page link targets are unchanged at the user-facing URL level
- [ ] Both old and new URLs render the same final page (GitBook routes file moves to the same slug, so this should hold automatically — verify after merge)

## Context

- Surfaced during `/review DOC-040` (2026-04-30, this batch) — the WebFetch of `https://docs.opendatadiscovery.org/deployment` returned 404, while `https://docs.opendatadiscovery.org/configuration-and-deployment/deployment` rendered the full page correctly.
- Alternative approach considered and rejected: leave the file at root and restructure SUMMARY to host the entry above (rather than inside) the `## Configuration and Deployment` section. That approach preserves the "umbrella above the subtree" semantic but breaks the established IA convention that pages indexed under a section heading live in the matching subdirectory.
- Chosen direction (proposed): move the file to match the URL slug. The "umbrella" semantic is encoded in SUMMARY ordering (first item under the section heading) rather than in file-path placement.
