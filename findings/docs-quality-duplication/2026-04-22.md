### F-001: Third S2S/M2M duplicate hidden in `odd-platform.md`

- **Page A (canonical)**: `docs/configuration-and-deployment/enable-security/authentication/s2s.md`
- **Page B (duplicate)**: `docs/configuration-and-deployment/odd-platform.md`, section "Machine-to-Machine (M2M) Tokens Configuration", lines 544-567
- **Shared surface**: Same config keys (`auth.s2s.enabled`, `auth.s2s.token`), same env-var pair (`AUTH_S2S_ENABLED`, `AUTH_S2S_TOKEN`), describes the same feature with full YAML + env-var tabs.
- **Current state**: No cross-link in either direction. DOC-052 collapsed the `Features.md` M2M section to a teaser and cross-linked it with `s2s.md`, but left this **third** copy untouched. All three live on main today.
- **Live verification**: `https://docs.opendatadiscovery.org/configuration-and-deployment/odd-platform` renders the section with zero references to the dedicated S2S page.
- **Severity**: **High** — substantive duplicate with a full config block that readers may follow instead of the canonical page (which has the complete security considerations, header contract, and curl example). Drift risk: if token handling changes in `s2s.md`, nobody will update this section.
- **Recommended action**: Collapse `odd-platform.md` lines 544-567 to a one-paragraph teaser pointing at `s2s.md`. Delete the YAML/env-var blocks (the canonical page has them). Keep the "disabled by default" note. No new alias needed — S2S↔M2M already in Terms & Aliases.
- **Terms & Aliases row needed?** No (already present)

### F-002: `Adapters.md` is an orphaned page that duplicates main-concepts.md

- **Page A (canonical)**: `docs/main-concepts.md` — Adapter vs Collector vs Push-client; Push vs Pull strategies
- **Page B (duplicate)**: `docs/Adapters.md` (entire file — 32 lines)
- **Shared surface**: Adapters section, Push-client section, Push/pull strategies section — three of the four main-concepts.md sections.
- **Current state**: `Adapters.md` is **not in SUMMARY.md** (orphan — reachable only by direct URL). Opens with `> :exclamation: Adapters will be deprecated soon!` but the page itself is still live on the site. No link from or to `main-concepts.md`.
- **Severity**: **High** — the page contradicts current guidance ("install the Collector") but is not deleted, and its "Push-client is a provider which sends information directly..." sentence is used verbatim in `Architecture.md:7` (see F-003). Classic undetected-drift surface.
- **Recommended action**: Delete `Adapters.md`. Its content is superseded by `main-concepts.md`. If anything is worth preserving, the "Adapters will be deprecated" note can move as a one-sentence deprecation warning into `main-concepts.md` under Adapter vs Collector vs Push-client.
- **Terms & Aliases row needed?** No (Push-client↔Push adapter already present)
- **Cross-ref**: This also qualifies as a `docs/quality/rendering` orphan-page finding — noted in that findings file under F-R01.

### F-003: `Architecture.md` lacks cross-reference to `main-concepts.md`

### F-004: Permissions page uses "business glossary" alias with no cross-link

- **Page**: `docs/configuration-and-deployment/enable-security/authorization/permissions.md`
- **Hits**: 5 occurrences of "business glossary" across permission descriptions (lines 6, 32, 33, 46, 97)
- **Shared surface**: The permissions regulate term operations in the in-app Business Glossary feature.
- **Current state**: Lowercase `business glossary` is an alias of the Business Glossary product feature documented in `Features.md`. The permissions page never links to Features.md Business Glossary section or to `main-concepts.md` Terms & Aliases where the alias is registered.
- **Severity**: **Medium** — the page describes *capabilities* related to a feature without letting the reader jump to the feature's primary documentation. Users landing here by search lose the next hop.
- **Recommended action**: Add a one-line pointer near the top of `permissions.md` or at the first mention: *"These permissions control access to the in-app [Business Glossary](../../Features.md#dictionary-terms) feature."* (or whatever the Features.md anchor resolves to — verify the anchor).
- **Terms & Aliases row needed?** No (already present: "Business Glossary ↔ Glossary, Terms")
