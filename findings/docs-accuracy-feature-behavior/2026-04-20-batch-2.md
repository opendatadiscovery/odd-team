### F-010: "Labels" terminology is stale — labels were merged into tags in v0.17.0
- **Location**: `documentation/docs/Features.md` lines 94-97 (Manual Object Tagging)
- **Issue**: Docs say "You may apply **tags** to metadata entities or use **labels** to mark elements of these entities." Labels and tags were originally separate concepts — tags for data entities, labels for dataset fields (columns). In Sep 2023, the team decided to eliminate labels and universally apply tags (issue [#1425](https://github.com/opendatadiscovery/odd-platform/issues/1425)). PR [#1428](https://github.com/opendatadiscovery/odd-platform/pull/1428) refactored the backend and PR [#1440](https://github.com/opendatadiscovery/odd-platform/pull/1440) completed the frontend replacement. Release v0.17.0 (Oct 16, 2023) shipped the breaking change: the labels DB table was dropped and all labels were auto-converted to tags. The docs were never updated to reflect this merge.
- **Evidence**: No Label controller, service, or repository exists post-v0.17.0. Activity events confirm: `DATASET_FIELD_TAGS_UPDATED` — fields now use tags. v0.17.0 release notes: "Not Backward compatible. The labels database table will be removed, with all existing labels automatically converted to tags."
- **Severity**: high
- **Suggested fix**: Remove "labels" terminology. Clarify that tags can be applied to both entities and their fields (columns). This is a documentation update that should have shipped with v0.17.0.

### F-011: Search facets significantly understated
- **Location**: `documentation/docs/Features.md` lines 27-28 (Advanced search)
- **Issue**: Docs say "Filtering by datasources, owners and tags" — listing only 3 filters. The actual search UI has 7 facet filters: Datasource, Type, Namespace, Owner, Tag, Groups, Statuses. Docs omit 4 major filters.
- **Evidence**: `odd-platform-ui/src/components/Search/Filters/Filters.tsx` lines 47-65 renders all 7 filters. `FacetType.java` enum has: ENTITY_CLASSES, TYPES, NAMESPACES, DATA_SOURCES, OWNERS, TAGS, GROUPS, STATUSES, DATA_ENTITY.
- **Severity**: high
- **Suggested fix**: Update search description to list all available filters. Namespace and Groups filters are particularly important for multi-team setups.

### F-012: Data quality "Pandas" compatibility claim is misleading
- **Location**: `documentation/docs/Features.md` lines 57-58 (Data Quality Test Results Import)
- **Issue**: Docs say "The Platform is compatible with **Pandas** and **Great expectations**." This is misleading. The `odd-collector-profiler` repo (https://github.com/opendatadiscovery/odd-collector-profiler) exists and is active (last commit Apr 2024, v0.2.2), but it uses **Capital One's DataProfiler** — not pandas-profiling or ydata-profiling. Zero references to pandas-profiling in its code or dependencies. The profiler sends dataset statistics via `/ingestion/entities/datasets/stats`. Great Expectations integration exists as `odd-great-expectations`. The "Pandas" claim doesn't correspond to any current integration.
- **Evidence**: `odd-collector-profiler/pyproject.toml` depends on `dataprofiler==0.10.9` (Capital One DataProfiler), not pandas-profiling. No pandas-profiling or ydata-profiling dependency anywhere. README states: "uses DataProfiler for getting statistics."
- **Severity**: critical
- **Suggested fix**: Replace "Pandas" with "DataProfiler (Capital One)" or reference `odd-collector-profiler` by its actual profiling library. List actual supported integrations: Great Expectations, DataProfiler, dbt tests, Soda, or note that any DQ framework can push results via the Ingress API.

### F-013: SLA endpoint URL uses inconsistent parameter name
- **Location**: `documentation/docs/Features.md` lines 144-145 (Dataset Quality Statuses SLA)
- **Issue**: Docs say URL is `https://{platform_url}/api/datasets/{dataset_id}/sla`. The actual OpenAPI spec uses `{data_entity_id}` not `{dataset_id}`. While functionally equivalent, a user copy-pasting the URL and trying to construct it from documentation will be confused about what ID to use.
- **Evidence**: `odd-platform-specification/openapi.yaml` line 1880: `/api/datasets/{data_entity_id}/sla` with parameter `$ref: DataEntityIdParam`.
- **Severity**: medium
- **Suggested fix**: Change `{dataset_id}` to `{data_entity_id}` to match the actual API contract.

### F-014: Activity event types documentation is wrong
- **Location**: `documentation/docs/Features.md` lines 129-134 (Activity Feed for Monitoring Changes)
- **Issue**: Docs list 4 event types: CREATED, DELETED, UPDATED, ADDED (ASSIGNED). Code has 26 event types, and "DELETED" is NOT one of them. The docs' categories don't map to reality — there is no generic DELETED event. Entity deletion is tracked via `DATA_ENTITY_STATUS_UPDATED`. The simplified model actively misleads users about what events they can filter and monitor.
- **Evidence**: `ActivityEventTypeDto.java` — 26 enum values, none named "DELETED". UI Activity Filters (`Filters.tsx`) exposes ~19 of these as filterable event types. The code specifically excludes 7 internal types from the filter dropdown.
- **Severity**: critical
- **Suggested fix**: Replace the 4 generic event types with the actual visible types or provide a meaningful categorization that maps to the real events.

### F-016: ML Experiment feature overpromised
- **Location**: `documentation/docs/Features.md` lines 83-88 (ML Experiment Logging)
- **Issue**: Docs say "The Platform helps track and compare your experiments" and "Log the most successful experiments." In code, ML_EXPERIMENT is just a DataEntityType(6) classified under DATA_ENTITY_GROUP. There is no dedicated experiment comparison UI, no experiment logging system, no "success" tracking. Experiments are rendered as generic entity groups containing ML-related entities.
- **Evidence**: `DataEntityTypeDto.java` line 20: `ML_EXPERIMENT(6)`. `DataEntityClassDto.java` line 50: `DATA_ENTITY_GROUP(8, Set.of(ML_EXPERIMENT, DAG, ...))`. No ExperimentController, ExperimentService, or comparison component exists.
- **Severity**: high
- **Suggested fix**: Rewrite to describe what actually exists: ML experiments are data entity groups that organize related ML entities. Remove "compare" and "log the most successful" claims.

### F-019: Term-to-term API linking and Query Examples undocumented
- **Location**: `documentation/docs/Features.md` (Dictionary terms section)
- **Issue**: Docs describe term-to-entity linking via description text. Code additionally supports: (1) direct term-to-term API linking via `addLinkedTermToTerm`/`deleteLinkedTermFromTerm`, and (2) Query Example associations via `createQueryExampleToTermRelationship`/`deleteQueryExampleToTermRelationship`. Neither of these features is documented.
- **Evidence**: `TermController.java` lines 237-250 (term-to-term linking), lines 219-234 (query examples).
- **Severity**: medium
- **Suggested fix**: Document the direct term-to-term linking API and the Query Example feature.

### F-020: Data Collaboration docs missing config property details
- **Location**: `documentation/docs/configuration-and-deployment/odd-platform.md` (Enable Data Collaboration)
- **Issue**: Docs say data collaboration "allows users to initiate discussion about specific data entity in messengers directly from the ODD Platform. Thread replies are tracked." The actual collaboration API has dedicated endpoints (`/api/datacollaboration/providers/slack/channels`, `/api/datacollaboration/providers/slack/messages`, `/api/dataentities/{id}/messages/*`) which aren't listed. Also, the Slack Events API webhook URL pattern (`/api/slack/events`) in the manifest template differs from actual endpoint in code.
- **Evidence**: `openapi.yaml` lines 3704-3728: Slack channel and message endpoints. `DataCollaborationProperties.java` doesn't include `message-partition-period` (bound differently), and the property `slack-oauth-token` binding mechanism isn't in the properties class.
- **Severity**: medium
- **Suggested fix**: Add API endpoints reference and verify the manifest template URL is current.
