## What

`AlertHousekeepingJob` is documented (and intuitively expected) to delete alerts in `RESOLVED` or `RESOLVED_AUTOMATICALLY` status only after `housekeeping.ttl.resolved_alerts_days` (default 30) days have passed since `status_updated_at`. That is what the doc says; it is not what the code does.

Due to SQL operator precedence in the `.where(...).or(...).and(...)` chain, the actual query selects:
- **Every** `RESOLVED` alert, regardless of `status_updated_at` (no retention window applied at all).
- Only `RESOLVED_AUTOMATICALLY` alerts whose `status_updated_at` is older than the cutoff.

In practice this means **a manually resolved alert is hard-deleted on the very next housekeeping run after it's resolved** — typically within hours, not 30 days. The audit trail of operator-resolved alerts on production data entities is silently destroyed.

## Where

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/housekeeping/job/AlertHousekeepingJob.java:28-34`:

```java
final List<Long> alertsToDelete = dslContext.select(ALERT.ID)
    .from(ALERT)
    .where(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED.getCode()))
    .or(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED_AUTOMATICALLY.getCode()))
    .and(ALERT.STATUS_UPDATED_AT.lessOrEqual(
        DateTimeUtil.generateNow().minusDays(housekeepingTTLProperties.getResolvedAlertsDays())))
    .fetch(ALERT.ID);
```

jOOQ renders this directly to SQL as:

```sql
WHERE status = 2 OR status = 3 AND status_updated_at <= ?
```

By SQL operator precedence (`AND` binds tighter than `OR`), Postgres evaluates this as:

```sql
WHERE (status = 2) OR (status = 3 AND status_updated_at <= ?)
```

i.e. all `RESOLVED` alerts match unconditionally, only `RESOLVED_AUTOMATICALLY` alerts respect the cutoff.

The intended query is:

```sql
WHERE (status = 2 OR status = 3) AND status_updated_at <= ?
```

The fluent-API equivalent is to combine the OR inside the WHERE, e.g.

```java
.where(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED.getCode())
    .or(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED_AUTOMATICALLY.getCode())))
.and(ALERT.STATUS_UPDATED_AT.lessOrEqual(...))
```

This is the pattern already used elsewhere in the codebase, e.g.
`odd-platform-api/.../repository/reactive/ReactiveLineageRepositoryImpl.java:117`
(`.or(LINEAGE.CHILD_ODDRN.in(...).and(LINEAGE.PARENT_ODDRN.in(...)))`) — note the AND is grouped *inside* the OR by combining on the condition object, not by chaining `.and(...)` on the `SelectConditionStep`.

## Why it matters

- **Silent audit-trail loss.** Operators who manually resolve an alert have done deliberate work (acknowledged a real issue, fixed the root cause, marked it closed). The alert record — including its chunk descriptions, who resolved it, the timestamps — is the only on-platform evidence that the work happened. Hard-deleting it within hours of resolution destroys the operator audit trail. The 30-day retention is precisely to give postmortems and incident reviews time to find these records.
- **Inconsistent behavior between manual and auto resolutions.** Two alerts of the same type, on the same entity, resolved one minute apart — one manually, one auto-resolved by the next ingest — will have wildly different lifetimes in the database. The auto-resolved one survives 30 days; the manually-resolved one is gone tomorrow. Users notice this when they revisit an alert URL and get a 404 for one but not the other.
- **`housekeeping.ttl.resolved_alerts_days` is partially meaningless.** Operators who raise this value (e.g. for compliance: "we keep alert history for 90 days") gain longer retention only for auto-resolved alerts. Manual resolutions ignore the setting entirely.
- **Pre-existing since 2023.** The buggy query was introduced in commit `c8dc0544` (2023-08-03 "Housekeeping on empty partitions of Activity and Message tables") and has not been touched since. Older deployments have been silently losing manual-resolution audit data for years.
- **Compounds DOC-021 / PLT-003.** Documentation we are about to ship (DOC-021) describes the retention as applying to both states. Documentation we already shipped (DOC-009 → `odd-platform.md:679`) makes the same claim. PLT-003's body (lines 56) also assumes the retention applies to all resolved alerts. All three need correction once this is fixed.

## Suggested fix

Change `AlertHousekeepingJob.java:28-34` to group the OR inside the WHERE clause. Minimal diff:

```diff
 final List<Long> alertsToDelete = dslContext.select(ALERT.ID)
     .from(ALERT)
-    .where(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED.getCode()))
-    .or(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED_AUTOMATICALLY.getCode()))
+    .where(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED.getCode())
+        .or(ALERT.STATUS.eq(AlertStatusEnum.RESOLVED_AUTOMATICALLY.getCode())))
     .and(ALERT.STATUS_UPDATED_AT.lessOrEqual(
         DateTimeUtil.generateNow().minusDays(housekeepingTTLProperties.getResolvedAlertsDays())))
     .fetch(ALERT.ID);
```

Equivalent alternative using `DSL.or(...)`:

```java
.where(DSL.or(
    ALERT.STATUS.eq(AlertStatusEnum.RESOLVED.getCode()),
    ALERT.STATUS.eq(AlertStatusEnum.RESOLVED_AUTOMATICALLY.getCode())))
.and(ALERT.STATUS_UPDATED_AT.lessOrEqual(...))
```

Or `IN`:

```java
.where(ALERT.STATUS.in(
    AlertStatusEnum.RESOLVED.getCode(),
    AlertStatusEnum.RESOLVED_AUTOMATICALLY.getCode()))
.and(ALERT.STATUS_UPDATED_AT.lessOrEqual(...))
```

The `IN` form is the most readable and matches the surrounding codebase's preference for set membership when the condition is "status is one of N values".

- Integration test (Testcontainers Postgres):
  - Insert two alerts: one `RESOLVED` with `status_updated_at = now()`, one `RESOLVED_AUTOMATICALLY` with `status_updated_at = now()`.
  - Run `AlertHousekeepingJob.doHousekeeping(connection)` with `resolvedAlertsDays = 30`.
  - Pre-fix: assert the `RESOLVED` row is gone, the `RESOLVED_AUTOMATICALLY` row remains. (Documents the bug.)
  - Post-fix: assert **both** rows remain (neither has aged out yet).
  - Repeat with `status_updated_at = now() - 31 days`: assert both are deleted.
- Regression test for chunk cleanup ordering — chunks must be deleted before alerts to avoid foreign-key violations.
