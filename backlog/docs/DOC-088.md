## Description

DOC-083 established `developer-guides/api-reference` as the canonical home for HTTP endpoint documentation, with ~46 endpoints across 6 H2 sections after migrating embedded fragments out of feature pages. DOC-087 rounded out the home with `## Alerts`, `## Lineage`, and `## Glossary` sections, bringing the total to **~86 endpoints across 9 sections** on a single 300+ line page.

The original DOC-083 plan called out a soft promotion threshold (30+ endpoints) at which the canonical home would split from a single page into a **hub page + per-feature sub-pages** (`developer-guides/api-reference/{section}.md`), with the hub carrying a one-line summary per section + cross-links and the feature-specific endpoint lists living on each sub-page. DOC-083 deferred the split until the next API-surface accretion to avoid splitting an already-shipped page; DOC-087 was that accretion. The Phase D decision in DOC-087 (2026-05-03) was to **defer the split to a separate item** rather than bundle it into DOC-087 — keeping the structural refactor (anchor rerouting + SUMMARY restructuring + cross-link sweep) on its own review surface so the DOC-087 round-out reviews cleanly on its content additions alone.

This item is the structural split.

## Context

- **Discovered during DOC-087 Phase D (2026-05-03).** Phase D explicitly delegated the structural split to a separate item so the round-out review (Alerts + Lineage + Glossary content + Phase C collapses) lands cleanly without anchor-rerouting churn.
- **Why not bundle into DOC-087.** DOC-087's commits add content; the (b)-promotion replaces the canonical home's URL shape. Bundling means the round-out review has to also verify every rewired anchor — multiplies the review cost without value. The two items are sequenced (DOC-088 builds on DOC-087's content) and the split is a recurring maintenance pattern (DOC-079, DOC-081, DOC-085 are the per-page slug-realignment analogues), so a dedicated item gets a focused review surface.
- **Why this is medium and not high.** The single-page canonical home is fully functional today — it just is large and will keep accreting. The split improves discoverability and rendering-cost-per-section but is not blocking any operator-facing flow. The sequence after DOC-088: DOC-084 (adapter vocabulary refactor + odd-tracing-gateway sub-page) lands its API content directly on the new per-feature sub-page rather than against the single-page shape.

## Implementation Notes

- **Anchor preservation strategy.** GitBook anchors are derived from H1 / H2 text; per-sub-page H1 = section name preserves `#alerts` etc. as `developer-guides/api-reference/alerts.md` (which renders at the URL slug). The single-page anchors on `developer-guides/api-reference.md#alerts` are NOT preserved by the split — every cross-link rewires.
- **Live-site verification scope.** Every sub-page (9 URLs) + the hub page + every page that carried a cross-link to a previously-anchored section. Plan ~15 URLs in Gate 8.
- **Do not split the OpenAPI specifications + Swagger UI sections.** Those are about the whole API; they live on the hub.
- **Sequence vs DOC-084.** DOC-084 (adapter vocabulary refactor + odd-tracing-gateway page) will probably accrete a new "Tracing Gateway" or similar API surface. Decide before DOC-088 starts whether DOC-084 lands first (DOC-088 then includes the new sub-page) or DOC-088 lands first (DOC-084 adds a new sub-page directly to the hub). Either order works; the open question is which feels less like churn.
