## Findings

### F-001: Email notification config documents wrong property name for SMTP host
- **Location**: `documentation/docs/configuration-and-deployment/odd-platform.md` lines 242-243
- **Issue**: Docs say `notifications.receivers.email.smtp` is the SMTP host. Code uses `notifications.receivers.email.host`. A user following the docs will set the wrong property and email notifications won't work.
- **Evidence**: `EmailSenderProperties.java` binds `@ConfigurationProperties("notifications.receivers.email")` with field `host` (line 11). `application.yml` line 188 shows commented `#      host:`. Docs say `notifications.receivers.email.smtp: {smtp_host}` and env var `NOTIFICATIONS_RECEIVERS_EMAIL_SMTP={smtp_host}`.
- **Severity**: critical
- **Suggested fix**: Replace `smtp` with `host` in both YAML and env var examples. Correct env var is `NOTIFICATIONS_RECEIVERS_EMAIL_HOST`.

### F-002: Email config missing 3 required properties
- **Location**: `documentation/docs/configuration-and-deployment/odd-platform.md` lines 242-246
- **Issue**: Docs omit `notifications.receivers.email.protocol`, `notifications.receivers.email.smtp.auth`, and `notifications.receivers.email.smtp.starttls`. Without `protocol`, the `JavaMailSender` bean throws `IllegalArgumentException("protocol is empty")`.
- **Evidence**: `EmailSenderProperties.java` lines 13-17: `protocol` field + `SmtpProperties` inner class with `auth` and `starttls`. `application.yml` lines 190-193 show all three commented out. `NotificationConfiguration.java` line 49 throws on blank protocol.
- **Severity**: critical
- **Suggested fix**: Add `protocol`, `smtp.auth`, `smtp.starttls` to both config variable list and YAML/env examples.

### F-003: Docs list nonexistent env var NOTIFICATIONS_RECEIVERS_SLACK_PLATFORM-BASE-URL
- **Location**: `documentation/docs/configuration-and-deployment/odd-platform.md` line 287
- **Issue**: Env var section lists `NOTIFICATIONS_RECEIVERS_SLACK_PLATFORM-BASE-URL` which doesn't exist as a config property. No code binds this. The platform base URL is set via `odd.platform-base-url` (env var: `ODD_PLATFORM_BASE_URL`), which IS separately listed — making the Slack-specific one both wrong and confusing.
- **Evidence**: `grep -r "SLACK.*PLATFORM.BASE.URL\|slack.*platform-base-url"` across odd-platform returns zero matches. YAML example correctly places it under `odd:` section, not under `notifications.receivers.slack`.
- **Severity**: high
- **Suggested fix**: Remove `NOTIFICATIONS_RECEIVERS_SLACK_PLATFORM-BASE-URL` line from env var section. Only `ODD_PLATFORM_BASE_URL` is needed.

### F-004: AlertManager webhook endpoint completely undocumented
- **Location**: Missing from all documentation pages
- **Issue**: `POST /ingestion/alert/alertmanager` endpoint exists in `AlertManagerController.java`, accepting AlertManager-format webhook payloads (labels, generatorURL, startsAt). Users with Prometheus/AlertManager setups cannot discover this integration point.
- **Evidence**: `AlertManagerController.java` lines 21-25: `@PostMapping(path = "ingestion/alert/alertmanager")`. `ExternalAlert.java`: labels (Map), generatorURL (URI), startsAt (LocalDateTime).
- **Severity**: high
- **Suggested fix**: Add section to alerting docs describing AlertManager integration, payload format, and setup.

### F-005: Per-entity alert halt configuration undocumented
- **Location**: Missing from all documentation pages
- **Issue**: Platform supports disabling alert notifications per data entity, per alert type, for configurable durations (30min to 1 week). API endpoint `GET/PUT /api/dataentities/{id}/alert_config` and full UI exist. Users cannot discover this feature from docs.
- **Evidence**: `AlertHaltConfigServiceImpl.java`, OpenAPI spec at line 1514: `/api/dataentities/{data_entity_id}/alert_config`. UI component `AlertTypeRange.tsx` shows 4 alert types with 5 duration options.
- **Severity**: high
- **Suggested fix**: Add section describing per-entity notification snooze/halt feature, including UI walkthrough.

### F-006: Resolved alert auto-cleanup (housekeeping) not documented
- **Location**: Missing from alert notifications configuration section
- **Issue**: Resolved alerts are automatically deleted after 30 days (configurable via `housekeeping.ttl.resolved_alerts_days`). Users should know this to avoid surprise data loss when investigating historical alerts.
- **Evidence**: `AlertHousekeepingJob.java` lines 28-33: deletes alerts with RESOLVED or RESOLVED_AUTOMATICALLY status older than `resolvedAlertsDays`. `application.yml` line 168: `resolved_alerts_days: 30`.
- **Severity**: medium
- **Suggested fix**: Add note to alert notifications section about automatic cleanup and the config key to adjust retention.

### F-007: Alert auto-resolution behavior not described
- **Location**: `documentation/docs/Features.md` lines 62-79 (alerting section)
- **Issue**: Docs mention a "resolve" button but don't explain that alerts can auto-resolve (e.g., when a previously failed job succeeds). Code has `RESOLVED_AUTOMATICALLY` status distinct from user-initiated `RESOLVED`. Users may be confused by alerts disappearing without manual action.
- **Evidence**: `AlertStatusEnum.java` line 13: `RESOLVED_AUTOMATICALLY(3)`. `AlertActionResolverImpl.java` handles automatic resolution logic based on subsequent successful runs.
- **Severity**: medium
- **Suggested fix**: Document the three alert statuses (Open, Resolved, Resolved Automatically) and explain auto-resolution triggers.

### F-008: Alert UI views (All/My Objects/Dependents) not documented
- **Location**: `documentation/docs/Features.md` lines 62-79
- **Issue**: Features page describes alerts generically. Actual UI has three distinct views — "All" (all alerts), "My Objects" (alerts for entities the user owns), "Dependents" (alerts for downstream entities). These correspond to separate API endpoints. Users don't know about these filtering capabilities.
- **Evidence**: `AlertsTabs.tsx` lines 20-37: three tabs with links to `alertsPath('all')`, `alertsPath('my')`, `alertsPath('dependents')`. `AlertController.java`: `getAllAlerts`, `getAssociatedUserAlerts`, `getDependentEntitiesAlerts`.
- **Severity**: medium
- **Suggested fix**: Update alerting section to describe the three alert views and their use cases.

### F-009: Backwards-incompatible schema detection logic undocumented
- **Location**: `documentation/docs/Features.md` line 68 ("Backwards incompatible schema change")
- **Issue**: Docs mention the alert type name but don't explain what triggers it. Code detects: missing dataset fields (column deletions/renames), missing transformer sources/targets, missing consumer inputs. Users don't know what constitutes a "backwards incompatible" change.
- **Evidence**: `AlertLocatorImpl.java`: `locateInDSDelta` (lines 49-66) detects missing fields by comparing penultimate vs latest schema version. `locateInDTDelta` (lines 68-89) detects missing sources/targets. `locateInDCDelta` (lines 91-101) detects missing inputs.
- **Severity**: medium
- **Suggested fix**: Add brief explanation of what changes trigger each alert type.
