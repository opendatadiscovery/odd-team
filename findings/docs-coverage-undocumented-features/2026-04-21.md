## Findings

### F-036: Attachments feature completely undocumented — users must read source code
- **Location**: Missing from all documentation pages
- **Issue**: The attachment system is a full subsystem: `DataEntityAttachmentController.java` (7 endpoints), `AttachmentService` + `FileService` + `LinkService`, LOCAL and REMOTE (S3/MinIO) storage backends, chunked upload support, file download, max-file-size config. The UI has `OverviewAttachments/` components. Config in `application.yml` lines 215-224 includes `attachment.storage` (LOCAL/REMOTE), `attachment.max-file-size`, `attachment.local.path`, and `attachment.remote.{url,access-key,secret-key,bucket}`. A real user (Stefano Messina) had to dig through source code to discover REMOTE storage was needed for pod-persistent file storage. The default LOCAL storage writes to `/tmp/odd/attachments` which is lost on pod restart.
- **Evidence**: `DataEntityAttachmentController.java` (116 lines, 7 endpoints). `application.yml` lines 215-224. `OverviewAttachments/` UI component. `DATA_ENTITY_ATTACHMENT_MANAGE` permission in components.yaml. User report confirms zero discoverability.
- **Severity**: high
- **Suggested fix**: Create documentation page covering: attachment types (files + links), storage backends (LOCAL vs REMOTE), S3/MinIO configuration, chunked upload API, max file size, and the critical note that LOCAL storage is ephemeral in container deployments.
- **Cross-ref**: enriched by F-060 in `findings/docs-accuracy-config-options/2026-04-21-platform.md` — severity escalated to critical

### F-037: GenAI / AI Assistant feature completely undocumented
- **Location**: Missing from all documentation pages
- **Issue**: The platform has a GenAI integration: `GenAIController.java` (POST `/api/genai/ask`), `GenAIService` with configurable external AI service URL, request timeout, and enabled/disabled toggle. Config in `application.yml` lines 17-20: `genai.enabled` (default: false), `genai.url`, `genai.request_timeout`. The service proxies natural-language questions to an external AI endpoint and returns responses. This is a user-facing feature (exposed via API) with zero documentation.
- **Evidence**: `GenAIController.java`, `GenAIServiceImpl.java` (53 lines), `GenAIProperties.java`, OpenAPI tag `genai`, `application.yml` lines 17-20. Feature toggle via `genai.enabled`.
- **Severity**: high
- **Suggested fix**: Document the GenAI feature: what it does, how to configure the external AI endpoint, and how to enable it. Note any requirements for the external service API contract.

### F-038: Query Examples feature completely undocumented
- **Location**: Missing from all documentation pages
- **Issue**: Query Examples is a full feature with: dedicated UI page (`/data-modelling/query-examples`), `QueryExampleController.java` (11 endpoints including CRUD, search, facets, suggestions), `QueryExampleService`, `QueryExampleSearchService`, dedicated search with facets, 7 RBAC permissions (`QUERY_EXAMPLE_*`). Query examples can be linked to datasets and terms. The feature has its own top-level navigation section ("Data Modelling") in the UI. Zero documentation exists.
- **Evidence**: `QueryExampleController.java` (125 lines, 11 endpoints). UI: `DataModelling/QueryExamples.tsx`, `QueryExampleDetails/`, `QueryExampleForm/`, `QueryExampleSearchResults/`. Routes: `queryExamplesRoutes.ts`. OpenAPI tag: `queryExample`. 7 permissions in components.yaml: `QUERY_EXAMPLE_DATASET_CREATE/DELETE`, `QUERY_EXAMPLE_TERM_CREATE/DELETE`, `QUERY_EXAMPLE_CREATE/UPDATE/DELETE`.
- **Severity**: high
- **Suggested fix**: Create documentation page for Query Examples under a "Data Modelling" section. Cover: creating/editing examples, linking to datasets and terms, searching, and permissions.

### F-039: Directory feature undocumented
- **Location**: Missing from all documentation pages
- **Issue**: The Directory is a top-level UI page (`/directory`) with its own controller (`DirectoryController.java`, 4 endpoints), service (`DirectoryService`), and route module. It provides a browsable hierarchy: data source types → data sources → entity types → entities. This is a primary navigation mechanism for users who prefer browsing over searching, but it's not mentioned in any documentation.
- **Evidence**: `DirectoryController.java` (52 lines). `directoryRoutes.ts` (62 lines). UI: `Directory/` component directory. OpenAPI tag: `directory`. 4 API endpoints: `/api/directory`, `/api/directory/datasources`, `/api/directory/datasources/{id}/types`, `/api/directory/datasources/{id}`.
- **Severity**: medium
- **Suggested fix**: Document the Directory feature as a browsable data source catalog. Explain the hierarchy and how it complements the Search feature.

### F-040: Data Modelling section undocumented (parent of Query Examples + Relationships)
- **Location**: Missing from all documentation pages
- **Issue**: "Data Modelling" is a top-level UI section (`/data-modelling`) containing two sub-features: Query Examples and Relationships. It has its own route module, tab navigation (`DataModellingTabs.tsx`), and is a primary navigation entry. The section itself is never mentioned in docs — users wouldn't know this page exists.
- **Evidence**: `DataModelling.tsx`, `DataModellingRoutes.tsx`, `DataModellingTabs.tsx`. Routes: `dataModelling.ts` (base path), `queryExamplesRoutes.ts`, `relationshipsRoutes.ts`. Two tabs visible in UI.
- **Severity**: medium
- **Suggested fix**: Create a "Data Modelling" section in docs that introduces the concept and links to Query Examples and Relationships sub-pages.

### F-041: Custom Links (additional navigation links) feature undocumented
- **Location**: Missing from all documentation pages
- **Issue**: The platform supports configurable additional links via `LinksController.java` (GET `/api/links`) backed by `AdditionalLinkProperties`. Administrators can add custom navigation links (title + URL) that appear in the platform UI. This is a deployment/customization feature with no documentation.
- **Evidence**: `LinksController.java` (37 lines). `AdditionalLinkProperties` config class. OpenAPI tag: `links`. Endpoint: `/api/links`.
- **Severity**: medium
- **Suggested fix**: Document the additional links config property in the platform configuration page.

### F-042: Integration Wizard feature undocumented
- **Location**: Missing from all documentation pages
- **Issue**: The platform has an Integration management system: `IntegrationController.java` (2 endpoints), `IntegrationService`, UI Management section (`/management/integrations`), and a dedicated route with integration detail pages. This appears to be the in-platform wizard for setting up data source connections. Zero documentation exists for this feature.
- **Evidence**: `IntegrationController.java` (28 lines). `managementRoutes.ts` includes `INTEGRATIONS` with detail pages (`integrationsPath` with `integrationId` param). OpenAPI tag: `integration`. Endpoints: `/api/integrations`, `/api/integrations/{integration_id}`.
- **Severity**: medium
- **Suggested fix**: Document the Integration Wizard — how to use the Management > Integrations page to set up data source connections.

### F-043: Data Quality dashboard page undocumented
- **Location**: Missing from all documentation pages
- **Issue**: The platform has a top-level Data Quality page (`/data-quality`) with dedicated UI components: `DataQuality.tsx`, `DataQualityContent.tsx`, `DataQualityFilters/`, `DataQualityStore/`, `TestResults/`. While DQ test results are partially documented in Features.md, the dedicated DQ dashboard page itself — with its filters, test result views, and store management — is not documented as a navigable feature.
- **Evidence**: `dataQualityRoutes.ts` (route: `/data-quality`). UI: `DataQuality/` component directory with 7 files. Separate from the per-entity test reports documented in Features.md.
- **Severity**: medium
- **Suggested fix**: Document the Data Quality dashboard as a standalone page. Explain filters, how it differs from per-entity test reports, and the overall DQ monitoring workflow.
