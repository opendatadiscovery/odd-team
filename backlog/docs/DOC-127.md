## Description

Editorial audit at post-merge state of `feature/docs-coherence-closeout` surfaced a **cross-page coherence drift** on the use-cases subtree. The parent page `docs/use-cases.md` carries 5 H3 section titles introducing each use case; the same use cases also exist as child pages with H1 titles + SUMMARY entries. The two surfaces drift on three of the five use cases.

| use-cases.md H3 (L3-12) | Child page H1 + SUMMARY entry | Drift |
|---|---|---|
| L3 `### Data Compliance for Data Scientists` | `# Data compliance for Data Scientists` (use-cases/dc-data-compliance.md L1) + SUMMARY L16 `[Data compliance for Data Scientists]` | Capital `C` on parent vs lowercase `c` on child + SUMMARY |
| L5 `### Deprecation for Data Engineer \ Analyst` | `# Deprecation for Data Engineer \ Analyst` (use-cases/de-deprecation.md L1) + SUMMARY L17 same | No drift ✓ |
| L7 `### Data quality and visibility for Quality Engineer` | `# Visibility for Data Quality Engineer` (use-cases/dq-visibility.md L1) + SUMMARY L18 `[Visibility for Data Quality Engineer]` | Different headline framing |
| L9 `### Data preparation for Viz Engineer` | `# Data preparation for Visualization Engineer` (use-cases/viz-preparation.md L1) + SUMMARY L19 `[Data preparation for Visualization Engineer]` | `Viz` (abbreviation) on parent vs `Visualization` (full) on child + SUMMARY |
| L11 `### Service Provider and Pre-Sales` | `# Service Provider and Pre-Sales` (use-cases/service-presales.md L1) + SUMMARY L20 same | No drift ✓ |

A reader navigating via the parent page sees one set of names; the same reader navigating via SUMMARY/sidebar or landing on a child page directly sees a different set of names for the same content. This is **cross-page coherence drift** of the kind the editorial audit catches that the per-item gates do not (the per-item gates fire on what is *authored*, not on what *drifts across pages*).

## Failure shape

**A8 (Vocabulary consistency) + A3 (Cross-page coherence)** — same canonical concept named consistently across all surfaces (per `pillars/documentation/quality-framework.md`).

## Acceptance Criteria

Three drifts to align. The canonical form is the **child page H1 + SUMMARY entry** (these two already match — they form a 2-of-3 majority among the three surfaces and are the form the live site's navigation panel displays). Pick one path:

- [ ] **Path A (recommended): align use-cases.md H3s with the child + SUMMARY canonical form.** Three edits:
  - L3 `### Data Compliance for Data Scientists` → `### Data compliance for Data Scientists` (lowercase `c`).
  - L7 `### Data quality and visibility for Quality Engineer` → `### Visibility for Data Quality Engineer`.
  - L9 `### Data preparation for Viz Engineer` → `### Data preparation for Visualization Engineer`.
- [ ] **Path B (alternative): align child page H1s + SUMMARY entries with the parent's H3 names.** Higher cost (touches 6 files) and conflicts with the child pages' established titles; not recommended.
- [ ] Live-site verification (Gate 8 / `/review`): WebFetch `https://docs.opendatadiscovery.org/use-cases` and confirm the 5 H3 titles match the SUMMARY sidebar entries char-for-char.
- [ ] Sources footer: `Sources: none (cross-page coherence sweep, no factual claim)`.

## Context

- **Why low**: each drift is small; no operator hits a hard failure. A user reading the parent first then clicking [Read more] sees a slightly different page title than the H3 they came from — minor cognitive friction. But the world-class doc bar is "every alias logged, every term consistent" — a child-page title that drifts from the same-page H3 that links to it is the failure shape this rule catches.
- **Why log at all**: per Cornerstone 2 (aspect-level deep dive with single canonical source) and the world-class mission, a name drift between a parent's section heading and the child page that section opens with is a vocabulary-consistency bug. The doc tree must read as a single coherent product; named twice in two slightly different ways within one click is the failure mode.
- **Why surfaced now**: the parent page (use-cases.md) was modified by DOC-107 (Path B trim) and DOC-085 (rename); the H3 drifts are pre-existing — they were not introduced by this batch. The editorial audit catches them at post-merge state because the audit reads the page end-to-end as the doc owner. None of the per-item gates fire on this class.
- **Recurring pattern**: every parent page that uses H3 sections to introduce child pages should be audited for H3-vs-child-H1 alignment. Worth folding into `playbooks/doc-product-editorial-read.md` failure-shape table as an A8/A3 sub-shape: "parent H3 vs child H1 drift".
