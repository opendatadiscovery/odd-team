## Description

DOC-086 review (2026-05-03) verified the canonical-home `## GenAI Configuration` H2 at `docs/configuration-and-deployment/odd-platform.md:1001-1036` against the consumer code. The bullet for `genai.url` (line 1006) currently reads:

> `genai.url` (string, env `GENAI_URL`) — base URL of the external AI service. … **No `@ConfigurationProperties` default — the Java field default is the empty string.** The example in `application.yml` line 19 (`# url: http://localhost:5000`) is commented out, not a default.

The "**Java field default is the empty string**" claim is incorrect. `GenAIProperties.java` declares the field as:

```java
@ConfigurationProperties("genai")
@Data
public class GenAIProperties {
    private boolean enabled;
    private String url;             // <-- no initializer; Java field default is null, not ""
    private int requestTimeout;
}
```

Java's default for an uninitialized object-type field is `null`. With `genai.url` commented out in `application.yml`, Spring's `Binder` leaves the field at its Java default — `null`, not `""`. The behaviour at the bean-construction site is the same end-state ("WebClient with no working baseUrl" → first request fails) so the operator-facing claim "set explicitly when enabling" stays correct, but the technical detail is wrong and a careful reader checking the code will see `null` where the doc says `""`.

The same wrong claim was carried in `docs/genai.md` `## Configuration` (the parallel surface DOC-086 collapsed); the genai.md surface is now a one-paragraph cross-link, so the only remaining surface carrying the error is the canonical home. Fix in place.

The DOC-086 commit footer (`a32c1e9`) `Config-consumer:` line repeats the same wrong claim ("Java field defaults `false`, `""`, `0`"). Commit footers are an audit trail of the implementer's reads; the actual doc surface is what matters for operator-facing correctness, but this is also worth flagging for future-implementer hygiene.
