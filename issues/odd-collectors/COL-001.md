## What

The AWS SSM secrets backend (`odd-collector-sdk`) fetches plugin parameters with a single `get_parameters_by_path` call and does not paginate. The AWS default `MaxResults` for that API is **10**. A collector with more than 10 plugin parameters under `collector_plugins_prefix` silently starts with only the first 10 plugins; every plugin past the 10th is dropped with **no error, no log warning, and no startup failure**.

The failure is silent by construction — the response body includes a `NextToken` for the subsequent page, which the current code never reads. Operators only notice the issue by comparing the number of SSM parameters against the number of plugins the collector actually instantiated at startup.

## Where

`odd-collector-sdk/odd_collector_sdk/secrets/aws/ssm_parameter_store.py:130-153`

```python
def _get_secrets_by_prefix(self, prefix: str, decrypt: bool = True) -> list[dict]:
    try:
        response = self._ssm_client.get_parameters_by_path(
            Path=prefix, WithDecryption=decrypt, Recursive=True
        )
        secrets = response.get("Parameters", [])   # truncated to 10 by AWS default MaxResults
        return secrets
    except self._ssm_client.exceptions.ParameterNotFound as e:
        logger.info(f"ParameterNotFound: {e}")
        return []
```

Downstream consumer is `get_plugins()` on the same file at lines 168–177, which feeds `CollectorConfigLoader._build_collector_config` (`odd_collector_sdk/domain/collector_config_loader.py:48`). From there, the truncated list becomes the active plugin set with no cross-check against the local YAML or a count validation.

## Why it matters

- **Silent data loss in production.** A collector fleet scaled beyond 10 plugins per collector silently drops metadata. The affected plugins' data sources simply never get ingested, and the platform receives no signal that anything is wrong.
- **Affects every deployment using the secrets backend.** The feature's primary use case is production deployments (the AWS SSM provider is the only backend currently shipped; teams that adopt it are teams running in a managed AWS environment).
- **The defect is not discoverable via docs.** The ODD documentation now carries a "Known limitations" caveat describing the 10-cap (`https://docs.opendatadiscovery.org/configuration-and-deployment/collectors-secrets-backend` → Known limitations), but the only real fix is SDK-side pagination — the doc caveat is a stopgap, not a resolution.
- **Precedent.** The ODD project had a silent S3-region data-loss caveat in the platform (`MinioConfig.java` never calling `.region(...)`) that shipped undetected until an operator lost production attachments. This is the same class of silent, operator-invisible failure mode.

## Suggested fix

Use the boto3 SSM paginator, which iterates `NextToken` transparently:

```python
def _get_secrets_by_prefix(self, prefix: str, decrypt: bool = True) -> list[dict]:
    try:
        paginator = self._ssm_client.get_paginator("get_parameters_by_path")
        secrets: list[dict] = []
        for page in paginator.paginate(
            Path=prefix, WithDecryption=decrypt, Recursive=True
        ):
            secrets.extend(page.get("Parameters", []))
        return secrets
    except self._ssm_client.exceptions.ParameterNotFound as e:
        logger.info(f"ParameterNotFound: {e}")
        return []
```

This matches the canonical boto3 pattern for SSM `get_parameters_by_path` and handles arbitrary parameter counts without special casing.

Optional follow-on improvement: log the parameter count at startup (`logger.info("Loaded %d plugins from SSM under %s", len(secrets), prefix)`) so operators can sanity-check the count against their SSM parameter inventory, even after the fix lands. This would have surfaced the current defect long before the 2026-04-23 re-audit.

## How discovered

Surfaced during a Gate-5 (unset-parameter) audit on the boto3 SSM client construction in `ssm_parameter_store.py`, while reviewing ODD documentation item DOC-013 (Collector secrets backend page). The consumer-read pass enumerated the SDK builder parameters and followed the call sites — `get_parameters_by_path` without a paginator stood out because pagination is the canonical pattern for that API.

A doc caveat has already shipped (see the Known limitations section on the public docs page linked above) so operators are not blindsided while this issue is open. The SDK fix is the real resolution; the doc caveat can be removed once a release with pagination lands.
