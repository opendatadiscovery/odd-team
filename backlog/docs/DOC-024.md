## Description

Features.md mentions the "Backwards incompatible schema change" alert type by name but doesn't explain what triggers it. `AlertLocatorImpl` detects three distinct cases: missing dataset fields (column deletions/renames), missing transformer sources/targets, and missing consumer inputs. Users receiving this alert have no way to understand why short of reading code.

## Acceptance Criteria

- [ ] Features.md explains what triggers the "Backwards incompatible schema change" alert
- [ ] All three detection paths are covered: dataset field deletions/renames, transformer source/target removal, consumer input removal
- [ ] A brief note explains the comparison logic (penultimate vs latest ingested version)

**Caveats surfaced and shipped in scope:**
- **First ingest never fires BIS.** Detection requires a non-empty penultimate version; the `locateInDSDelta` short-circuit at line 51-53 is the canonical implementation. Documented as an info admonition.
- **Type changes also trigger BIS, not only column removals.** Field-key includes `(oddrn, type)` so changing a column's data type on the same ODDRN is detected as a "missing field" with the old type. Documented in the table's Detail column.
- **Renames are detected as deletions.** The renamed column has a new ODDRN that produces a different key from the penultimate version. Documented in the table's Detail column.
- **Adding fields, sources, targets, or inputs does NOT trigger.** Only removals (and dataset type changes). Stated explicitly in the opening paragraph.
- **BIS alerts do not auto-resolve.** Cross-linked to the lifecycle subsection so operators understand the alert sticks until manually resolved.

**No upstream issues filed** — detection logic is internally consistent; operator-impact behaviors are all in the doc.
