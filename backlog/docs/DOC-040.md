## Description

`docs/deployment.md` is a 5-line broken stub currently on disk. It contains:

- 3 empty section headers (`## Helm Chart`, `### Platform`, `### Collector`)
- An orphan paragraph about `Tracing Gateway` with multiple defects: typo `Platfrom`, Cyrillic-homoglyph-corrupted `Gеуway`, typo `standartize`, literal `(???)` placeholder

The page is not currently linked from `SUMMARY.md`, so readers don't land on it — but it's an abandoned-mid-write artefact that competes for the name "deployment" and should be fixed rather than left to rot.

Chosen direction (per user): **populate**, not remove. The page becomes a **Deployment Options landing guide** covering every supported deployment path end-to-end with step-by-step instructions, examples, and prerequisites. It sits above the existing `configuration-and-deployment/` subtree as the reader's entry point: "which path is right for me?" → pick a section → follow the steps inline or jump to the deep-dive page.

## Implementation Notes

- **Start with the comparison table.** It forces clarity on what each option is *for*. Write the table first, then write each section to match its table row.
- **Lean on includes/transclusion sparingly** — GitBook supports link-style cross-references cleanly; this is better than copying chunks of deep-dive pages and letting them drift.
- **Production-readiness tags** (Not production / Evaluation only / Production-ready) should be a visible admonition at the top of each option section, not buried in prose. Operators making choices skim for this.
- **Chart versions**: don't hard-code chart versions in examples (they'll go stale in weeks). Either use `helm repo update` + latest, or show a `--version` placeholder and point at the chart repo's release page.
- **`odd-docs` repo is not in the local workspace.** The implementer must fetch it (e.g. `gh repo clone opendatadiscovery/odd-docs /tmp/odd-docs` or `gh api repos/opendatadiscovery/odd-docs/contents/...`) and cross-check any deployment material there against what's in `documentation/` and `charts/`. Where `odd-docs` has content not mirrored in `documentation`, port what's still accurate; flag anything contradictory as a separate finding rather than silently resolving it.
- **Tracing Gateway fact-check**: before writing that section, read `charts/charts/odd-tracing-gateway/README.md` and `values.yaml` and (if present) the source repo for the tracing gateway, so the description isn't guesswork. The existing stub's description is unreliable — rebuild it from primary sources.
- **Final review pass**: after writing, read the page as a brand-new ODD user who only knows "I want to run this on my Kubernetes cluster." If they can't pick an option and run it inside 10 minutes, the page is not done.
