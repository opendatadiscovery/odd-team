## Description

`docs/GLOSSARY.md` contains 5 empty section headers (`ODD Specification`, `Adapter`, `Pull strategy`, `Push-client`, `Push strategy`) with no bodies. The file is not listed in `SUMMARY.md`, but GitBook treats `GLOSSARY.md` as a reserved filename (legacy auto-link behavior), and `Overview.md` carries a commented-out link to it — so the stub is visible to readers of the source and at risk of becoming a surprise published page.

Two related problems make simple "populate the stub" the wrong fix:

1. **Name collision.** ODD Platform has a first-class **Business Glossary** feature (term entities, term-to-term linking, term-to-data-entity linking). A doc page called "Glossary" directly competes with and confuses that feature. Users clicking a sidebar "Glossary" entry expect the product feature, not a definitions list.
2. **Scope mismatch.** Five term headers are not what new users need. Onboarding hinges on understanding the **architecture chain and core abstractions** — how the pieces fit together — not isolated definitions.

The fix is to replace the stub with a **Main Concepts** onboarding page that orients a new reader to the system before they dive into Features / Configuration / Use cases.

## Acceptance Criteria

- [ ] `docs/GLOSSARY.md` is deleted
- [ ] A new page `docs/main-concepts.md` (or equivalent path — avoid any filename containing "glossary" to prevent confusion with the Business Glossary feature) is created
- [ ] The new page is linked from `SUMMARY.md` in an onboarding-appropriate position (suggested: immediately after `Overview` and before `Architecture`, so readers meet the vocabulary before the architecture diagram uses it)
- [ ] The commented-out `[Glossary](GLOSSARY.md)` line in `Overview.md:37` is removed or replaced with a link to the new page
- [ ] The page explicitly calls out the distinction between this docs section and the in-app **Business Glossary** feature, so readers aren't confused about which one they've landed on
- [ ] No broken links remain anywhere (grep `GLOSSARY.md` in the documentation repo must return zero hits after the change)
- [ ] Every concept is grounded in existing authoritative docs (`oddrn.md`, `Architecture.md`, `Features.md`, collector READMEs) — no new contradictory definitions

### Content scope for the Main Concepts page

At minimum, the page must cover:

- [ ] **The architecture chain**: Data Source → Adapter → Collector → Platform — one concise paragraph per link in the chain, explaining what each component does and where it lives (which repo, which process). A small diagram is strongly preferred.
- [ ] **ODDRN** (Open Data Discovery Resource Name): what it is, why it exists (stable cross-system identity), and a link to `oddrn.md` for the full reference.
- [ ] **Push vs Pull strategies**: how metadata gets to the platform. Pull = collector polls the source on a schedule; Push = source (or a push-client) sends metadata directly. Mention which adapters support which, and when an operator would pick one over the other.
- [ ] **Adapter vs Collector vs Push-client**: the three producer-side abstractions disambiguated — adapter is the per-source integration code, collector is the process that runs one or more adapters on a schedule, push-client is a library embedded into a data-producing application.
- [ ] **ODD Specification**: the ingress API contract (`opendatadiscovery-specification` repo), why it exists (decouples producers from the platform), and link out.
- [ ] **Data Governance aspects** — a short pointer section (not a rewrite) covering the governance features the platform surfaces: ownership, tags, lineage, Business Glossary (terms), policies/permissions, alerts and data quality. Each bullet links to the canonical page in `Features.md` or its subsection.
- [ ] **AI aspects** — a short pointer section covering AI/GenAI capabilities: GenAI assistant, DataProfiler-driven profiling, any ML-experiment surfacing. Again, links out rather than rewrites. (Coordinate with DOC-045 GenAI page — if that page doesn't exist yet when this is implemented, link to the existing `Features.md` GenAI reference and note the follow-up.)

Each concept should be 2-5 sentences. The page is a **map**, not a manual — its job is to give a new reader the mental model and route them to the deep-dive pages.

## Implementation Notes

- **Do not name the file `glossary.md` in any case**. Even lowercase collides semantically with the Business Glossary feature. `main-concepts.md` or `core-concepts.md` is the right shape.
- In `SUMMARY.md`, the title should be "Main Concepts" (or "Core Concepts"). Avoid "Glossary" / "Terminology" — both imply a definitions dictionary, which is the product feature.
- Before writing each concept, read the authoritative source (listed above) so the wording matches and doesn't drift. Push/pull descriptions in particular are worth cross-checking against the collector SDK README.
- If a diagram is added for the architecture chain, put the asset under `docs/.gitbook/assets/` (or wherever existing diagrams live) and reference it inline.
- Leave a one-line disclaimer at the top of the page: something like "This page introduces the core vocabulary of the ODD project. For the platform's in-app Business Glossary feature (term entities you can link to data), see [Features → Business Glossary]."
