## Description

Editorial audit on 2026-05-03 (during `/review batch:feature/docs-features-md-spotcheck`) surfaced two **additional** absolute `https://docs.opendatadiscovery.org/...` URLs on `docs/Features.md` that DOC-092's `grep -rn 'source=post' docs/` sweep did not catch (the Medium-tracking URLs DOC-092 fixed all carried the `?source=post_page-...` query parameter, but these don't):

### Site A — line 700 (Data Entity Statuses → housekeeping TTL)

```
A Time-to-Live (TTL) mechanism is implemented by [**housekeeping.ttl.data\_entity\_delete\_days**](https://docs.opendatadiscovery.org/configuration-and-deployment/odd-platform#housekeeping-settings-configuration) for a data entity with the DELETED status.
```

Per `pillars/documentation/authoring.md` "GitBook authoring" — intra-repo cross-links must be **relative paths**. Should read:

```
A Time-to-Live (TTL) mechanism is implemented by [**housekeeping.ttl.data_entity_delete_days**](configuration-and-deployment/odd-platform.md#housekeeping-settings-configuration) for a data entity with the DELETED status.
```

### Site B — line 780 (Recommended panel → User-Owner Association)

```
**Note**: to be able to see the "Recommended" panel on the ODD's main page the User-Owner Association shall be implemented first. More details can be found in ODD's [User-owner association documentation](https://docs.opendatadiscovery.org/configuration-and-deployment/enable-security/authorization/user-owner-association).
```

Should be:

```
**Note**: to be able to see the "Recommended" panel on the ODD's main page the User-Owner Association shall be implemented first. More details can be found in ODD's [User-owner association documentation](configuration-and-deployment/enable-security/authorization/user-owner-association.md).
```

Both are the same defect class as the three Medium URLs DOC-092 fixed (Sites 2-4 in DOC-092). They render correctly on the live site today (the absolute URL points at the same docs site we publish to), but per the GitBook authoring rule, intra-repo links must be relative — absolute URLs:

- Break offline rendering (e.g., when the doc is read in the GitBook editor preview, in a fork, or in a clone).
- Bypass GitBook's link-validation: a relative path is checked against the doc tree at publish time; an absolute URL is not.
- Persist stale anchors when the target page is renamed (relative paths break loudly at publish; absolute URLs silently 404).

## Why this slipped past prior batches

DOC-092's bidirectional-sweep step `grep -rn 'source=post' docs/` was scoped to the Medium-tracking URL pattern. The DOC-092 ACs explicitly mentioned that step as a "sweep for sibling Medium-URL patterns" — but did not extend to the broader "any absolute `https://docs.opendatadiscovery.org/` URL on a doc page". The two URLs above don't carry the Medium tracking parameter, so they were invisible to the regex.

This is the same regex-too-narrow miss that DOC-090 recognised in the canonical-home enumeration sweep (the original DOC-083 hint admonition's section list). Pattern: an authoring sweep matches the *exact shape* of the past defect rather than the *defect class*.

Worth folding into `playbooks/duplication-sweep.md` or a Gate-1 extension: every commit that touches a feature-page cross-link should sweep `grep -rn 'https://docs.opendatadiscovery.org' docs/` (or `grep -rEn 'https?://docs\.opendatadiscovery\.org' docs/`) and classify each hit.

Editorial-audit case-law for `playbooks/doc-product-editorial-read.md` — failure shape: Cross-link mismatch (link uses an absolute URL where a relative path is required).
