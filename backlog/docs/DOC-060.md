## Description

DOC-018 removed the nonexistent `NOTIFICATIONS_RECEIVERS_SLACK_PLATFORM-BASE-URL` and clarified that `odd.platform-base-url` is shared between Slack and email senders. A consumer-read re-audit found a **third** consumer of the same key, with a **different default**, that is not mentioned in the documentation anywhere.

`StaticArgumentMappingContext.java:16`:

```java
public StaticArgumentMappingContext(
    @Value("${odd.platform-base-url:http://your.odd.platform}")
    final String platformUrl
) {
    this.parameterToValue = Map.of(PLATFORM_URL_PARAM_NAME, platformUrl);
}
```

This consumer exposes `odd.platform-base-url` as a `platform_url` parameter substituted into integration configurations (external tools like Airflow plugins, dbt artifacts, etc. that reference the ODD platform URL). The default value is `http://your.odd.platform` — a placeholder-looking string that is obviously different from the notification consumers' `http://localhost:8080` default.

Operator impact:
- If `odd.platform-base-url` is unset, integrations that use `{platform_url}` substitution receive `http://your.odd.platform` — which looks like a misconfigured-template placeholder and will not connect to anything.
- An operator reading the DOC-018 fix believes `odd.platform-base-url` is a notifications-only concern, misses that integrations also consume it, and may not set it when deploying integrations-only.

## Acceptance Criteria

- [ ] The `odd.platform-base-url` doc entry lists all three consumer code paths (email notifications, Slack notifications, integration-parameter substitution)
- [ ] The inconsistent default in `StaticArgumentMappingContext` is documented: default value `http://your.odd.platform` (not `http://localhost:8080` as the notifications consumers use)
- [ ] A warning admonition explicitly says: operators deploying integrations must set `ODD_PLATFORM_BASE_URL` even if they do not use notifications
- [ ] The documentation links to `navigation/domains/notifications.md` and the integrations domain (or equivalent published doc) for each consumer

## Implementation Notes

The inconsistent defaults are themselves a code smell (a single config key should not have two defaults across consumers). Consider drafting a companion GitHub Issue on `odd-platform` to unify the default (probably by removing the inline default from `StaticArgumentMappingContext` and inheriting from `application.yml`). Do not modify platform code from this docs item — log the issue and link it.
