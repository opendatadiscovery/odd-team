## Description

Three platform-level `odd.*` config keys are entirely undocumented:

- `odd.tenant-id` — multi-tenancy identifier, no default (line 210)
- `odd.data-entity-stale-period` — days before an entity is marked stale if not re-ingested (default: 7, line 211)
- `odd.activity.partition-period` — PostgreSQL activity-feed partition period in days (default: 30, line 213)

`data-entity-stale-period` particularly matters for users running collectors on schedules longer than 7 days — they'll see entities marked stale unexpectedly. `partition-period` matters for high-volume deployments where the default partition granularity is wrong.

## Acceptance Criteria

- [ ] All three keys are documented with defaults, types, and env-var equivalents
- [ ] `data-entity-stale-period` explanation includes what "stale" means in the UI/API (how it surfaces to users)
- [ ] `tenant-id` documentation states clearly that it's intended for multi-tenant deployments and has no default
- [ ] `activity.partition-period` is grouped with activity-feed settings (or placed in an advanced/operational subsection) and explains when an operator would tune it

**What shipped** (commit `fd8d78f` on `feature/docs-odd-platform-config`, `documentation` repo):

1. **Section restructured** — "Detecting Stale Metadata" renamed to "Platform-level settings (`odd.*`)" with three key-scoped subsections. Anchor changed from `#detecting-stale-metadata` to `#platform-level-settings-odd` — no external references existed (verified via grep before rename).
2. **`odd.data-entity-stale-period`** documented with UI/API surfacing explanation (labels entity as "Stale") and operator guidance for long collector cadences. Integer, days, default `7`.
3. **`odd.tenant-id`** — **scope-corrected.** The original finding framed this as "multi-tenancy identifier, no default"; the consumer-code audit (ExternalMetricReader + 4 TimeSeriesExtractor classes) showed it is actually a Prometheus metric label, ignored entirely when `metrics.storage=INTERNAL_POSTGRES`. Documented accurately as "Prometheus tenant label" with cross-link to the metric-storage-backend subsection. Acceptance criterion #3 ("intended for multi-tenant deployments") is met in a narrower, accurate form: the key enables multi-tenant *metric separation* when sharing a Prometheus instance, not general multi-tenant deployment isolation.
4. **`odd.activity.partition-period`** — documented with operational tuning guidance (narrower partitions help high-volume vacuum). Placed in own "Activity-feed partitioning" subsection satisfying AC #4.
5. **Env-var typos fixed** (incidental, in-scope):
   - `ODD_TENANT-ID_DATA_EMTITY_STALE_PERIOD=7` (nonsense single key, with a dash and a misspelling) → split into `ODD_DATA_ENTITY_STALE_PERIOD=7` + `ODD_TENANT_ID=my-odd-deployment` (two distinct keys).
   - `ODD_ACTIVITY_PARTITION-PREIOD=30` (dash + "PREIOD" typo) → `ODD_ACTIVITY_PARTITION_PERIOD=30`.
   - Spring Boot relaxed binding requires underscores, not dashes, in env-var keys; the prior forms did not bind.
6. **`odd.platform-base-url` deliberately not duplicated** — its only consumers are in the notifications path and it is already fully documented above in the "Enable Alert Notifications" section. A pointer from the new section's lede preserves discoverability.
