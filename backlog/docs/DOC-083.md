## Description

User feedback 2026-04-30 surfaced a class of failure that the existing Quality Bar (Gates 1-9) did not catch: **content of a recognizable type authored on a feature page when a canonical home for that type exists elsewhere in the doc tree, but is empty or sparse.** The trigger case:

- `https://docs.opendatadiscovery.org/master-data-management/lookup-tables#api-reference` carries ~1300 words of detailed HTTP endpoint tables (16 endpoints across 4 groups: Table CRUD, Column CRUD, Row CRUD, Search). Operation IDs, paths, methods, response schemas — all there.
- `https://docs.opendatadiscovery.org/developer-guides/api-reference` carries only a "use Swagger UI" pointer. No endpoints, no operation IDs, no schemas. It is the dedicated page that should be the canonical home for HTTP endpoint documentation, but it sits empty.

This is **not** Gate 1 duplication (the content isn't duplicated; it exists in only one place). It is a **content-type homing failure**: API-reference content lives on a feature page when the canonical home for that type is the dedicated page.

The 2026-04-30 bigbatch shipped four MORE feature pages with the same drift class:

- `docs/configuration-and-deployment/odd-platform.md` `### API surface` for data-collaboration (DOC-034) — 7 routes across 3 tables (Outbound to Slack, Per-entity threads/history, Inbound webhook).
- `docs/directory.md` `## API surface` (DOC-047) — 4 endpoints (Directory hierarchy levels 1-4).
- `docs/integrations/integration-wizard.md` `## API surface` (DOC-050) — 2 endpoints (`getIntegrationPreviews`, `getIntegration`).
- (existing) `docs/lookup-tables.md` `## API reference` — 16 endpoints (the trigger case).

In every case the work shipped reads well in isolation, but the global picture is: **API-reference content is scattered across feature pages, the dedicated `developer-guides/api-reference` is empty, and a developer asking "where do I find every HTTP endpoint?" has no canonical answer.**

The Quality Bar was extended on 2026-04-30 with **Cornerstone 5 (One canonical home per content type)** and **Gate 10 (Content type homing)** to prevent recurrence. This item is the systemic refactor that brings the existing tree up to the new bar.

## Acceptance Criteria

### Phase A — Audit (run first, on disk before authoring)

- [ ] **Content-type homing audit on `docs/`.** Grep for `## API reference`, `## API surface`, `### API surface`, `## API endpoints`, `## HTTP API`, and similar. Produce a table of every embedded API-reference fragment, file:line, endpoint count, and current content scope. The four feature pages above are confirmed; the audit may surface more.
- [ ] **Confirm or extend the canonical-home choice.** Today: `docs/developer-guides/api-reference` (page exists, mostly empty). Decide whether the page should be:
  - (a) **Single page** with sections per feature/area (Reference Data, Data Collaboration, Directory, Integrations, …) — simpler navigation, can grow large.
  - (b) **Hub + per-feature sub-pages** (`developer-guides/api-reference/README.md` + `developer-guides/api-reference/reference-data.md`, etc.) — scales better but creates a new sub-tree to maintain.
  - The Cornerstone-5 decision: pick (a) initially; promote to (b) when the single page exceeds ~3000 words or 30+ endpoints. Document the threshold in the page itself so future maintainers know when to split.
- [ ] **Sweep the canonical home's current content** to understand what's already there ("Use Swagger UI" pointer + Ingestion vs Platform spec links + the operationId-less prose). Decide what to keep as the page's introduction (Swagger UI is a legitimate "how to interactively test" section) and what to pull out.

### Phase B — Migrate (the canonical home becomes authoritative; feature pages link)

For each embedded API-reference fragment from Phase A:

- [ ] **Move the content** to `developer-guides/api-reference` under a section per feature/area (e.g., `## Reference Data`, `## Data Collaboration`, `## Directory`, `## Integrations`).
- [ ] **Preserve the Sources footer.** Each migrated section carries the same `Spec:` / `Builder:` lines that backed the original embedded version — Gate 9 doesn't relax just because content moved.
- [ ] **Replace the embedded section on the feature page** with a 1-2 sentence pointer + link to the canonical home's section. Example pattern: `For the full HTTP API surface (operation IDs, paths, request/response schemas), see [API Reference → Reference Data](developer-guides/api-reference.md#reference-data).`
- [ ] **In-page TOC sync** (Gate 7): if the feature page's in-page TOC contained the now-removed H2/H3, update the TOC row in the same commit.
- [ ] **Update bidirectional cross-links**: the API reference page's per-feature section links back to the feature page in the section header or footer. Cornerstone 3-style bidirectionality applied to API-reference content.
- [ ] **Live-site verification** (Gate 8) per affected page after merge: confirm the feature page renders the new pointer + link; confirm the API reference page renders the migrated section; confirm anchors resolve.

### Phase C — Hub coverage on Features.md

- [ ] Add a 1-line cross-link from `Features.md` (or `developer-guides/README.md` if it exists) noting that the canonical API surface lives at `developer-guides/api-reference`. This is the discoverability path for developers landing on Features and wondering "where's the API?"

### Phase D — Cornerstone-5 audit on other content types

The same homing failure exists for other content types in subtler forms. Sweep and log follow-up items if warranted:

- [ ] **Configuration reference fragments** on feature pages — feature pages occasionally embed YAML examples or env-var tables that should be cross-linked to `configuration-and-deployment/odd-platform.md`. Spot-check `docs/lookup-tables.md` and `docs/genai.md` and `docs/integrations/integration-wizard.md`. Log DOC-NNN if drift is found.
- [ ] **ADR fragments** — feature pages occasionally embed decision rationale that should live in `adrs/`. Spot-check the longer feature pages.
- [ ] **Glossary fragments** — terminology defined inline on feature pages that should be a `main-concepts.md` Terms & Aliases row instead. Spot-check during the same sweep.

These are scope-deferred unless the audit surfaces a specific instance — log a separate DOC-NNN per content type as needed.

### Phase E — Update CLAUDE.md Cornerstone 5 table

- [ ] When Phase A picks (a) or (b) for the canonical-home shape, update the CLAUDE.md Cornerstone 5 table's "API reference" row to reflect the chosen shape. The current table marks the page as "today: stub only; DOC-083 expands to the canonical home" — this should resolve to the actual chosen shape when DOC-083 ships.

## Implementation Notes

- **Phase A is the longest single block** — the audit must be exhaustive before Phase B begins, otherwise we'll re-discover drift mid-migration. Run on a clean branch from `documentation` `origin/main`; produce the audit table as part of the work item's body before authoring any migration.
- **Per-feature commits**, batched on a single branch. Each migration (one feature page → corresponding section on the canonical home) is its own commit so reviewers can verify per-feature correctness; reviewer Gate 8 (live-site) runs once per batch after merge.
- **The canonical home page name**: `developer-guides/api-reference` is the existing slug. After this refactor it should also be the live URL `https://docs.opendatadiscovery.org/developer-guides/api-reference` with substantive content. Verify the slug doesn't drift from SUMMARY hierarchy (DOC-079 / DOC-081 lesson).
- **Be explicit about what Swagger UI complements vs replaces**: the canonical API reference page should explain the relationship — Swagger UI is for interactive testing of a running platform; the doc reference is for design-time understanding (what endpoints exist, what they do, when to use them, links to the OpenAPI source). Don't make the doc page redundant with Swagger; make it the place a developer reads *before* opening Swagger.
- **Sources footer per migrated section**: every migrated table carries forward its original `Spec:` and `Builder:` lines. The Sources footer for the api-reference.md change will be long but legitimate — Gate 9 expects every claim to trace.
- **Live-site verification scope**: every feature page that loses an embedded section AND the api-reference page itself need WebFetch verification post-merge. Plan for ~10-15 URLs in Gate 8.
- **Cornerstone-5 table update** in CLAUDE.md is part of this item, not a follow-up — the table currently marks this page as "DOC-083 expands"; closing DOC-083 means the table reflects the actual shipped shape.
