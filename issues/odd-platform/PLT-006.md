## What

Navigating to a term's overview page (`/terms/{term_id}/overview`) renders a blank white page with no in-app error message when the term's response payload contains a linked `TermRef` whose `namespace` is `null`. The page is effectively inaccessible — operators cannot view, edit, or remove the term from the SPA.

The trigger is two collaborating bugs:

1. **Backend (root cause)**: the term-details SQL aggregation references the wrong `NAMESPACE` table alias, so the JSON-aggregated `ASSIGNED_TERM_NAMESPACES` array carries only the parent term's namespace. Looking up a linked term whose namespace differs from the parent's returns `null`, and the `LinkedTermDto` is serialized with `namespace: null`. This violates the spec contract, which declares `TermRef.namespace` as `required`.
2. **Frontend (crash site)**: a non-defensive lazy `useState` initializer dereferences `term.namespace.name` for every linked term. With a `null` namespace it throws an unhandled `TypeError` during the first render. No error boundary wraps the term-overview subtree, so React unmounts the whole tree → blank page.

Reproduced on the public demo: `https://demo.oddp.io/terms/10/overview` (term "Customer Segmentation" in namespace "Samples", whose description references `[[Business:Business to Business (B2B) customer]]` — a linked term in a different namespace).

## Where

### Backend — wrong table alias in the term-details query

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/repository/reactive/ReactiveTermRepositoryImpl.java:194-238`, line 211:

```java
.select(jsonArrayAgg(field(NAMESPACE.asterisk().toString())).as(ASSIGNED_TERM_NAMESPACES))
```

`NAMESPACE` here is the outer `NAMESPACE` table joined to the parent term at line 218, **not** the namespaces of the assigned/linked terms. The correctly-aliased table is declared at line 196 and joined at 232-233:

```java
// line 196
final Table<NamespaceRecord> assignedTermsNamespace = NAMESPACE.asTable("assigned_terms_namespace");
...
// lines 232-233
.leftJoin(assignedTermsNamespace)
    .on(assignedTerms.field(TERM.NAMESPACE_ID).eq(assignedTermsNamespace.field(NAMESPACE.ID)))
```

The aggregation at line 211 ignores `assignedTermsNamespace` and aggregates the outer `NAMESPACE` instead. Consequence: the resulting JSON array contains only the parent term's namespace.

`ReactiveTermRepositoryImpl.extractTerms` (lines 610-636) then builds a namespace lookup map from that aggregation and resolves each linked term:

```java
// line 614-617 — map keyed by namespace ID, but only the parent's namespace is present
final Map<Long, NamespacePojo> namespaces = jooqRecordHelper
    .extractAggRelation(record, ASSIGNED_TERM_NAMESPACES, NamespacePojo.class)
    .stream()
    .collect(Collectors.toMap(NamespacePojo::getId, identity()));

// line 628 — returns null for any linked term whose namespace differs from the parent's
.namespace(namespaces.get(pojo.getNamespaceId()))
```

The bug therefore affects **any** linked term (description-linked or manually-attached) whose namespace differs from the parent term's namespace.

### Spec contract violation

`odd-platform-specification/components.yaml:2551-2570` declares `namespace` as required:

```yaml
TermRef:
  type: object
  properties:
    id:
      type: integer
      format: int64
    name:
      type: string
    definition:
      type: string
    updated_at:
      type: string
      format: date-time
    namespace:
      $ref: '#/components/schemas/Namespace'
  required:
    - id
    - name
    - definition
    - namespace
```

The current `GET /api/terms/{id}` response with `terms[].term.namespace = null` violates this schema. Generated clients that trust the contract will misbehave.

### Frontend — non-defensive lazy initializer

`odd-platform-ui/src/lib/hooks/useTermWiki.ts:55-57`:

```ts
const [fetchedTerms, setFetchedTerms] = useState<Record<string, TermRef>>(() =>
  Object.fromEntries(
    terms.map(term => [makeTermKey(term.namespace.name, term.name), term])
  )
);
```

When `term.namespace === null`, `term.namespace.name` throws `TypeError: Cannot read properties of null (reading 'name')` synchronously inside the lazy initializer of the first render. The hook is mounted by `odd-platform-ui/src/components/Terms/TermDetails/Overview/TermDefinition/TermDefinition.tsx:21`, which has no error boundary around it — React unwinds the term-details subtree, and the operator sees a blank page with no in-app error.

The same dereference pattern recurs in the matchAll loops further down (`useEffect` at line 60 and `handleMarkdownChange` at line 95) for any future call paths that build a `makeTermKey` from a `TermRef`.

### Live API evidence — data is intact, only the aggregation strips it

The same term row (id 35, "Business to Business (B2B) customer") is reachable through two endpoints. The direct path returns the namespace correctly; the linked-term aggregation path returns `null`. This rules out missing or deleted data — the row is fine in the database, the bug is purely in the JSON-aggregation step inside `getTermDetailsDto`.

**Buggy path — `GET /api/terms/10` (parent term, linked-term aggregation):**

```bash
$ curl https://demo.oddp.io/api/terms/10
{
  "id": 10,
  "name": "Customer Segmentation",
  "namespace": { "id": 5, "name": "Samples" },
  "terms": [{
    "term": {
      "id": 35,
      "name": "Business to Business (B2B) customer",
      "namespace": null            <-- contract violation
    },
    "is_description_link": true
  }]
}
```

**Correct path — `GET /api/terms/35` (linked term fetched directly):**

```bash
$ curl https://demo.oddp.io/api/terms/35
{
  "id": 35,
  "name": "Business to Business (B2B) customer",
  "namespace": { "id": 34, "name": "Business" },   <-- the data IS there
  "terms": []
}
```

The direct path uses `mapRecordToRefDto` (`ReactiveTermRepositoryImpl.java:534-539`), which reads `record.into(NAMESPACE).into(NamespacePojo.class)` from the row produced by the simple `TERM JOIN NAMESPACE ON NAMESPACE.ID = TERM.NAMESPACE_ID` (line 218). The buggy path uses `extractTerms` (lines 610-636), which depends on the `ASSIGNED_TERM_NAMESPACES` aggregation at line 211 — and that aggregation is taken from the wrong table alias.

**Three-step reproduction on the public demo:**

1. `curl https://demo.oddp.io/api/terms/10` → observe `terms[0].term.namespace == null` (contract violation; symptom of the aggregation bug).
2. `curl https://demo.oddp.io/api/terms/35` → observe `namespace == {id: 34, name: "Business"}` (same DB row, correct value via direct fetch — proves the data exists).
3. In a browser logged into the demo, open `https://demo.oddp.io/terms/10/overview` → observe a blank white page. DevTools console will show `TypeError: Cannot read properties of null (reading 'name')` originating in `useTermWiki.ts`.

Steps 1 and 2 isolate the failure to the aggregation; step 3 demonstrates the operator-visible consequence.

## Why it matters

- **Term page is unreachable.** Any term whose definition references a term in a different namespace via the wiki-link pattern (`[[Namespace:Term]]`) renders a blank page. Operators cannot view, edit, link, or delete the term from the SPA, and there is no in-app error message — they have to open DevTools to even know what failed.
- **Hits the documented happy path.** Cross-namespace term references are the canonical use case for the wiki-link feature — the in-app help tooltip on every term-definition input shows `[[Finance:User]]` as the example pattern (`TermDefinition.tsx:28-31`, `InternalDescriptionHeader.tsx:23-26`, `DatasetFieldDescription.tsx:46-49`). The bug specifically breaks the example operators are guided toward.
- **Silent at the API layer.** The backend returns `200 OK` with a payload that violates its own published OpenAPI schema. No server log, no exception, no metric — only the SPA fails, and only as an unhandled JS exception. Generated clients in any language that trust the contract will misbehave too.
- **Reproducible on the public demo.** `https://demo.oddp.io/terms/10/overview` is a working reproduction today, so the bug is verifiable independently of any local data setup.

## Suggested fix

Two independent fixes are recommended; either alone removes the white page, but both are wanted for defense in depth and to bring the API back into spec compliance.

### Backend — primary fix (restores spec contract)

`odd-platform-api/.../repository/reactive/ReactiveTermRepositoryImpl.java:211`:

```diff
- .select(jsonArrayAgg(field(NAMESPACE.asterisk().toString())).as(ASSIGNED_TERM_NAMESPACES))
+ .select(jsonArrayAgg(field(assignedTermsNamespace.asterisk().toString())).as(ASSIGNED_TERM_NAMESPACES))
```

After the fix, `extractTerms` line 628 finds the correct namespace for every linked term, and the API response stops emitting `namespace: null`.

### Frontend — defense in depth

`odd-platform-ui/src/lib/hooks/useTermWiki.ts:55-57` — filter out terms missing a namespace before keying, so a future contract violation does not crash the page:

```diff
- Object.fromEntries(
-   terms.map(term => [makeTermKey(term.namespace.name, term.name), term])
- )
+ Object.fromEntries(
+   terms
+     .filter(term => term.namespace?.name)
+     .map(term => [makeTermKey(term.namespace.name, term.name), term])
+ )
```

Apply the same null-safe guard in any other site that constructs `makeTermKey` from a fetched `TermRef`.

### Tests to add

- **Backend (integration, Testcontainers Postgres)**: seed term A in namespace X with a `TERM_TO_TERM` row pointing to term B in namespace Y; call `getTermDetailsDto(A.id)`; assert the returned `LinkedTermDto.term.namespace.id == Y.id`. Pre-fix this fails with `namespace == null`; post-fix it passes.
- **Frontend (unit)**: render `TermDefinition` with a `terms` array containing an entry whose `namespace` is `null`; assert no throw and that the unresolved wiki-pattern is left as raw text (since the lookup cannot complete).

## How discovered

Found while interactively browsing the public demo to verify navigation between term-search results and term-detail pages. Of the 13 terms in the default term-search, term 10 ("Customer Segmentation") was the only one whose detail page rendered as a blank screen; the others loaded normally.

A direct `curl https://demo.oddp.io/api/terms/10` revealed the spec-violating payload (`terms[0].term.namespace = null`). Reading `odd-platform-ui/src/lib/hooks/useTermWiki.ts` identified the unhandled null deref on the SPA side; tracing back through `ReactiveTermRepositoryImpl.extractTerms` → `getTermDetailsDto` led to the misaligned table reference at line 211.

A second-pass validation directly fetched `/api/terms/35` and confirmed the namespace is intact in the database (`{id: 34, name: "Business"}`). Same term row, same column, same join target — the field is correctly returned by the direct query path and only stripped by the linked-term aggregation. This isolates the failure to the JSON-aggregation step and rules out any data-integrity confounder.
