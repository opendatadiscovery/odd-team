## Description

DOC-006 corrected the `SESSION_PROVIDER` example value from `IN_MEMORY` to `INTERNAL_POSTGRESQL` — a 1-line fix. A consumer-read re-audit of `SessionConfiguration.java` and the housekeeping job surfaced operator-visible behaviors for each of the three providers that are not documented anywhere.

Findings:

- **`IN_MEMORY`** — `ReactiveMapSessionRepository` wraps a raw `ConcurrentHashMap`. Sessions are only evicted by Spring Session's expiry check; there is no secondary eviction policy. A long-running process with many short-lived sessions can accumulate map entries until the Spring Session TTL expires them. Under aggressive load this is an OOM risk. Not documented.

- **`INTERNAL_POSTGRESQL`** — `PostgreSQLSessionHousekeepingJobHandler.deleteExpiredSessions()` is annotated `@Scheduled(fixedRate = 1, timeUnit = TimeUnit.HOURS)`. Expired sessions therefore sit in `SPRING_SESSION` / `SPRING_SESSION_ATTRIBUTES` for up to **one hour** past their TTL before the job cleans them. The cadence is not configurable. Not documented — an operator sizing the Postgres pool / disk will under-estimate session-table growth.

- **`REDIS`** — the `@EnableRedisWebSession` bean has no local connection wiring; operators must supply `spring.data.redis.*` (or `spring.redis.*` on older Spring Boot) to configure host, port, credentials, TLS, and pool sizing. None of these keys are documented in `odd-platform.md`. Without them, the platform will fail to start against a non-default Redis deployment.

## Review-ready notes (2026-04-26)

Implemented on branch `feature/docs-odd-platform-operator-caveats`. Restructured the entire "Select session provider" section in `docs/configuration-and-deployment/odd-platform.md` from a tabs-based shared layout into per-provider H3 subsections, each with a "Characteristics & caveats" H4 list and its own YAML/env-var example tabs.

Three additional accuracy corrections shipped in the same change:

1. **Obsolete Redis property prefix.** The pre-existing docs used `spring.redis.*` keys (Spring Boot 2.x form), but odd-platform runs Spring Boot 3.4.10 — Spring Boot 3.0 relocated all Redis properties under `spring.data.redis.*` and removed the legacy prefix entirely. Every Redis config snippet (YAML and env-var) was migrated to `spring.data.redis.*` / `SPRING_DATA_REDIS_*`. A `hint style="warning"` admonition was added at the end of the REDIS subsection explicitly warning operators that the legacy prefix is silently ignored — this is the most-likely upgrade footgun for anyone migrating from a Spring Boot 2.x ODD Platform deployment.
2. **`INTERNAL_POSTGRESQL` repository implementation correctly identified.** ODD Platform implements a custom `JooqSessionRepository` rather than using the standard Spring Session JDBC. The new `INTERNAL_POSTGRESQL` subsection makes this explicit so operators don't fruitlessly look for `spring.session.jdbc.*` keys.
3. **Hardcoded housekeeping cadence documented.** The `@Scheduled(fixedRate = 1, timeUnit = HOURS)` annotation in `PostgreSQLSessionHousekeepingJobHandler` is hardcoded — the new subsection documents both the cadence and the sizing implication (up to one hour of post-expiry stragglers in the session tables).

**Caveats surfaced (shipped in the same commit):**
- **In-scope, shipped:**
  - IN_MEMORY: OOM risk under high-traffic deployments running with the shipped `spring.session.timeout: -1` default — documented in the IN_MEMORY caveats list with an explicit pointer to set a finite timeout.
  - INTERNAL_POSTGRESQL: hourly hardcoded housekeeping cadence and the up-to-1-hour post-expiry overhang in `SPRING_SESSION` / `SPRING_SESSION_ATTRIBUTES` — documented in the INTERNAL_POSTGRESQL caveats list with sizing implications.
  - REDIS: `spring.redis.*` legacy prefix silently ignored on Spring Boot 3.x — documented as a `hint style="warning"` admonition under the REDIS subsection. This is the most-impactful caveat (silent connection failure with confusing fallback to `localhost:6379`).
  - REDIS: TLS / pool sizing / timeout inherit Spring Data Redis defaults — documented inline in the caveats list.
- **Out-of-scope, not logged as upstream issue:**
  - The hourly housekeeping cadence could be made configurable (a `@Scheduled` cron expression backed by a config property would be a small change). Severity is low — operators can tolerate the 1-hour overhang once they know about it. The doc caveat is the operator-facing fix; no PLT issue drafted.
