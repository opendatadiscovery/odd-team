## Implementation log (2026-04-27 — review-ready)

Batch branch: `feature/docs-data-modelling-subtree` (documentation, commit `2da8bab`), paired with `feature/docs-data-modelling-subtree-state` (odd-team).

### Resolution

`docs/relationships.md` created. Covers:
- The two relationship classes — `ENTITY_RELATIONSHIP(25)` (API filter `ERD`, foreign-key-style edges) and `GRAPH_RELATIONSHIP(26)` (API filter `GRAPH`, graph-store edges) — both children of `DATA_RELATIONSHIP(9)` per `DataEntityClassDto.java:51`.
- The ERD cardinality model (4 classifiers from the upstream `odd-collectors/README.md` Relationships section).
- UI walkthrough — table columns, ALL/ERD/GRAPH tab strip, search input, infinite-scroll pagination at size=30.
- 3-endpoint API surface: `getRelationships` (paginated list with type filter and free-text query), `getERDRelationshipById`, `getGraphRelationshipById`.
- Per-adapter ingestion matrix: PostgreSQL and Snowflake adapters in `odd-collector` extract foreign-key constraints and emit `ENTITY_RELATIONSHIP`; **no adapter currently emits `GRAPH_RELATIONSHIP`** per the upstream Relationships matrix. Documented as a known gap (info admonition) — the GRAPH type is reserved in the platform model but the ingestion path on the collector side is not yet shipped, so operators using a Neo4j adapter today get nodes catalogued without their inter-node edges.

`docs/Features.md` `## Relationships and ERDs` H2 added between the Query Examples teaser (now collapsed by DOC-046) and `## Data Quality Dashboard`. In-page TOC entry added at line 24.

`docs/SUMMARY.md` nests `[Relationships](relationships.md)` under the new Data Modelling parent.
