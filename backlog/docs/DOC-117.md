## Description

Baseline per-page audit on 2026-05-03 (`docs/Features.md` and `docs/master-data-management/lookup-tables.md` Mode 1 axis A4 + A11f audit) surfaced a fragility class the prior reactive audits had not catalogued: **doc tree depends on Medium image hosts for 66 figure references across 2 pages**.

### The defect class

`<figure>` blocks throughout the doc tree use `<img src="https://cdn-images-1.medium.com/...">` or `<img src="https://miro.medium.com/...">` for screenshots and feature illustrations. Counts:
- `docs/Features.md` — 52 Medium-hosted images.
- `docs/master-data-management/lookup-tables.md` — 14 Medium-hosted images.

Total: 66 outbound image references on hosts ODD does not control.

### Why this is the same class as DOC-101

DOC-101 catalogued the equivalent issue for outbound documentation **URLs** (`api.slack.com/...` legacy form being phased out by Slack). The doc tree was depending on Slack's continued maintenance of redirect chains; once Slack migrated, the canonical URLs changed and the docs needed updating. The same dependency applies to Medium image hosts:

- Medium has tightened access to `cdn-images-1.medium.com` and `miro.medium.com` for non-Medium-signed-in readers in the past (returning 403 on cold-cache fetches).
- Medium can deprecate either CDN host at any time without notice (image hosting is not a contractual service for non-Medium properties).
- The images appear to have been imported into ODD docs via the GitBook editor's "paste from Medium" workflow during early authoring; they were never migrated to the doc tree's own `.gitbook/img/` or `.gitbook/assets/` directories.

When (not if) Medium changes the hosting, every affected `<figure>` renders as a broken image on the live site. Currently 66 broken images on the most-visited page (`Features.md`) is a high-impact failure even if it has not yet manifested.

## Failure shape

**A4 (reference integrity)** + **A11f (forward fidelity — claims about the platform are visually anchored to images that may not render)**. Plus **A11-reverse** signal: every Medium-hosted figure represents a screenshot of platform UI that should also exist in the local `.gitbook/img/` or `.gitbook/assets/` if the page is meant to survive Medium's hosting policy changes.

## Acceptance Criteria

Multiple paths — the choice depends on whether the project wants to (a) self-host the images, (b) regenerate them from current platform UI, or (c) accept the fragility.

- [ ] **Path A (self-host the existing images)**: download every Medium-hosted image referenced in the doc tree (66 across 2 pages); save them to `documentation/docs/.gitbook/img/` (or `.gitbook/assets/`) with stable filenames; update every `<img src="https://...medium.com/...">` to the relative `.gitbook/img/{filename}` form. This preserves the existing visual content but removes the Medium dependency.
- [ ] **Path B (regenerate from current platform UI)**: many of the screenshots are 2-3 years old and may not match the current platform UI exactly. Capture fresh screenshots of the equivalent UI states; save to `.gitbook/img/`; replace the Medium URLs. Higher effort but closes A11f-forward gaps too (some screenshots may show deprecated UI elements — phantom-feature-class A11 violations).
- [ ] **Path C (accept the fragility, document the dependency)**: log this as an accepted risk; add a comment on each affected `<figure>` noting the Medium host. Lowest effort; preserves the fragility.
- [ ] **Sweep**: `grep -rEln 'cdn-images-1\.medium\.com|miro\.medium\.com' docs/` — list every file. Currently only Features.md + lookup-tables.md, but verify no other files have crept in. Per-file count: `grep -rEcn 'cdn-images-1\.medium\.com|miro\.medium\.com' docs/`.
- [ ] **Verify image freshness** (in either Path A or B): for each affected `<figure>`, confirm the screenshot reflects current platform UI. Phantom-UI screenshots become A11f-forward violations once they're noticed.
- [ ] Live-site verification (Gate 8 / `/review`): WebFetch the affected pages and confirm every figure renders. (Live-site verification today: probably PASS because Medium hosting is up; the verification protects against the day Medium changes.)
- [ ] Sources footer: per the chosen path. Path A: `Sources: none (image-host self-migration, no factual claim)`. Path B: per the new screenshots' UI provenance.

## Context

- **Why low (not medium)**: today the images render. Today's reader experience is unaffected. Action is hygiene + fragility insulation, not a current bug.
- **Why log at all**: per the world-class-doc mission, every image / link on the doc tree is a future-maintenance liability if its host changes. Medium has already restricted CDN access at least once. The fix is well-bounded (66 images on 2 pages) and worth doing before the deprecation forces it under time pressure.
- **Recurring pattern**: every outbound asset (URLs, images, embedded-content references) on the doc tree is a candidate for self-hosting / canonicalisation. Worth a Gate-extension: every commit introducing or modifying an outbound asset cites the host's stability classification (`stable: self-hosted` / `stable: vendor-canonical` / `fragile: third-party-CDN`) in the `Sources:` footer.
