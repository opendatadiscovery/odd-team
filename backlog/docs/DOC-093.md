## Description

Discovered during the same user spot-check as DOC-091 + DOC-092 (2026-05-03). The platform uses **Slack** for two unrelated purposes — but no surface (configuration, feature pages, API reference, glossary) makes the distinction explicit.

### The two Slack integrations

| Aspect | Slack as notification target | Slack as collaboration backend |
|---|---|---|
| Feature | Alert notifications (`## Alerting` on Features.md) | Data Collaboration (DOC-091 — pending Features.md section) |
| Config H2 | `## Enable Alert Notifications` (`odd-platform.md:325`) | `## Enable Data Collaboration` (`odd-platform.md:622`) |
| Config keys | `notifications.receivers.slack.url` (Slack incoming webhook) | `datacollaboration.slack-oauth-token`, `datacollaboration.receive-event-advisory-lock-id`, `datacollaboration.sender-message-advisory-lock-id`, `datacollaboration.message-partition-period`, `datacollaboration.sending-messages-retry-count` |
| Slack-side setup | Add an [incoming webhook](https://api.slack.com/messaging/webhooks) to the target channel; copy URL into config | Create a full Slack app with OAuth scopes; install to workspace; configure Events API webhook back to ODD; copy bot OAuth token into config |
| Direction | Outbound only (platform → Slack) | Bidirectional (platform ↔ Slack via Events API) |
| What flows | Alert messages on alert-creation events | Per-entity discussion threads + replies |
| API surface | None on the platform side (webhook is operator-configured) | 7 endpoints under `/api/datacollaboration/...` + `/api/slack/events` (`developer-guides/api-reference.md#data-collaboration`) |
| Code | `notifications.*` consumers (TBD navigation entry) | `DataCollaborationProperties.java`, `SlackAPIClient.java`, `SlackMessageProviderClient.java`, `MessageServiceImpl.java` (see `navigation/domains/collaboration.md`) |

These are entirely different uses of Slack: an incoming webhook vs a full Slack app. They use different Slack-side artefacts, different config keys, different code paths. An operator configuring one is not configuring the other.

### Why the silence is dangerous

- An operator reading `## Enable Alert Notifications` first sees Slack and configures `notifications.receivers.slack.url`; later, reading `## Enable Data Collaboration`, sees Slack again and may think "I already set this up" → skips the OAuth + Events API setup → Data Collaboration silently doesn't work.
- Reverse failure: an operator reading `## Enable Data Collaboration` first sets up the full Slack app; later, reading `## Enable Alert Notifications`, sees Slack again and may think "the same Slack integration handles alerts too" → doesn't add the incoming webhook URL → alerts never reach the channel even though Slack is "configured".
- A user reading [`Features.md#alerting`](Features.md#alerting) (post-DOC-092) sees Slack and asks "is this the same Slack as the in-app discussions?" — there is currently no way to answer this from the docs.
- A developer browsing `developer-guides/api-reference.md#data-collaboration` (with its 7 routes) and `#alerts` (with its 9 routes) has no signal that the two surfaces both involve Slack but for different things.

This is **Cornerstone 4** (three audiences, AI-maintained consistency) — operators specifically need this taxonomic clarity. It is also a **Gate 2** (synonyms and aliases) failure: "Slack" is being used as an umbrella term for two distinct integrations without a `main-concepts.md` Terms & Aliases entry that resolves the ambiguity.
