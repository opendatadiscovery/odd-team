## Description

`application.yml` (lines 46-48) defines `auth.ingestion.filter.enabled` with a default of `false`. This means that even when authentication is enabled for the UI, the ingestion API (`/ingestion/*`) remains open by default. Operators enabling security for public-facing deployments may not realize that ingestion endpoints are unprotected unless this flag is set.

This is a security-sensitive default that must be visible in the security configuration documentation.

### How ingestion auth actually works

The `/ingestion/**` path is in `SecurityConstants.WHITELIST_PATHS` — the UI auth chain (OAuth2/LDAP/Login-Form) never touches it. Ingestion protection is done by two separate `WebFilter`s, with different activation rules:

| Endpoint | Filter | Active when | Behavior |
|---|---|---|---|
| `POST /ingestion/datasources` | `IngestionDataSourceFilter` | **always** (no `@ConditionalOnProperty`) | Requires `Authorization: Bearer <token>`; looks up collector by token; 401 if missing/invalid |
| `POST /ingestion/entities` | `IngestionDataEntitiesFilter` | **only when `auth.ingestion.filter.enabled=true`** | Validates Bearer token against the datasource's (or its collector's) stored token |

`auth.type` (UI auth: DISABLED / LOGIN_FORM / OAUTH2 / LDAP) and `auth.ingestion.filter.enabled` are **independent** flags. Enabling UI auth does not automatically enable ingestion filtering.

Collectors using `odd-collector-sdk` (`datasource_api.py:21-24`) always attach `Authorization: Bearer <token>` to both calls — but the platform only *validates* it on `/ingestion/entities` when the flag is on.

### Security risk when the flag is off

With `auth.ingestion.filter.enabled=false` (default) and the platform reachable on the network, any caller who can speak the ingress API can:

- **Poison existing datasources** via `POST /ingestion/entities` with a spec-valid `DataEntityList`. If the `data_source_oddrn` matches an existing datasource, fake entities/schemas/lineage/owners/tags get upserted into the catalog. Consumers see the polluted metadata as authoritative. The payload is spec-aligned, so this isn't an exploit — it's the documented ingress flow being called by an unauthenticated party.
- Not *create* a new datasource — `/ingestion/datasources` is always token-protected, so the attacker is limited to datasources already registered. `data_source_oddrn` values follow predictable patterns (e.g. `//postgresql/host:port/…`) so guessing is feasible.

The de facto control in most deployments is the network perimeter (VPN / internal network). Documentation must make this explicit so operators don't assume UI auth closes the ingestion hole.

## Acceptance Criteria

- [ ] The security README (or equivalent authentication overview page) documents `auth.ingestion.filter.enabled`
- [ ] The documentation clearly states that enabling UI auth does NOT protect ingestion endpoints by default — the two flags are independent
- [ ] The default value (`false`) is stated explicitly
- [ ] The nuance is explicit: `/ingestion/datasources` is always token-protected; `/ingestion/entities` is only protected when the flag is on
- [ ] The risk is stated concretely: an unauthenticated caller with network reach can inject spec-valid metadata into existing datasources (poisoning catalog contents, lineage, owners, alerts-adjacent state) when the flag is off
- [ ] A recommendation is given for when the flag should be enabled (any deployment where the platform is internet-accessible or where collectors run outside a trusted network perimeter — essentially: any non-local-dev deployment)
- [ ] The env-var equivalent is listed
- [ ] A cross-reference is added to the known limitation that the filter only covers `/ingestion/entities` (see the separate code bug on unprotected ingestion sub-paths) so readers aren't falsely reassured

## Implementation Notes

A warning callout/admonition is appropriate here — the default is deliberately lax for zero-config local deployments but can cause catalog-integrity issues if missed in production. Lead with the risk (what a caller can do), then the flag, then the recommendation. Avoid burying the security-relevant default in a config table.
