## Description

Two doc pages claim Pandas (or Pandas Profiling) is a supported data-quality integration:

- `Features.md` lines 57-58: "The Platform is compatible with **Pandas** and **Great expectations**."
- `dq_visibility.md` line 7: "The ODD Platform is compatible with Great Expectations and Pandas Profiling."

Neither is accurate. `odd-collector-profiler` (active, last release v0.2.2 Apr 2024) uses **Capital One's DataProfiler** (`dataprofiler==0.10.9`). No pandas-profiling or ydata-profiling dependency exists anywhere in the ecosystem. Users searching for a Pandas-based integration will be misled.

## Acceptance Criteria

- [ ] `Features.md` replaces "Pandas" with accurate integration names (at minimum: "DataProfiler", "Great Expectations", "dbt tests", "Soda") — or a general "any framework that pushes via the Ingress API"
- [ ] `dq_visibility.md` keyword list no longer contains "Pandas profiling"; body text no longer makes the Pandas claim
- [ ] `odd-collector-profiler` is referenced by name in at least one of the two pages
- [ ] A short note explains that custom frameworks can push results via `/ingestion/entities/datasets/stats`

## Post-review amendment (2026-04-23, user-caught)

The "defensible" dbt-link observation in the Review notes above was **wrong**. dbt integration does not live inside `odd-collectors` — it is a separate repo `github.com/opendatadiscovery/odd-dbt` implementing the push-client approach. The user caught this immediately after the review signoff.

**Correction shipped** as commit `d82559e docs: fix wrong dbt link target in DOC-027 edit [DOC-027]` on the same `feature/docs-accuracy-features-fixes` branch:

- `Features.md:98` — "dbt tests" is now a clickable link to `github.com/opendatadiscovery/odd-dbt` and explicitly called out as a push-client alongside Great Expectations.
- `dq_visibility.md:7` — `[dbt tests]` link target changed from `odd-collectors` to `odd-dbt`, with the same push-client qualifier.

**Evidence for correction**:

- `github.com/opendatadiscovery/odd-dbt` README: "CLI tool helps run and ingest dbt test to platform" — confirms odd-dbt is the canonical dbt push-client.
- `docs/developer-guides/github-organization-overview.md:47` already lists `odd-dbt` as a separate repo in the ODD organization (distinct from odd-collectors).
- `docs/main-concepts.md:20,36` — the push-client architecture is described generically and names dbt alongside Airflow, Spark, and Great Expectations as push-client sources.

**Process observation** (not logged as a formal item, but surfaces a Quality Bar weakness):

Gate 4 (consumer-read) as currently written verifies functional *claims* against consumer code. It does not explicitly cover **outbound link-target accuracy** — e.g., "integration X is at repo Y" is a factual claim that should be verified against the actual repo contents, not against the text on the source page. The reviewer rationalized the wrong link as "defensible" rather than verifying it. A future Quality Bar refinement could extend Gate 4 to cover "every named repo URL resolves to a repo whose README describes the claimed feature" — but I'm not logging it as an ADR/backlog item unilaterally; it's a judgment call for the team lead.
