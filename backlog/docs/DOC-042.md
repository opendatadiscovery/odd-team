## Description

`docs/collector.md` is **empty** (0 lines on disk). It's the implicit landing for readers following the "Adapters will be deprecated — install the Collector" notice on `Adapters.md:7`. Those readers hit a dead end.

Chosen direction (per user): **populate** and expand into a comprehensive **Integrations hub** documenting every way metadata reaches the platform — both **collectors** (pull strategy, from the `odd-collectors` monorepo) and **push adapters** (push strategy, standalone packages like `odd-airflow-2`, `odd-dbt`, `odd-spark-adapter`, `odd-great-expectations`). The hub is the single entry point; per-integration details live on sub-pages so the landing stays scannable.

### Layout note

A single giant `collector.md` would be unreadable. Use a multi-page structure:

- `docs/collector.md` becomes a **redirect/short bridge** (1-paragraph: "The Collector landing has moved to Integrations →") and points at the new hub. This preserves any external links that already hit `collector.md`.
- `docs/integrations/README.md` is the **real hub** (overview, comparison table, "which do I need?" chooser).
- `docs/integrations/collectors/*.md` — one page per collector (generic, AWS, Azure, GCP, profiler).
- `docs/integrations/push-adapters/*.md` — one page per push adapter (Airflow, dbt, Spark, Great Expectations, CLI).

If the implementer prefers a flatter structure (all integration pages directly under `docs/integrations/`), that's fine — the requirement is clear separation of **pull collectors** vs **push adapters** in the navigation, not a specific folder layout.

### Terms & Aliases registration (Gate 9 prerequisite)

- [ ] `docs/main-concepts.md` Terms & Aliases table contains, at minimum, these rows — add in the same commit as the hub landing. Definitions must match `navigation/architecture.md` "Canonical vocabulary":
  - **Adapter** — set of scripts mapping metadata from a source system (PostgreSQL, MySQL, Airflow, Kafka, …) to the ODD specification. Push or pull. The job is extract-and-map.
  - **Collector** — container of **pull** adapters + runtime (launcher, logging, Platform-API client, config reader). A collector is what you deploy; the pull adapters inside it are the mappers. **Not** a synonym for "pull adapter".
  - **Plugin** — a *configured* adapter instance inside a collector. Carries one adapter's connection/schedule settings. One collector can host many plugins (different adapter types or repeated instances of the same type with different settings).
  - **Push adapter** (alias: **push-client**) — push-strategy adapter that runs **inside** the source system's own runtime (dbt plugin, GE checkpoint action, Airflow plugin, Spark listener). Still an adapter (extract + map); differs from a pull adapter in deployment topology (lives in the source's process, not a collector container).
  - **Integration** — umbrella term for any path metadata takes from a source into the Platform. Prefer this in user-facing prose unless direction matters.
  - (Retire in prose) **Connector** — collides with "collector"; don't introduce.
  - (Retire bare usage) **Adapter** as a deployment noun — ambiguous with the legacy pre-collector service; use "push adapter" / "pull adapter" / "collector" explicitly.
- [ ] Every usage of these terms in the hub and sub-pages matches the table (no silent aliases introduced).

## Implementation Notes

- **Sequence tip**: write the hub landing first (comparison table forces clarity on scope), then each sub-page in priority order — most-used integrations first (generic collector, AWS collector, Airflow, dbt).
- **Terminology**: pick one term for each concept and stick with it across all pages. Recommended: "collector" for pull-daemon integrations, "push adapter" for embedded-plugin integrations (matches the user's term and the `-adapter` suffix on several repos). Avoid "adapter" as a standalone noun in this hub — it collides with (a) the legacy deprecated Adapter service and (b) the per-source code inside a collector.
- **Status markers**: every integration page should have a visible status admonition at the top — Stable / Beta / Experimental / Deprecated. This is the fastest way to set expectations. Extract status from the repo's README, last release date, and open-bug volume; if unclear, flag `(status: unknown — verify)` rather than guess.
- **Known-limitations sections must be specific** ("Snowflake adapter does not capture dynamic tables as of v0.17.0" beats "some object types not supported"). Link to GH issues when the limitation is a tracked bug.
