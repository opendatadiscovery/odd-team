## Description

Editorial audit on 2026-05-03 surfaced a **coverage hole** on `docs/Use_cases.md`. The page lists 8 use cases. **Five** of them carry a `[Read more]` link to a dedicated page:

| # | Use case | Linked? |
|---|---|---|
| 1 | Data Compliance for Data Scientists | ✓ → dc_data_compliance.md |
| 2 | Deprecation for Data Engineer / Analyst | ✓ → de_deprecation.md |
| 3 | Data quality and visibility for Quality Engineer | ✓ → dq_visibility.md |
| 4 | Data preparation for Viz Engineer | ✓ → viz_preparation.md |
| 5 | Service Provider and Pre-Sales | ✓ → service_presales.md |
| 6 | Newcomer onboarding | ✗ — no link, 1-paragraph teaser only |
| 7 | Customer churn analysis for Data Scientist | ✗ — no link, 1-paragraph teaser only |
| 8 | Recommendation engine with observable and manageable pipelines | ✗ — no link, 1-paragraph teaser only |

The 5 linked use cases also appear in `SUMMARY.md` lines 16-20 as sub-pages of `Use cases`. Use cases 6-8 are mentioned only on Use_cases.md itself — they have no dedicated pages, no SUMMARY entry, and no further reading path.

A reader scrolling through Use_cases.md will read the first 5 use cases, follow [Read more], get a complete walkthrough; then for the next 3 use cases they get a single paragraph and no follow-through. This is a **reader-flow defect** — the page promises a consistent shape and breaks halfway through.

## Failure shape

**Coverage hole** + **half-finished narrative** + **reader-flow defect** (per `playbooks/doc-product-editorial-read.md` failure-shape table).

## Acceptance Criteria

Three paths — pick one:

- [ ] **Path A (author the missing pages)**: create three new pages — `newcomer_onboarding.md` (or similar), `customer_churn_analysis.md`, `recommendation_engine.md` — each following the same Challenge / Solution / Scenario / Result structure as the existing 5. Add each to SUMMARY.md under Use cases. Add `[Read more]` links from Use_cases.md. Run the Pre-authoring stance check before authoring.
- [ ] **Path B (remove the orphaned entries)**: delete the three orphaned use cases (lines 13-18) from Use_cases.md. Page becomes 5 use cases with consistent shape.
- [ ] **Path C (convert to single sentence)**: replace each orphaned multi-line entry with a single sentence noting the use case category exists but is not yet documented. Less honest than Path B but preserves the marketing intent.
- [ ] Live-site verification (Gate 8 / `/review`): WebFetch `https://docs.opendatadiscovery.org/use-cases` and confirm the use-cases list matches the chosen path's intended state.
- [ ] Sources footer: `Sources: none (use-case coverage closure, no factual claim)` for Paths B/C; for Path A, per the authored content.

## Context

- **Why medium (not low)**: coverage hole undermines reader trust — they expect every use case to have a [Read more] path. The page itself looks like a marketing landing; readers don't expect three of the eight items to dead-end.
- **Why not high**: the 5 linked use cases work; the orphaned 3 are not actively misleading — just thin.
- **Recurring pattern**: every "list of N items with [Read more] links" surface should have its links audited for completeness. Worth folding into the `playbooks/doc-product-editorial-read.md` failure-shape table as a "promised-but-empty link list" sub-shape of half-finished narrative.
