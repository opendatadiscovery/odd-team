## Description

User argument 2026-04-30: there are conceptually only **two** adapter types — pull and push — distinguished by who initiates the data flow. Everything else (in-process plugin vs standalone gateway vs SDK call) is a deployment-shape sub-axis under push, not a peer category. Concretely:

- **Pull adapter** — adapter is the initiator; knows the source endpoint, credentials, schema. Sub-shapes: hosted in a generic collector container (today's `odd-collector`, `odd-collector-aws`, `odd-collector-azure`, `odd-collector-gcp`, `odd-collector-profiler`), or conceivably standalone.
- **Push adapter** — source system is the initiator; knows the catalog endpoint, token. Sub-shapes:
  - **(a) In-process plugin / extension** — embedded in source runtime (today's `odd-dbt`, `odd-airflow-2`, `odd-spark-adapter`, `odd-great-expectations`).
  - **(b) Standalone proxy / gateway** — separate process source systems push to, which forwards to the platform (today's `odd-tracing-gateway`).
  - **(c) Direct SDK / CLI use** — custom code calling `/ingestion/entities` via `odd-models-package`, or `odd-cli push`.

The current `navigation/architecture.md` has 11 vocabulary categories that mix "what the repo is" (Platform, Spec, SDK, Deployment, Examples, Meta) with "how the adapter works" (Collector, Push adapter, Auxiliary platform component). The category mixing is the source of the confusion: `odd-tracing-gateway` lives under "Auxiliary platform component" — technically correct under the current vocabulary, but operators searching "how do my custom services report to ODD?" don't think to look under "Auxiliary platform component."

The user's argument: **collapse the adapter-related categories to a two-axis taxonomy** (Pull / Push at the top; deployment-shape underneath); keep the non-adapter categories (Platform, Spec, SDK, Tooling, Deployment-charts, Examples, Meta) as is. The Tracing Gateway becomes a Push adapter (sub-shape: standalone proxy/gateway) — not an orphan category.

## Context

- **User argument summary**: "I will still argue that there could be 2 types of adapters — pull and push — main distinguish point: 1) extracts data (initiator is the adapter) and so it knows where to take data from; 2) a system is extended to make possible to actively send information (push) into the platform. Second one could be achieved with wrappers, umbrella applications, extensions (add-ons), could be done via proxies like gateway. I could not now imagine any other type except pull/push." (2026-04-30)
- **Why the user is right**: I (Claude) tried to find a counter-example and couldn't. Bidirectional sync isn't part of ODD's catalog model. Scheduled batch ETLs collapse to either pull or push depending on who invokes. `odd-cli collect` is a runner around an embedded pull adapter (the CLI is a launcher, not a category). The data-flow direction (initiator side) is the meaningful conceptual axis; everything else is implementation detail.
- **Why the current vocabulary fails**: the current `architecture.md` mixes "what the repo is" (Platform, SDK, Deployment) with "how the adapter works" (Collector, Push adapter, Auxiliary platform component). `odd-tracing-gateway` ended up under "Auxiliary platform component" — technically correct under the existing vocabulary but operationally an orphan: operators searching "how do my custom services push to ODD?" don't think to look under "Auxiliary." The two-axis taxonomy makes the gateway findable.
- **Why this is medium priority not critical**: the current vocabulary is internally consistent and not actively misleading; the orphaning costs operators time but doesn't lose data. The high-priority work is DOC-083 (API-reference homing). DOC-084 is the next-priority Cornerstone-2-level decision.
- **Relationship to DOC-082** (IA hierarchy refactor): DOC-082's drift-3 (`Build a custom collector` next to `Build and run`) is fixed by the SUMMARY hierarchy; DOC-084's vocabulary refactor cleans up the *labels* that operators see when they navigate that hierarchy. They're complementary.
- **Relationship to DOC-083** (content-type homing): Phase E of DOC-084 explicitly follows Cornerstone 5 / Gate 10 — the new gateway sub-page links to `developer-guides/api-reference`, does not embed endpoint tables. DOC-083 should ship first or in the same batch so the canonical API reference page exists when DOC-084's gateway page links to it.
- **Relationship to DOC-078**: DOC-078 ("tighten Tracing Gateway description in deployment.md") was blocked because the upstream `odd-tracing-gateway` README is 404. DOC-084 unblocks DOC-078 by establishing the canonical home (the new sub-page) and providing source-code-as-SoT (read the gateway's actual code rather than waiting for the README). Once DOC-084 ships, DOC-078 either closes as superseded or its remaining scope (just the deployment.md collapse) absorbs into DOC-084 Phase F.
