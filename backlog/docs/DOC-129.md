## Description

Editorial audit at post-merge state of `feature/docs-coherence-followups-2026-05-05` ran a magic-link regex sweep across the doc tree (`grep -rEn '\[(here|read|click here|this|see|link|details|more)\]\('`) and surfaced 3 sites where the link text is uninformative — the reader cannot tell from the link text alone what they will get when they click. Same defect class as DOC-110 Site A (link text "technical details" → Java source file), the canonical case for A4 magic-link.

### The 3 magic-link sites

| File | Line | Defect |
|---|---|---|
| `docs/Architecture.md` | 9 | `**Push-client** is a provider which sends information directly to the central repository of the Platform. Also [read](https://github.com/opendatadiscovery/opendatadiscovery-specification/blob/main/specification/specification.md#push-model) about a push-strategy.` — link text "read" doesn't name the target (the spec's push-model section). |
| `docs/developer-guides/how-to-contribute.md` | 14 | `OpenDataDiscovery GitHub organization has over 20 repositories including ODD Platform, ODD Collectors, utilities, etc. An overview can be found [here](github-organization-overview.md).` — link text "here" is the canonical magic-link form. |
| `docs/configuration-and-deployment/trylocally.md` | 80 | `See examples [here](https://github.com/opendatadiscovery/odd-collectors/blob/main/odd-collector/README.md)` — same magic-link form as how-to-contribute.md. |

### Reader-experience cost

A reader scanning the page can't predict from the link text alone what page or section the link will deliver. Link text should describe the target so a reader's expectation matches what they get. Same rule that DOC-110 Site A applied: rephrase so the link text names the target ("FTSConstants.java" / "the push model" / "the github-organization-overview page" / "the odd-collector README's examples").

### Why this slipped past prior batches

A4 (reference integrity) audit cue covers this — "Does the link text describe what the reader will get?" — but the magic-link sub-shape (`[here]`, `[read]`, `[click here]`, `[this]`) was not run as a regex sweep across the tree until this audit. DOC-110 caught one instance via per-page audit on Features.md; the 3 hits surfaced here are tree-wide residue.

## Failure shape

**A4 (Reference integrity) — magic-link sub-shape** (per `pillars/documentation/quality-framework.md`). Link text doesn't match what the link delivers; reader has no expectation primer before clicking.

## Context

- **Why low**: each defect is a small reader-flow friction; no operator hits a hard failure. But the world-class doc bar holds that link text should match what the reader gets — magic-links are friction the reader pays per click.

Worth folding into the framework's A4 audit cue as a routine sweep that runs every few batches (similar cadence to DOC-124's cross-tree double-space sweep).
