## Description

User feedback on `/review batch:feature/docs-bigbatch-2026-05-06` (2026-05-06): Features.md should carry a **front-of-page disclaimer** explaining the page's purpose, audience, and place in the doc-product's IA. Today the page opens with a 35-line manual TOC (lines 3-37) and then dives straight into `## Metadata Storage` — a reader has to infer from skimming that:

1. Features.md is the **one-page index** of the most important features, intended as a quick-discovery surface.
2. The detailed feature documentation lives on **separate per-pillar / per-feature pages** grouped by the Data Governance pillars (Data Discovery, Data Modelling, Master Data Management, Management) and by Category (developer-guides, configuration-and-deployment, integrations).
3. Each H2 section on Features.md ends with **cross-links to the canonical homes**: the per-feature deep-dive page, the API Reference sub-page, and the Configure ODD Platform section that hosts the configuration keys.

A front-of-page purpose statement makes those three facts explicit, so a reader landing on Features.md knows immediately what they're looking at and where the deeper content lives.

### Why this matters

Features.md is the most-trafficked page after the Catalog Overview hint info-block at line 39-41 already hints at the pattern ("Looking for the HTTP API? Every Platform endpoint is documented at API Reference"). But that hint is buried below the manual TOC and is API-only — it doesn't capture the broader pattern that **every** feature on this page has a per-pillar canonical home, a per-feature deep-dive, configuration keys, and an API surface, all of which live on dedicated pages.

A reader who lands on Features.md from a Google search for "ODD lookup tables" sees a one-paragraph summary at line 725-729 with two cross-links (to the master-data-management/lookup-tables.md page + the master-data-management.md pillar landing). That's the **right** pattern — but the page's H1 + first paragraph never tell the reader the pattern is intentional. A reader who reads top-to-bottom may go through 30 features before realising "oh — this is the index, not the manual."

### Failure shape

**Purpose-not-declared / front-of-page orientation missing** (per `playbooks/doc-product-editorial-read.md` failure-shape catalog — sub-shape worth adding to the catalog if not already present). The page's role in the IA is implicit; readers infer it from skimming. The world-class doc bar is that **every page declares its own purpose at the top** so a reader can decide in 5 seconds whether they're on the right page for their question.

### What the disclaimer should say

A short paragraph (3-5 lines) immediately after the H1 "Features" and before the manual TOC, conveying:

1. **Purpose**: "This page is the one-page index of ODD Platform's most important features — a quick-scan surface for discovering what the platform does."
2. **Audience**: "Read this page when you're orienting yourself to the platform's capabilities or browsing for a feature you've heard mentioned. For step-by-step walkthroughs and configuration detail, follow the cross-links from each feature's section to its dedicated page."
3. **Pattern**: "Each feature below carries cross-links to (a) its **per-pillar canonical home** (Data Discovery / Data Modelling / Master Data Management / Management — the four governance pillars), (b) its **API surface** ([API Reference](developer-guides/api-reference.md) hub + per-feature sub-pages), and (c) its **operator-configuration keys** ([Configure ODD Platform](configuration-and-deployment/odd-platform.md))."
4. **Cross-link to map**: "For the broader vocabulary and the Data Governance map showing which pillars ODD covers, see [Main Concepts](main-concepts.md)."

The exact wording should be authored against the existing line 39-41 hint info-block (which can stay as-is — the API-specific call-out remains useful) so the front-of-page disclaimer doesn't contradict it.

## Context

- **Why medium (not low)**: Features.md is one of the most-visited pages on the doc product (820 lines, 30+ feature sections, primary navigation surface). A missing purpose statement on this page has higher reader-experience cost than a missing one on a sub-page. Worth fixing on its own, not bundled into a larger refactor.
- **Why not high**: no operator data loss; no security exposure; readers can infer the pattern from skimming. This is a coherence / discoverability lift, not a correctness fix.
- **Why this surfaces now**: the bigbatch added the four pillar landings (Data Discovery, Management) and re-homed several features under per-pillar pages (Directory, custom-collectors). Now that the per-pillar canonical homes are real, Features.md's role as the **index** is more pronounced — and the page should explicitly own that role rather than leaving it implicit.
- **Recurring pattern**: every primary navigation page should declare its purpose, audience, and place in the IA at the top. Worth folding into `pillars/documentation/gates.md` Gate 7 as an extension cue: "every top-level SUMMARY page opens with a 3-5-line purpose statement before any content / TOC / list."
