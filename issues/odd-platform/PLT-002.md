## What

`POST /ingestion/alert/alertmanager` — the inbound webhook that accepts Prometheus AlertManager notifications and converts them into ODD Distribution Anomaly alerts — is not described in the platform's OpenAPI specification. The controller carries an explicit `// TODO: define OpenAPI spec based on alert provider contract` comment acknowledging the gap.

Because the endpoint is absent from the spec:

- It is not rendered in the Swagger UI served by the platform.
- The generated client libraries (Java / TypeScript) do not expose a typed method for it, forcing operators and downstream tooling to hand-roll HTTP calls.
- The endpoint's auth behavior, status codes, and payload shape are only discoverable by reading Java source.
- The documentation team had to author the payload shape and response code in prose on `docs.opendatadiscovery.org` by reading the controller and service layer directly, rather than cross-linking the spec.

## Where

`odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/controller/AlertManagerController.java:17-32`

```java
@RestController
@RequiredArgsConstructor
public class AlertManagerController {
    private final AlertService alertService;

    // TODO: define OpenAPI spec based on alert provider contract
    @PostMapping(path = "ingestion/alert/alertmanager")
    public Mono<ResponseEntity<Void>> alertManagerWebhook(@RequestBody final Mono<AlertManagerRequest> request) {
        ...
    }

    @Data
    @NoArgsConstructor
    public static class AlertManagerRequest {
        private List<ExternalAlert> alerts;
    }
}
```

The payload model is `org.opendatadiscovery.oddplatform.dto.alert.ExternalAlert` with fields `labels: Map<String,String>`, `generatorURL: URI`, `startsAt: LocalDateTime`.

## Why it matters

- **Discoverability**: operators running Prometheus/AlertManager stacks have no way to discover this native integration from the Swagger UI they already browse; they have to search source or docs.
- **Typed client access**: CI/CD pipelines and internal tools that drive the platform via the generated client cannot call this endpoint today without bypassing the client.
- **Schema drift**: the Java DTO is the de facto contract. Without a spec, any field rename or addition lands silently and has to be re-discovered by doc and client authors.
- **Auth surface**: the endpoint sits under `/ingestion/*` but is **not** protected by the ingestion auth filter today (see **PLT-003** — `auth.ingestion.filter.enabled` only guards `/ingestion/entities`, and `/ingestion/**` is in `SecurityConstants.WHITELIST_PATHS`). The spec should describe the endpoint's current authentication posture accurately rather than implying API-key protection that does not exist. Once PLT-003 ships, the spec addition should declare the `X-API-Key` ingestion auth scheme in the same way the other `/ingestion/*` endpoints will.

## Suggested fix

Add the endpoint to the OpenAPI spec alongside the existing ingestion endpoints:

- Path: `/ingestion/alert/alertmanager`
- Method: `POST`
- Request body: `AlertManagerRequest { alerts: ExternalAlert[] }` where `ExternalAlert` has `labels: object (additionalProperties: string)`, `generatorURL: string (format: uri)`, `startsAt: string (format: date-time)`.
- Response: `204 No Content` on success.
- Security: none enforced today. **Do not** advertise ingestion-API-key protection in the spec until PLT-003 lands (extends the ingestion auth filter to cover this endpoint). When PLT-003 ships, add the `X-API-Key` ingestion auth scheme to match the other `/ingestion/*` endpoints.

Suggested description for the spec: "Receives Prometheus AlertManager webhook notifications. Each alert becomes a Distribution Anomaly alert on the data entity identified by the `entity_oddrn` label. Alerts without `entity_oddrn` are orphaned."

The labels contract (the `entity_oddrn` routing label) should be called out in the endpoint description since it is load-bearing for the integration to work.

## How discovered

Surfaced while documenting the endpoint for ODD-docs item DOC-019 (AlertManager webhook reference). The controller's TODO comment states the intent, but the spec addition has not been made. Filing as an enhancement rather than a bug because the endpoint works — it just isn't advertised in the machine-readable contract.
