## Description

DOC-019 (review `2026-04-24`) surfaced that the authored Authentication subsection at `odd-platform.md:555-559` makes an incorrect security claim. The current text says:

> The endpoint is served under `/ingestion/*`, which is controlled by the platform's **ingestion auth filter**. When the filter is enabled (`auth.ingestion.filter.enabled=true`), AlertManager must present a valid ingestion API key on each webhook call — configure this as an `Authorization: Bearer <token>` header on the AlertManager receiver (or a custom `http_config`). When the filter is disabled, the endpoint accepts unauthenticated POSTs from any network path that can reach the platform.

Every sentence of that paragraph is factually wrong on the current `odd-platform` codebase:

1. **`auth.ingestion.filter.enabled` does not guard this endpoint.** The only filter conditional on that property (`IngestionDataEntitiesFilter.java:20,28`) matches **only** `/ingestion/entities` (POST). The AlertManager endpoint at `/ingestion/alert/alertmanager` is not covered.
2. **`/ingestion/**` is in `SecurityConstants.WHITELIST_PATHS[96]`** — the whole ingestion namespace bypasses Spring Security's permission chain. No authentication is required for the AlertManager webhook regardless of how `auth.ingestion.filter.enabled` is set.
3. **`Authorization: Bearer <token>` is the wrong header anyway.** `S2sAuthenticationFilter.java:20` reads `X-API-Key`, not `Authorization`. Even if S2S were enforced here (it isn't), a Bearer-token configuration on the AlertManager receiver would not be honoured.

This is Critical priority per CLAUDE.md: "**Incorrect documentation (actively misleading)** — including any caveat that would lead to data loss or **security exposure in a default deployment**". An operator enabling `auth.ingestion.filter.enabled=true` and relying on it to protect the AlertManager endpoint has a false sense of security — anyone with network reach to the platform can POST arbitrary alerts.

DOC-004 (pending, high) already documents the correct filter scope in its Description — the DOC-019 authoring did not cross-check that canonical source before writing the Authentication section.
