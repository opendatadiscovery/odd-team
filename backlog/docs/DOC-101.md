## Description

Editorial audit on 2026-05-03 (during `/review batch:feature/docs-features-md-spotcheck`) verified the outbound Slack URLs introduced or reused by DOC-091 / DOC-093. Both URLs returned **302 redirects** to a different host:

| Legacy URL (in our docs) | 302 redirects to |
|---|---|
| `https://api.slack.com/messaging/webhooks` | `https://docs.slack.dev/messaging/sending-messages-using-incoming-webhooks` |
| `https://api.slack.com/apis/connections/events-api` (which itself first redirects to `https://api.slack.com/apis/events-api`) | `https://docs.slack.dev/apis/events-api/` |

Slack has migrated their API documentation from `api.slack.com` to `docs.slack.dev`. The legacy URLs continue to resolve via 302 redirects, so the live-site reader experience is unbroken (Gate 8 PASS for the DOC-090 / DOC-091 / DOC-092 / DOC-093 batch — recorded in `/review` notes 2026-05-03).

But over time, a maintainer-controlled redirect can be deprecated, the link text can become stale ("Slack API" vs "Slack Developer Docs"), and search-engine indexing can lag. The right action is to update our cross-references to the canonical destination.

Additionally, the legacy `api.slack.com/apis/connections/events-api` URL was already a redirect-of-a-redirect: `api.slack.com/apis/connections/events-api` → `api.slack.com/apis/events-api` → `docs.slack.dev/apis/events-api/`. Two redirects per click is fragile.

### Affected occurrences

Run `grep -rn 'api\.slack\.com' docs/` to enumerate exact lines. Estimated count from audit:

- `docs/Features.md` — 3 occurrences (line 153 carries two `api.slack.com/messaging/webhooks` links in the same Alerting paragraph; line 435 carries `api.slack.com/apis/connections/events-api` and `api.slack.com/messaging/webhooks` in the Data Collaboration disambiguation; possibly more)
- `docs/configuration-and-deployment/odd-platform.md` — 5+ occurrences (Enable Alert Notifications hint admonition + Slack key + Enable Data Collaboration intro + hint admonition + Creating Slack application step)
- `docs/main-concepts.md` — 1 occurrence (Slack collaboration app Terms & Aliases row added by DOC-093)
- `docs/developer-guides/api-reference.md` — 2 occurrences (Alerts intro paragraph added by DOC-093 + Data Collaboration intro paragraph)

The `https://api.slack.com/apps` URL on `odd-platform.md:642` (Slack apps website for creating an app) is **not** a docs URL — it's the live Slack admin console. That URL is unaffected; do not include in this sweep.

## Why this slipped past prior batches

The DOC-093 batch reused these URLs because they were already on the page from earlier authoring rounds (per the DOC-093 commit body's Caveats section: "all cross-doc links are relative; outbound URLs already on the page; live-known"). The "live-known" assertion was true at authoring time but did not anticipate Slack's URL canonical migration.

There is no scanner today that checks for outbound-URL redirects (302s). Worth a `playbooks/live-site-verification.md` extension: per outbound URL fetched, record the final-URL after redirects; flag any URL whose final-URL is on a different host as a candidate for canonical update.

Editorial-audit case-law for `playbooks/doc-product-editorial-read.md` — failure shape: Outdated baked-in assumption (URL canonical changed; redirect masks the change).

## Context

- **Failure shape**: Outdated baked-in assumption (per `playbooks/doc-product-editorial-read.md` failure-shape table).
- **Why low**: the legacy URLs work via 302 redirect today; readers experience no broken link. Action is hygiene, not user-impacting fix.
- **Why log at all**: per the world-class-doc mission, every link should resolve directly to canonical content. A redirect is acceptable as a contingency, not as a permanent state. And — per the recurring pattern — Slack has already moved twice (`api.slack.com/apis/connections/events-api` → `api.slack.com/apis/events-api` → `docs.slack.dev/apis/events-api/`); a future move could finally retire the redirect.
- **Recurring pattern**: every outbound URL on the doc tree is a future maintenance liability if its canonical host changes. Worth a Gate-9 extension: outbound-URL sweep records final-URL-after-redirects, flags host changes.
