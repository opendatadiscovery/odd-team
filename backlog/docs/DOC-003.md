## Description

`application.yml` (lines 39-42) defines `auth.s2s.enabled` and `auth.s2s.token` which enable API-key authentication via the `X-API-Key` header. This is the documented path for server-to-server integrations (CI/CD jobs, scripts, automated pipelines) but it is not mentioned anywhere in the documentation. Users needing programmatic API access don't know it exists.

## Acceptance Criteria

- [ ] A new S2S authentication page exists (or equivalent section in the auth README)
- [ ] The page documents `auth.s2s.enabled` (default: false) and `auth.s2s.token` with env-var equivalents
- [ ] Usage is explained: requests authenticated via `X-API-Key: <token>` header
- [ ] The page notes that S2S works alongside other auth mechanisms (OAuth2/LDAP/login-form) rather than replacing them
- [ ] A curl example against an ingestion endpoint is included
- [ ] The authentication README's index lists S2S as an available mechanism

## Re-Audit (2026-04-23)

Flipped from `done` to `review-ready` after consumer-read re-audit under the new eight-gate Quality Bar (Phase C).

**No in-scope amendment required.** The S2S page at `docs/configuration-and-deployment/enable-security/authentication/s2s.md` already carries the Security considerations section (admonition block warning about token handling, HTTPS, and the `ADMIN` impersonation effect). Consumer-read confirms every runtime claim on the page:

- `S2sTokenProvider.java:21-22` — `@Value("${auth.s2s.token:#{null}}")` and `@Value("${auth.s2s.enabled:false}")` match the documented keys and defaults.
- `S2sTokenProvider.java:29-33` — `isValidToken` does a constant-time-equivalent `.equals` comparison against the configured token. The doc's "X-API-Key header → ADMIN role" claim is correct.
- `S2sTokenProvider.java:41-46` — `@PostConstruct validate()` throws `IllegalStateException("Long Term Token is not defined")` when `auth.s2s.enabled=true` without a token. Matches the doc's "token must be set when enabled" claim.
- `S2sAuthenticationFilter.java:35-61` — reads the `X-API-Key` header, builds `User.withUsername("ADMIN").roles("ADMIN")` on a successful match, and falls through to the next filter on mismatch (no 401 from this filter alone — the downstream OAuth2/form filters handle unauthenticated requests). The doc correctly describes this as "S2S works alongside other auth mechanisms."
- `OAuthSecurityConfiguration.java:108-110` — `if (s2sEnabled) { sec.addFilterAt(s2sAuthenticationFilter, SecurityWebFiltersOrder.HTTP_BASIC); }` confirms the filter is registered conditionally. The Spring Security chain is correctly documented at the page's "How it works" section.

**Platform-side follow-up logged**: `S2sTokenProvider.isValidToken` has an NPE path when `auth.s2s.enabled=false` and a request arrives with an `X-API-Key` header — `s2sToken` is never bound, and `s2sToken.equals(token)` dereferences null. The filter is only registered when `s2sEnabled=true`, so the NPE is unreachable in normal deployments, but the method is public and defensive-null handling is cheap. Not a doc issue; tracked as a GitHub Issue candidate on `odd-platform`.
