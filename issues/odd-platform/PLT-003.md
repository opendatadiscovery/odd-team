## What

`POST /ingestion/alert/alertmanager` — the Prometheus AlertManager webhook endpoint that ingests external alerts as Distribution Anomaly alerts — is reachable **without any authentication** in every deployment, regardless of how `auth.ingestion.filter.enabled` is configured.

Anyone with network reach to the platform can POST arbitrary AlertManager-shaped payloads and create alerts on any data entity whose ODDRN they can guess. Because the endpoint creates rows in the `alert` / `alert_chunk` tables with `status = OPEN`, the attacker can also fill the Alerts UI with noise until an operator notices.

## Where

Three files participate in the gap:

- `odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/auth/util/SecurityConstants.java:95-96` — `/ingestion/**` is in `WHITELIST_PATHS`, bypassing Spring Security's permission chain:
  ```java
  public static final String[] WHITELIST_PATHS =
      {"/actuator/**", "/favicon.ico", "/ingestion/**", "/img/**", "/api/slack/events"};
  ```

- `odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/auth/filter/IngestionDataEntitiesFilter.java:20,28` — the only filter conditional on `auth.ingestion.filter.enabled`, scoped to `/ingestion/entities` only:
  ```java
  @ConditionalOnProperty(value = "auth.ingestion.filter.enabled", havingValue = "true")
  public class IngestionDataEntitiesFilter extends AbstractIngestionFilter {
      public IngestionDataEntitiesFilter(...) {
          super(new PathPatternParserServerWebExchangeMatcher("/ingestion/entities", HttpMethod.POST));
          ...
      }
  }
  ```

- `odd-platform-api/src/main/java/org/opendatadiscovery/oddplatform/auth/filter/IngestionDataSourceFilter.java:20` — the second ingestion filter, always active (no @ConditionalOnProperty), scoped to `/ingestion/datasources`:
  ```java
  public IngestionDataSourceFilter(final ReactiveCollectorRepository collectorRepository) {
      super(new PathPatternParserServerWebExchangeMatcher("/ingestion/datasources", HttpMethod.POST));
      ...
  }
  ```

Neither filter covers `/ingestion/alert/alertmanager`, so `AlertManagerController.java:21` (`@PostMapping(path = "ingestion/alert/alertmanager")`) is reachable with no auth enforcement at all.

## Why it matters

- **Default-deploy security regression when the endpoint is documented.** ODD Team's DOC-019 (AlertManager integration) shipped to `docs.opendatadiscovery.org` on 2026-04-24, advertising the endpoint publicly for the first time. Before DOC-019, the endpoint was discoverable only by source-reading; after DOC-019, every operator running ODD Platform knows it exists and knows its path. Anyone scanning a public ODD Platform URL now has a known unauthenticated write endpoint.
- **The sibling endpoints are better guarded.** `/ingestion/datasources` is always token-protected (`IngestionDataSourceFilter` is unconditional); `/ingestion/entities` is token-protected when `auth.ingestion.filter.enabled=true`. Operators reading DOC-004 (pending) or the sibling code will reasonably expect `/ingestion/alert/alertmanager` to follow the same model.
- **Writes have side effects.** Each POST creates `ALERT` + `ALERT_CHUNK` rows, routed by the `entity_oddrn` label. An attacker who can guess ODDRN patterns (e.g. `//postgresql/host/db/schemas/public/tables/orders`) can fabricate Distribution Anomaly alerts on production entities.
- **Housekeeping does not garbage-collect these alerts promptly.** The housekeeping job only deletes alerts after they are resolved — auto-resolved (`RESOLVED_AUTOMATICALLY`) alerts age out after `housekeeping.ttl.resolved_alerts_days` (default 30); manual `RESOLVED` alerts are deleted on the next housekeeping run regardless of retention (separate SQL-precedence bug — see PLT-005). Fake `OPEN` alerts created via this unauthenticated endpoint stay forever until an operator resolves them.

## Suggested fix

The simplest fix is to extend `IngestionDataEntitiesFilter` (or add a parallel `IngestionAlertManagerFilter`) so that `/ingestion/alert/alertmanager` (POST) is also matched when `auth.ingestion.filter.enabled=true`. The filter should require a valid S2S token via the `X-API-Key` header (matching the existing ingestion auth model — operators already know how to provision these).

Option A — extend the existing filter path matcher to cover both paths:
```java
public IngestionDataEntitiesFilter(...) {
    super(new OrServerWebExchangeMatcher(
        new PathPatternParserServerWebExchangeMatcher("/ingestion/entities", HttpMethod.POST),
        new PathPatternParserServerWebExchangeMatcher("/ingestion/alert/alertmanager", HttpMethod.POST)
    ));
    ...
}
```
(adjusts naming: rename filter or accept the slight naming drift.)

Option B — add a new `IngestionAlertManagerFilter` in the same package, same `@ConditionalOnProperty` gate, scoped to `/ingestion/alert/alertmanager`. Body-decorator logic can reject requests without a valid `X-API-Key`.

Option C (defense-in-depth, additional to A or B) — remove `/ingestion/**` from `SecurityConstants.WHITELIST_PATHS` and explicitly list `/ingestion/entities`, `/ingestion/datasources`, `/ingestion/alert/alertmanager` instead, so future endpoints added under `/ingestion/*` do not silently inherit whitelist coverage.

Documentation (DOC-065) should ship alongside the code fix so operators upgrading to the fixed version understand the new filter coverage, and operators on older versions know to apply perimeter controls.

## How discovered

Surfaced during `/review DOC-019` (2026-04-24) — the review's Gate 4 consumer-read audit cross-checked the doc's authentication claim against `IngestionDataEntitiesFilter`, `IngestionDataSourceFilter`, `S2sAuthenticationFilter`, and `SecurityConstants.WHITELIST_PATHS`. DOC-004 (pending) already describes the `/ingestion/entities`-only scope of `auth.ingestion.filter.enabled` — DOC-019 contradicted DOC-004 by asserting the flag guards the AlertManager endpoint.
