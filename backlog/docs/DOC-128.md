## Description

DOC-099's Phase A + Phase B shipped in `feature/docs-coherence-followups-2026-05-05` (commits `46316cc` Phase A + `0cc8382` Phase B): the Collector category-vs-implementation conflation on Architecture.md L11 was fixed, and main-concepts.md Terms & Aliases was extended with the Push-client client-server framing row. Phase C — the broader "expand Architecture.md to match its top-level SUMMARY weight or repurpose as a thin index" — was deferred to keep the per-item review surface clean.

The underlying coherence problem is unchanged: Architecture.md is **17 lines total** while Main Concepts is now **87 lines** (post-DOC-114) and Features is **~820 lines**. Per Cornerstone 2 ("hierarchy depth must reflect conceptual depth"), Architecture's top-level SUMMARY placement claims peer status with the other primary pillars; a 17-line stub sends the implicit signal that Architecture is unfinished, undermining the doc-product credibility the placement asserts.

## What's missing on Architecture.md today (Phase C scope)

- **Data flow** producer → ingest → store → query → render. Today the page has the diagram image and nothing in prose.
- **Deployment topology** — which components run where: Platform (server), Collectors (deployable services hosting plugins), Push adapters (embedded in source runtimes), the optional Tracing Gateway (via DOC-084), the UI (served from the platform). What is centralized vs distributed; what an operator sees as a deployable unit.
- **Cross-cutting concerns** — auth (UI + S2S + ingestion filter), alerting (in-app + Slack/email/webhook), lineage (cross-system via ODDRN), search, attachments, GenAI proxy. A landing-level paragraph per concern with cross-link to its canonical page.
- **Trade-off discussion** — pull vs push topology, when to choose which (some of this folded into main-concepts.md by DOC-114; here it lives at the architectural-overview level rather than the concept-definition level — different framing).
- **Where to read the code** — pointer to `navigation/domains/*.md` from the workspace OR pointer to the relevant feature pages and developer-guides for the contributor audience.

## Two candidate paths

### Path A — Expand to a real architectural-overview page

Author a Cornerstone-2-grade overview with the five sections above (data flow / deployment topology / cross-cutting concerns / pull-vs-push trade-off / where-to-read-the-code). Keep the diagram. Result: a 100-200-line page that operators and developers can land on for the structural mental model before they dive into Features or Configuration. The page operates in client-server topology framing throughout (uses **Push-client**, **Collector**, **Platform**, **Server**), per the multi-framing guidance now in main-concepts.md Terms & Aliases.

### Path B — Repurpose as a thin index

Replace Architecture.md content with a short index page (~30 lines) pointing at the canonical homes for each architectural concern: main-concepts.md for vocabulary, integrations/README.md for the integration surface, configuration-and-deployment/ for the operator surface, developer-guides/ for the developer surface. Keep the diagram. Result: Architecture.md is a navigation hub, not a content source; the SUMMARY entry stays meaningful (it's the front door for the architectural mental model) but the content lives where it canonically belongs (Cornerstone 5).

Path A is the more ambitious fix; Path B is the lighter-weight realignment. The decision is a Cornerstone-2 judgment call ("does Architecture warrant its own pillar-grade landing, or is it a hub that organizes the other pillars?") and warrants surfacing to the user before authoring begins.
