## Description

Editorial audit at post-merge state of `feature/docs-doc131-pillar-relpath-fix-2026-05-06` ran an end-to-end read of `docs/active-platform-features/alerting.md` (the migration target of DOC-131 Phase B.1). The page carries two H2 sections that describe the same operator action — the **Notification Settings** button on a data entity — at different depths:

- **`## Halt notifications per entity` (L107-146)** — substantial, ~40 lines: explains halts per alert type, per-toggle table, five fixed durations, the automatic-resume timer, the interaction with auto-resolution (info hint), the API-side endpoints, plus the **`### Distribution anomaly halt is currently unenforced`** known-limitation warning.
- **`## Setting up alert types` (L148-152)** — 5 lines, generic prose:
  ```
  Alert notifications within Open Data Discovery are customizable. Users can choose the type and period to receive alert notifications with ease. This setup requires hitting the `Notification Settings` button and configuring it.

  ![](../.gitbook/assets/medium-0-IqRcSL9Wi88jF8yo.png)
  ```

Both sections direct the reader to the same `Notification Settings` button. The Halt section already documents the per-alert-type toggles, the durations, and the per-entity scope in detail; the "Setting up" section adds nothing operationally — its prose ("type and period to receive alert notifications with ease") is a thin paraphrase of what the Halt section establishes precisely. A reader walking the page top-to-bottom hits the rich Halt content first, then a thin restatement, and is left wondering whether "Setting up" is a different operation than the Halt toggles.

### Why this slipped past DOC-131 review

DOC-131 Phase B.1 migrated the Alerting section verbatim from `docs/Features.md` (the original page had two duplicate-anchor H3 sections — `## Setting up alert types` at L173 and `## Setting up alert notifications` at L279, captured in the closed DOC-094). The "Setting up alert notifications" half migrated cleanly to `notifications.md` as `## Setting up notifications` (correctly thin and pointing at the canonical configuration reference). The "Setting up alert types" half migrated as-is to `alerting.md` — but on the new page it sits next to the more detailed `## Halt notifications per entity` section that DOC-131 also migrated, and the redundancy that was tolerable on the broader Features.md showcase becomes visible coherence drift on a focused feature page.

DOC-131 review verified each migrated section was present and rendered (Phase B.1 PASS criterion); it did not run an intra-page coherence check that would have surfaced the now-redundant section. The editorial audit's per-page systematic walk catches this class — same axis A2 (intra-page coherence) as DOC-114 (`main-concepts.md` 4× redundancy), different scale.

## Failure shape

**A2 (Intra-page coherence) — same operation described twice on the same page** (per `pillars/documentation/quality-framework.md`). Reader-flow defect: the second H2 invites the reader to re-read content already established earlier on the same page, with no signal that it's a recap rather than a new operation. Cousin of DOC-114 (`main-concepts.md` 4× repetition of Adapter / Plugin / Collector / Push adapter content), narrower scope.
