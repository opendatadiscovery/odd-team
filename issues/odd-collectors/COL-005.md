## What

`SagemakerPlugin` in `odd-collector-aws` re-declares every AWS auth field that already exists on the `AwsPlugin` base, but **without explicit defaults**. In Pydantic v2 (which the codebase uses), `Optional[str]` without `= None` declares a field that is **type**-optional but is still **required** by the model ÔÇö the operator must provide each field in the YAML (a value or `null`) or the collector errors out at startup.

This is asymmetric with every other AWS adapter in the package. `AwsPlugin` declares each auth field as `Optional[str] = None`, so `AthenaPlugin`, `DmsPlugin`, etc. inherit them as truly optional and the operator can omit them to fall back to the boto3 default credential chain. `SagemakerPlugin` breaks that pattern silently.

## Where

`odd-collector-aws/odd_collector_aws/domain/plugin.py:10-19` ÔÇö base class `AwsPlugin`:

```python
class AwsPlugin(Plugin):
    aws_secret_access_key: Optional[str] = None
    aws_access_key_id: Optional[str] = None
    aws_region: Optional[str] = None
    aws_session_token: Optional[str] = None
    aws_account_id: Optional[str] = None
    profile_name: Optional[str] = None
    aws_role_arn: Optional[str] = None
    aws_role_session_name: Optional[str] = None
    endpoint_url: Optional[str] = None
```

`odd-collector-aws/odd_collector_aws/domain/plugin.py:90-97` ÔÇö re-declaration on `SagemakerPlugin`:

```python
class SagemakerPlugin(AwsPlugin):
    type: Literal["sagemaker"]
    aws_secret_access_key: Optional[str]      # <-- no default; effectively required
    aws_access_key_id: Optional[str]          # <-- no default; effectively required
    aws_region: Optional[str]                 # <-- no default; effectively required
    aws_session_token: Optional[str]          # <-- no default; effectively required
    aws_account_id: Optional[str]             # <-- no default; effectively required
    experiments: Optional[list[str]]
```

The reference YAML at `odd-collector-aws/config_examples/sagemaker.yaml` works around this by setting every auth field to a placeholder, but operators who copy the AwsPlugin pattern from a working `athena` / `dms` / `glue` config and try to omit fields they don't need will see a Pydantic validation error.

## Why it matters

- Operators who deploy other AWS adapters first (Glue, S3, AthenaÔÇª) and then add SageMaker hit a Pydantic validation error at startup with no obvious explanation ÔÇö the field is named identically, and the type is identical, so the asymmetry isn't visible until they read the source.
- The IAM-role-based deployment pattern (recommended by the project's own docs and Compose templates) relies on the boto3 credential chain ÔÇö exactly the path that `SagemakerPlugin` blocks because the inline fields are required.
- Severity is **medium** ÔÇö workarounds exist (setting each field to `null` in YAML), but the discovery cost is high.

## Suggested fix

Either delete the re-declarations entirely (`SagemakerPlugin` already inherits the right fields from `AwsPlugin`) or add `= None` to each:

```python
class SagemakerPlugin(AwsPlugin):
    type: Literal["sagemaker"]
    experiments: Optional[list[str]] = None
```

The adapter's actual auth code path doesn't change ÔÇö it still pulls credentials from these fields when set, and from the boto3 chain when not. The fix is purely the model declaration.
