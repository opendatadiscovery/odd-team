## Description

DOC-124's mandatory `grep -rEn '  ' docs/` cross-tree sweep (per its ACs) caught the use-case-subtree defects DOC-124 was scoped to fix, plus four hits **outside** DOC-124's `affected_files` list:

| File | Line | Defect | Suggested fix |
|---|---|---|---|
| README.md | 10 | `It contains  best practices for managing` — double-space after `contains` | single space |
| configuration-and-deployment/odd-platform.md | 59 | `{database\_name}  but schema, where Lookup Tables are stored` — double-space before `but` (prose, not code block) | single space |
| configuration-and-deployment/odd-platform.md | 370 | `If you changed  max\_wal\_senders to 5` — double-space before `max\_wal\_senders` (prose, not code block) | single space |
| configuration-and-deployment/odd-platform.md | 426 | `      password:  {email_password}` — inside a YAML code block; double-space after the colon | inspect: if the surrounding YAML uses `key: value` (single-space) form throughout, fix; if other lines use `key:  value` for vertical alignment, leave |

DOC-124 was scoped to the use-case subtree (5 files); expanding scope mid-implementation to README.md + odd-platform.md is a scope-expansion judgment call. The cleaner play per `playbooks/follow-up-on-disk.md` is "small-batch-item" — log this on disk and pick up in the next sweep.

## Why this slipped past prior batches

Same root cause as DOC-124: pattern-based sweeps (`priviligies|crushes|invironment|Platfrom|recomendations|PosgreSQL` from DOC-113) catch what their patterns name. The literal-double-space pattern was added by DOC-124 and immediately surfaced these out-of-subtree residues — a sweep that runs the `grep -rEn '  ' docs/` cross-tree pattern routinely (e.g., as part of every batch's pre-flight) would catch this class as it accretes. Worth adding to the doc-quality framework as a Mode 1 axis A12 sub-procedure.
