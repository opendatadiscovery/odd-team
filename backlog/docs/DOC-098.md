## Description

Editorial audit on 2026-05-03 (during `/review batch:feature/docs-features-md-spotcheck`) surfaced two issues on `docs/developer-guides/github-organization-overview.md`:

### Issue 1 — broken markdown on line 7

```
**odd-platform** \[l[ink](https://github.com/opendatadiscovery/odd-platform)]
```

The intended rendering is `**odd-platform** [link](https://github.com/opendatadiscovery/odd-platform)`. The actual line has a typo: `\[l[ink](...)` — the escape `\[` opens a bracket that is never closed; the `l` is stray text outside any link; the inner `[ink](...)` becomes the actual link. Renders on GitBook as something like `[l` followed by a link labelled `ink`. Broken display on the canonical first repo entry of the page.

The other 16 repo entries in the file use the correct pattern `\[[link](...)]` (escaped opening bracket, real link, escaped closing bracket — yields the literal text `[link]` with the word `link` itself hyperlinked). The `odd-platform` entry on line 7 is the only one missing the inner-bracket escape.

### Issue 2 — three missing repos

The github-organization-overview page is meant to be the single canonical map of repositories under `github.com/opendatadiscovery`. Three repos are referenced from other doc pages but absent from this page:

| Repo | Referenced from | Why it matters |
|---|---|---|
| `opendatadiscovery/documentation` | (this very page lives in this repo) | The doc product itself; readers contributing should know which repo to file PRs against. |
| `opendatadiscovery/odd-spark-adapter` | `docs/integrations/README.md:20` | Push-adapter for Spark; primary integration. |
| `opendatadiscovery/odd-tracing-gateway` | `docs/configuration-and-deployment/deployment.md:286` (deployed alongside Platform; chart at `charts/odd-tracing-gateway`) | Optional companion component for ingesting metadata from custom microservices. |

A reader using github-organization-overview as the index for "what repos does ODD have?" gets an incomplete picture. A maintainer looking up "where do I file a docs bug?" finds no entry — the doc product itself is missing from the repo map.

## Why this slipped past prior batches

The page was authored once and not maintained as new repos joined the org. There is no scanner today that cross-checks the page against either the actual GitHub org listing (`gh api orgs/opendatadiscovery/repos`) or against the doc tree's outbound `github.com/opendatadiscovery/{repo}` references. Either of those would have surfaced the gaps.

`navigation/architecture.md` (the implementer-side canonical repo list per `pillars/documentation/gates.md` Gate 9 SoT table) is the source of truth for repo claims. The github-organization-overview page should be regenerable from architecture.md (or at minimum, sync-checked against it).

Editorial-audit case-law for `playbooks/doc-product-editorial-read.md` — failure shape: Coverage hole (canonical index incomplete) + Half-finished narrative (page was authored once and not maintained).
